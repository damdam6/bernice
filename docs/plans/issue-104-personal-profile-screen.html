1요약

src/pages/Players.tsx는 10줄짜리 "P4에서 채워집니다" 스텁이다. 개인 화면이 쓸 데이터(RecordsResponse.players[] — trends·personalBests 파싱 완료), 차트 컴포넌트(RadarChart·TrendChart·PerformanceBar), 정규화 함수(buildPerformanceScale)가 전부 구현·테스트됐지만 사용처가 0이다. 이 작업은 이미 만들어진 부품을 조립하는 일이다 — 차트/정규화 로직은 손대지 않고, Rankings.tsx가 이미 확립한 패턴(useRecords() → 로딩/에러/빈 상태 → FilterChip + 공유 scale)을 그대로 따른다. 파생 로직은 Rankings가 ranking-view.ts를 두는 것처럼 순수 함수 모듈 profile-view.ts로 분리해 단위 테스트하고, 화면은 배선만 담당한다. 정본은 docs/prd-design.html §05 👤 개인 매핑 표 + DesignSync 목업 PROFILE 섹션이다.

2현황 분석

  • 스텁: src/pages/Players.tsx는 정적 placeholder 카드만 렌더. 라우트는 src/App.tsx:23(/players, LoginGate 하위) + src/App.tsx:25 (/trends → /players 리다이렉트). 라우팅 변경 불필요.
  • 데이터 진입점: useRecords()(src/hooks/useRecords.ts)가 /api/records를 페치·검증해 RecordsResponse를 반환. LoginGate가 이미 페치하므로 이 화면은 캐시 히트(staleTime 5분).
  • 기준 레퍼런스: src/pages/Rankings.tsx가 로딩/에러/빈 상태 3분기, events.length===0 || sessions.length===0 → EmptyState, 회차 칩 {i+1}차, const scale = useMemo(() => buildPerformanceScale(events, sessions), …)를 이미 확립. 이 화면은 동일 골격을 재사용한다.
  • 부품 (구현·테스트 완료, 그대로 사용):
    • src/components/charts/radar-chart.tsx — axes: {label, value(0~1)}[]. 정규화는 호출자 책임.
    • src/components/charts/trend-chart.tsx — sessionLabels, highlight, background[][], goal(0~1). 배경→목표선→본인 순서로 그림.
    • src/lib/performance-scale.ts — buildPerformanceScale(events, sessions).normalize(key, rawValue). 랭킹·레이더·추이가 공유하는 단일 정규화(§07). 목표선도 같은 함수로 정규화.
    • FilterChip, Card, 공통 EmptyState/Spinner/ErrorPanel/CenteredPanel.
  • 도메인 계약(shared/domain.ts): PlayerSummary는 탈퇴 제외가 타입 수준 강제 (status: Exclude<PlayerStatus,'탈퇴'>) — 드롭다운 필터링 불필요. trends[]는 events 전원 존재(기록 없으면 points:[]), personalBests[]는 스파스(유효 기록 있는 종목만). TrendPoint.deltaFromPrevious/improved는 직전 유효 기록 대비(직전 회차가 아님).

3설계

구조: 순수 파생 모듈(profile-view.ts) + 재사용 컴포넌트 2개 (PlayerSelect, GrowthStatCard) + 조립 화면(Players.tsx). Rankings/ranking-view 분리 방식을 그대로 미러링한다.

상태(화면): selectedPlayerId(기본 players[0].id), selectedSessionDate(기본 최신 = sessions[last].date, Rankings와 동일), selectedEventKey(기본 events[0].key — 첫 성장 카드 확장, 목업 기본 동작과 일치). 카드 탭은 단일 개방 아코디언 + 열린 카드 재탭 시 닫힘(prev === key ? null : key).

데이터 매핑(전부 profile-view.ts 순수 함수, 정규화는 공유 scale 주입):

요소소스규칙
선수 드롭다운players[]탈퇴는 타입상 이미 제외. 선택 항목 체크(✓), 바깥 탭 닫힘(오버레이 방식).
회차 칩sessions[]FilterChip, 라벨 n차(index+1, 날짜 오름차순).
레이더 축값선택 회차 entries[].scores[key]recorded면 scale.normalize(key, value), 아니면 0(목업 perfs.map(p=>p==null?0:p)과 동일). 라벨 = event.key.
성장 카드 PBpersonalBests[]find(e=>e.event===key)?.display ?? '—'(스파스).
성장 카드 현재값선택 회차 scores[key]recorded → display, exempt → 면제, 그 외 → —.
성장 카드 델타trends[].points[]선택 회차 sessionDate의 point에서 deltaFromPrevious/improved 읽기. null(첫 기록/미기록) → — 회색, 0 → ─ 0 회색, 그 외 → ▲/▼ |Δ|{시간이면 초} + green/red.
추이 본인본인 trends[].points[]{sessionIndex: 날짜인덱스, value: scale.normalize(key, point.value)}. 스파스 유지.
추이 배경본인 제외 players[].trends[]선수별 동일 매핑, 빈 시리즈 제외. TrendChart.background로.
추이 목표선events[].targetValuescale.normalize(key, targetValue) → goal.

델타 색상 토큰(§06/§07): 개선 text-good(#3f9e6b) · 악화 text-bad(#c25151) · 동률/무비교 text-ink-muted. 카드 선택 보더 border-primary 1.5px(§06 "카드 선택 = primary 1.5px").

고려했으나 배제: 차트/정규화 컴포넌트 수정(이미 계약대로 완성 — 조립만 필요) · 델타를 화면에서 직접 재계산(trends[] 계약값이 "직전 유효 기록 대비"로 이미 정확, §01 콜아웃) · 바깥 탭 닫힘을 document 리스너로(목업의 position:fixed;inset:0 오버레이가 테스트하기 쉽고 목업과 동일).

리스크영향완화책

미기록 종목의 레이더 축값축이 비어 폴리곤 깨짐recorded 아니면 0으로 매핑(목업과 동일). RadarChart는 clamp01로 방어.
추이 point의 sessionDate ↔ 회차 인덱스 매핑x축 위치 어긋남sessions[](날짜 오름차순)로 Map<date,index> 1회 구성, 미존재 날짜 point는 방어적으로 제외.
선수/회차 0건, 기록 전무 선수런타임 오류·빈 화면Rankings 3분기 + players.length===0 빈 상태 추가. 기록 없는 선수는 레이더 0·카드 —로 자연 처리.
정규화 불일치(랭킹과 다른 값)§07 위반랭킹과 동일한 buildPerformanceScale(events, sessions) 인스턴스 로직 공유(별도 구현 금지).
