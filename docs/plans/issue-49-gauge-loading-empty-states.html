1요약

P4 전 화면이 공유할 프레젠테이션 컴포넌트 4종 — 달성률 Gauge(0~100%), Spinner, ErrorPanel, EmptyState — 을 src/components/common/에 새로 만든다. 데이터 계약(useRecords 훅 #46, 인증 401 처리 #42)이 아직 존재하지 않으므로 네 컴포넌트 모두 순수 프레젠테이션(props만 받고 fetch/쿼리 상태에 결합하지 않음)으로 설계한다. 이슈가 독립(의존 없음)이라 형제 이슈 #44(라우팅)·#46(TanStack Query)·#48(카드/뱃지/상태 pill)과 무관하게 진행하되, #48과 같은 common/ 디렉터리를 병행 작업 중이므로 파일 단위로만 겹치도록 배럴 index.ts는 만들지 않는다.

2현황 분석

  • src/는 아직 App.tsx(P0 placeholder) 하나뿐 — src/components/ 디렉터리 자체가 없다. 형제 이슈 #44/#46/#48/#50/#51의 워크트리도 전부 같은 베이스 커밋(6e66b2e)에 멈춰 있어(플랜 파일도 자기 이슈 번호로는 없음) 아직 아무도 src/components/ 컨벤션을 확정하지 않았다 — 이번 이슈가 사실상 첫 프론트 컴포넌트 작업.
  • 디자인 토큰은 src/index.css:1-30에 이미 확정: 브랜드 오렌지 스케일(--color-brand-50~950), 시맨틱 --color-good/warn/bad(주석: "목표 달성 뱃지 등"). App.tsx:11-14,36가 bg-good/warn/bad를 상태 dot에 이미 쓰고 있어 실사용 전례가 있다.
  • 아이콘 라이브러리 미설치(package.json 확인) — App.tsx:29는 이모지(🏀)를 아이콘 대신 쓴다. 이 관례를 그대로 따른다(EmptyState/ErrorPanel 아이콘도 이모지).
  • vitest.config.ts:4-6은 environment: 'node'이고 @testing-library/react·jsdom·happy-dom 모두 devDependency로 설치돼 있지 않음(package-lock.json엔 vitest의 옵셔널 peer로만 언급). 즉 이 레포엔 아직 React 컴포넌트를 렌더링해 검증하는 자동 테스트 인프라가 없다 — 기존 테스트는 전부 functions/lib/*.test.ts 순수 함수 테스트뿐.
  • 완료 조건 문구("게이지가 0·중간·100% 값에서 올바르게 렌더")는 자동 렌더 테스트가 아니라 수치 계산의 정확성 + 육안 확인으로 만족 가능 — 게이지의 각도/채움 비율 계산을 순수 함수로 분리하면 기존 functions/lib/ 컨벤션과 동일하게 vitest(node 환경)로 단위 테스트할 수 있다.
  • shared/domain.ts:124-129의 EventAchievementRate.rate(0~1)가 이 Gauge의 실사용처 중 하나(홈 화면 종목별 달성률, docs/build-plan.html P4 홈 항목)로 보이지만, 값 변환(0~1 → 0~100)은 소비 측(#46 훅 완성 후 P4 화면)의 책임으로 남기고 Gauge는 이슈 문구 그대로 0~100 스케일의 value prop만 받는다 —도메인 타입에 결합하지 않는다.
  • functions/api/records.ts:35-49의 에러 응답({ error, message, upstreamStatus? })과 401 구분(#42, 미착수)이 아직 확정되지 않아 ErrorPanel이 특정 에러 스키마에 맞출 근거가 없다 — title/message/onRetry 정도의 범용 props로 최소화한다.

3설계

디렉터리 — src/components/common/에 컴포넌트당 파일 하나(+게이지만 계산 헬퍼 분리). 배럴 index.ts는 만들지 않고 각 소비처가 파일을 직접 import — #48이 같은 디렉터리에 Card/Badge/StatusPill을 병행 작업 중이라, 공유 인덱스 파일을 함께 건드리면서 생기는 불필요한 머지 충돌을 피한다.

  • Gauge — 막대형이 아닌 원형(링) SVG 게이지로 결정. 이슈 문구가 형태를 못박지 않았고, "게이지"라는 표현과 홈 화면의 종목별 카드형 레이아웃(다건 나열)에 원형이 더 적합하다고 판단. value: number(0~100, 범위 밖은 클램프) · size?: 'sm' | 'md'(기본 md) · label?: string(게이지 아래 캡션, 선택) props. 원형 채움 비율·클램프 계산은 gauge-math.ts로 분리해 DOM 없이 단위 테스트. 색상은 브랜드 단색(트랙 brand-100, 채움 brand-500) — 달성률 구간별 good/warn/bad 색상 분기는 기각(아래 참고).
  • Spinner — Tailwind animate-spin 기반 CSS 스피너(테두리 원, JS 계산 불필요). size?: 'sm' | 'md' · label?: string(기본 "불러오는 중…", sr-only로 스크린리더에만 노출). role="status".
  • ErrorPanel — title?: string(기본값 제공) · message: string(필수) · onRetry?: () => void(있으면 재시도 버튼 렌더). role="alert". fetch/쿼리 관련 타입에 의존하지 않는 순수 프레젠테이션 — #46/#42가 확정되면 그쪽에서 이 props로 매핑.
  • EmptyState — icon?: string(이모지, 기본값) · title: string · description?: string · children?: ReactNode(선택적 CTA 슬롯 — 특정 버튼/링크 형태를 강제하지 않기 위해 구체적 action prop 대신 children으로 열어 둠).
  • 네 컴포넌트 모두 App.tsx:24-33의 기존 카드 톤(rounded-2xl/border-brand-200/shadow-sm/브랜드 텍스트 컬러)을 재사용해 시각적 일관성을 맞춘다.

기각한 대안

  • Gauge를 막대형(linear progress bar)으로 구현 — 기각: 여러 종목을 카드로 나열하는 홈 화면 레이아웃엔 원형이 더 적합하고, 링 형태가 "게이지"라는 이름에도 더 부합. 리뷰에서 막대형 요청 시 쉽게 교체 가능하도록 gauge-math.ts 계산 로직은 형태와 무관하게 재사용 가능한 순수 함수로 둔다.
  • Gauge 색상을 달성률 구간(예: <40% bad, 40~70% warn, ≥70% good)별로 분기 — 기각: 구간 경계값이 이슈·설계 문서 어디에도 정의돼 있지 않아 이번에 임의로 정하면 나중에 실제 정책과 어긋날 위험. 브랜드 단색으로 시작하고, 필요해지면 P4 구현 시점에 실제 기준과 함께 추가.
  • 이번 이슈에서 @testing-library/react + jsdom 도입 — 기각: 레포 전체의 프론트 테스트 전략을 바꾸는 결정이라 이 S급 단일 이슈 범위를 벗어난다. 대신 게이지 산술은 순수 함수로 분리해 기존 vitest(node) 컨벤션으로 테스트하고, 나머지 시각적 정확성은 npm run dev 수동 확인으로 커버.
  • 4개 컴포넌트를 배럴 src/components/common/index.ts로 재노출 — 기각: #48이 같은 디렉터리를 병행 작업 중이라 공유 파일을 만들면 두 PR이 같은 줄을 두고 충돌할 여지가 생긴다. 소비처가 늘어나 배럴이 실제로 필요해지면 그때 어느 한쪽 PR에서 추가.

4작업 단계

  1. gauge-math.ts 작성 — clampGaugeValue(value)(NaN→0, 범위 밖 클램프)와 getGaugeDashArray(value, radius)(SVG stroke-dasharray 문자열 계산)를 순수 함수로. 단위 테스트: 0%(채움 0) · 중간값(예: 50%, 절반 채움) · 100%(전체 채움, 원둘레와 일치) · 범위 밖(-10→0, 150→100) · NaN→0. src/components/common/gauge-math.ts, src/components/common/gauge-math.test.ts
  2. Gauge.tsx 작성 — viewBox="0 0 100 100" SVG에 트랙 circle + 채움 circle(-90deg 회전으로 12시 방향 시작, gauge-math의 dasharray 사용), 중앙에 반올림한 퍼센트 텍스트(절대 위치 span), size variant(sm/md)별 지름·스트로크 두께, role="img" aria-label="달성률 {value}%". src/components/common/Gauge.tsx
  3. Spinner.tsx 작성 — border-4 border-brand-100 border-t-brand-600 rounded-full animate-spin 조합의 CSS 스피너, size variant, role="status" + sr-only 레이블. src/components/common/Spinner.tsx
  4. ErrorPanel.tsx 작성 — role="alert" 카드, 제목/메시지/선택적 재시도 버튼(onRetry 있을 때만 렌더). src/components/common/ErrorPanel.tsx
  5. EmptyState.tsx 작성 — 이모지 아이콘(기본값)/제목/선택적 설명/선택적 children 슬롯의 중앙 정렬 카드. src/components/common/EmptyState.tsx
  6. npm run typecheck && npx vitest run으로 회귀 없음 확인(기존 테스트 + 신규 gauge-math.test.ts).
  7. 수동 시각 확인 — npm run dev 구동 후 App.tsx를 임시로 고쳐 네 컴포넌트를 한 화면에 나열(게이지는 0/50/100% 세 개, 나머지 셋은 기본 상태 하나씩)하고 브라우저로 직접 렌더 확인(특히 100% 게이지에서 strokeLinecap 시작점에 점처럼 겹쳐 보이는 흔한 SVG 아치 버그가 없는지). 확인 후 App.tsx 임시 변경은 되돌린다(App.tsx는 #44 라우팅 이슈가 교체할 P0 placeholder라 이번 커밋에 남기지 않음).

5리스크

리스크영향완화책
"게이지" 형태(원형 vs 막대)가 이슈 문구에 명시돼 있지 않음리뷰어가 다른 형태를 기대하면 재작업원형으로 결정한 근거를 PR 설명에 명시, 채움 계산은 형태 독립적인 gauge-math.ts로 분리해 교체 비용 최소화
SVG 원형 게이지의 흔한 버그: strokeLinecap="round" + 100% 채움 시 시작점에 점처럼 겹쳐 보임100% 달성 시 시각적 결함수동 검증 단계에서 100% 케이스를 반드시 육안 확인 — 필요 시 100%에서만 strokeLinecap="butt"로 전환
레포에 React 컴포넌트 렌더 자동 테스트 인프라(jsdom/RTL)가 없어 4개 컴포넌트의 JSX 출력 자체는 자동 검증되지 않음회귀를 사람이 놓칠 가능성이번 이슈 범위에서 계산 로직만 단위 테스트하고 렌더는 수동 확인으로 커버 — RTL 도입은 더 큰 별도 결정으로 남겨둠
#48(카드/뱃지/상태 pill)이 같은 src/components/common/ 디렉터리를 병행 작업 중두 PR이 먼저 머지되는 쪽 기준으로 디렉터리 컨벤션이 굳어짐배럴 인덱스를 만들지 않아 파일 단위로만 존재 — 실제 파일명 충돌 가능성 낮음(Gauge/Spinner/ErrorPanel/EmptyState vs Card/Badge/StatusPill)
ErrorPanel/EmptyState가 실제로 어떤 조건에서 뜨는지(401 구분 등)는 #42/#46 스코프라 아직 미정지금 만드는 props가 실사용 시 부족할 수 있음순수 프레젠테이션(런타임 결합 없음)으로 설계해 두어 #46 완성 후 소비 측에서 props만 채우면 되도록 함
