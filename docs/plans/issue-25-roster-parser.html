1요약

버니스명단 탭의 원시 2D 배열(이름 | 상태)을 순수 함수로 파싱해 shared/domain.ts의 Player[]로 변환한다. 상태 정규화는 이미 머지된 normalizeStatus(#20)를 그대로 재사용하고, 새로 만드는 건 행 위치를 신원으로 다루는 파서 로직뿐이다. 핵심은 "행 위치 보존" — 명단 탭은 정렬·삭제가 금지되고 회차 탭이 행 번호로 이름을 참조하므로, 파서가 중간에 빈 행을 만나거나 이상값을 걸러내더라도 이후 행의 id가 실제 시트 행 위치와 어긋나면 안 된다. 시트 연동 없이 픽스처만으로 개발 가능(이슈 요구사항 그대로).

2현황 분석

  • 파서 없음, 재사용할 계약은 이미 존재 — functions/lib/에는 normalize-status.ts·normalize-score.ts·sheetTabs.ts만 있고 명단을 도메인 타입으로 바꾸는 코드는 없다. shared/domain.ts:27-31의 Player { id: number; name: string; status: PlayerStatus }가 목표 출력 타입이고, 주석(shared/domain.ts:24-26)이 이미 "id는 행 위치 기반 파생값(1부터, 헤더 제외)"이라고 명시한다.
  • normalizeStatus 시그니처 — functions/lib/normalize-status.ts:9-16: (raw: string | null | undefined) => { kind: 'known'; value: PlayerStatus } | { kind: 'unknown'; raw: string }. NFKC + trim을 내부에서 처리하므로 파서는 원본 상태 셀 문자열을 그대로 넘기기만 하면 된다. 트림 전 원본을 raw로 보존하는 동작까지 이미 테스트됨(normalize-status.test.ts:49-51).
  • "모르는 값을 조용히 버리지 않는다" 컨벤션이 이미 확립돼 있음 — functions/lib/sheetTabs.ts:53-86의 classifySheetTabs가 매칭 안 되는 탭 이름을 unclassified 배열로 노출하고(sheetTabs.test.ts:45-50), normalize-score.ts는 이상값을 { kind: 'invalid'; reason }으로 표현한다(예외 throw 없음). 명단 파서도 이 패턴을 따르는 게 자연스럽다 — 아래 3절에서 구체화.
  • 실제 시트 모양은 확정·시딩 완료 — docs/sheet-integration.html:120-126과 scripts/seed-sheet.mjs:28-36: 헤더 이름 | 상태, 데이터 21행(선수1~선수21), 상태 열은 드롭다운으로 4종 제한. 다만 docs/sheet-rules.html:111-115가 "표 중간에 빈 행 넣지 않기"를 절대 규칙으로 걸어뒀는데도 이슈 완료 조건이 "빈 행 케이스"를 명시적으로 요구한다 — 운영 규칙은 사람이 지켜야 할 규칙이고, 파서는 그 규칙이 깨졌을 때도 죽지 않아야 한다는 뜻으로 해석.
  • 드롭다운은 데이터 검증이지 강제가 아님 — 구글 시트 드롭다운은 기본적으로 "경고만" 모드라 붙여넣기 등으로 4종 밖 문자열이 들어올 수 있다. normalizeStatus가 이미 unknown 케이스를 값으로 표현해주므로, 파서는 이걸 Player.status: PlayerStatus 타입 계약에 맞게 처리해야 한다(무시하면 타입을 위반한 채로 억지로 끼워 넣게 됨).
  • 이름 매칭 요구사항 — 이슈 본문 "이름 NFC 정규화(회차 탭 이름과 매칭 기준)"은 sheetTabs.ts:4-8이 탭 이름에 대해 이미 겪은 것과 같은 문제(Sheets API가 한글을 NFD로 돌려줄 수 있음)를 이름 셀에도 적용하라는 뜻. 단 sheetTabs.ts는 원본(비정규화) 문자열을 반환값으로 유지해야 했지만(Range 참조 때문), 명단의 Player.name은 Range 참조에 쓰이지 않는 순수 데이터 값이라 정규화해서 반환해도 무방 — 오히려 정규화해야 이후 회차 탭에서 얻는 이름과 정확히 비교 가능.

3설계

3-1. 반환 타입 — players + issues

새 파일 functions/lib/roster.ts에 다음을 추가한다(신규 타입이므로 shared/domain.ts는 건드리지 않음):

export interface RosterRowIssue {
  rowIndex: number // 1-based, 헤더 제외 원본 행 위치 (시트 행 번호 - 1)
  reason: string
}

export interface RosterParseResult {
  players: Player[]
  issues: RosterRowIssue[]
}

export function parseRoster(rows: string[][]): RosterParseResult

rows는 헤더를 포함한 원시 2D 배열(이슈 본문 표현 그대로) — 첫 행을 스킵하고 나머지를 데이터 행으로 다룬다. id·rowIndex는 데이터 행 배열의 원본 인덱스+1이라 항상 시트의 실제 행 위치(헤더 제외, 1부터)와 일치한다.

3-2. 행 분류 규칙

  • 완전 빈 행(이름·상태 셀 모두 공백/미존재) → players에도 issues에도 넣지 않고 조용히 스킵. 단 다음 유효 행의 id는 스킵으로 인해 당겨지지 않는다(원본 인덱스를 그대로 쓰므로 자동으로 보장됨) — 이게 "행 위치 보존" 완료 조건의 핵심이자 이 파서에서 가장 깨지기 쉬운 지점.
  • 이름 없음(상태만 있음) → issues에 '이름 없음'으로 기록, 그 행은 players에서 제외.
  • 상태 없음(이름만 있음) 또는 상태값이 4종 밖(normalizeStatus가 unknown 반환) → issues에 사유 기록(각각 '상태 없음' / 알 수 없는 상태값: "..."), players에서 제외.
  • 그 외 → { id: rowIndex, name: 이름.trim().normalize('NFC'), status: normalizeStatus 결과값 }을 players에 추가. 상태 4종(활동/탈퇴/비대상/휴식) 모두 여기 포함 — "탈퇴 = 사이트 전체 숨김" 같은 정책 필터링은 RecordsResponse.players: PlayerSummary[]를 구성하는 이후 단계(별도 이슈)의 책임이지, 명단을 있는 그대로 옮기는 이 파서의 책임이 아니다(이슈 완료 조건의 "상태 4종 혼합" 테스트가 4종 전부를 players에서 확인 가능해야 한다는 뜻으로 해석).

3-3. 기각한 대안

예외(throw)로 처리 — 순수 함수이긴 하지만 입력이 외부 시트 데이터라는 경계값이라, normalizeStatus·normalizeScore·classifySheetTabs 전부 예외 대신 판별 유니온/버킷으로 결과를 표현하는 기존 컨벤션을 따르는 게 일관적이다.
빈 행을 만나면 id를 연속 재계산(compact) — 회차 탭이 ='버니스명단'!A{n}처럼 절대 행 위치로 참조하므로, id를 당겨버리면 이후 정상 행들의 id가 실제 시트 행과 어긋나 잘못된 매칭의 원인이 된다. 원본 인덱스를 그대로 쓰는 쪽이 안전.
알 수 없는 상태값을 가장 가까운 4종으로 fallback — 예: '활동중' → '활동'으로 추측 매핑. 조용히 잘못된 데이터를 만들 위험이 있고, normalizeStatus가 이미 unknown으로 명시적으로 표현해주는 정보를 버리게 된다.
🔧 확인 필요 — issues 필드는 이슈 완료 조건에 명시되지 않은 추가 설계
완료 조건은 "정상 21명 · 상태 4종 혼합 · 빈 행 케이스" 3개 픽스처와 "행 위치 보존"만 명시한다. 다만 Player.status: PlayerStatus 타입 계약과 normalizeStatus의 unknown 반환을 동시에 만족하려면 이상값 행을 어딘가에는 표현해야 하는데, classifySheetTabs의 unclassified와 같은 패턴(정보를 조용히 버리지 않기)을 그대로 가져온 것이다. 과한 설계라고 판단되면 issues를 빼고 이상값 행을 그냥 players에서 누락시키는 형태(무시)로 단순화할 수도 있다 — 승인 시 어느 쪽으로 갈지 알려주면 그대로 진행한다.

5리스크

리스크영향완화책
Sheets API가 행마다 trailing 빈 셀을 생략해 row.length가 들쭉날쭉함(예: 상태가 빈 행은 ['선수7'] 1칸짜리로 옴)배열 인덱스 접근 시 undefined 처리 누락하면 런타임 오류row[i] ?? '' 방어적 접근, 짧은 배열 픽스처를 테스트에 포함
드롭다운이 강제가 아니라 실제 데이터에 4종 밖 상태값(오타·공백)이 섞일 수 있음그대로 통과시키면 Player.status: PlayerStatus 타입 계약 위반 위험normalizeStatus의 unknown 판정을 받은 행은 players에서 제외(타입 안전 유지), issues로만 노출
빈 행 스킵 로직을 잘못 구현하면(예: 연속 카운터 사용) id가 실제 시트 행과 어긋남이후 회차 파서가 playerId로 잘못된 선수를 가리키는 심각한 데이터 무결성 버그원본 배열 인덱스(i + 1)를 항상 사용, 이를 명시적으로 검증하는 테스트 케이스 필수 포함
이름 NFC 정규화가 실제로 회차 탭 이름과 매칭에 쓰이는지는 이 이슈만으로는 확인 불가(회차 파서가 아직 없음)정규화 기준이 이후 이슈에서 다를 경우 재작업 가능성이번 이슈는 "이름을 NFC로 정규화해 반환"까지만 보장 — 실제 매칭 검증은 회차 탭 파서 이슈(범위 밖)로 명시적으로 이관

6테스트 계획

  • 정상 21명: scripts/seed-sheet.mjs와 같은 형태의 21행(전원 활동) 픽스처 → players.length === 21, id가 1~21 순서대로, 이름이 원본과 일치.
  • 상태 4종 혼합: 21명 중 일부를 탈퇴·비대상·휴식으로 바꾼 픽스처 → 각 행의 status가 정확히 매핑되고 4종 모두 players에 포함(탈퇴 필터링은 이 파서 책임이 아님을 확인).
  • 빈 행 케이스: 중간(예: 5번째 데이터 행)에 완전 빈 행을 넣은 픽스처 → 그 행은 players·issues 어디에도 없고, 다음 유효 행의 id가 원래 시트 행 위치(스킵된 행 다음 번호)를 그대로 유지(연속 카운트로 당겨지지 않음) — 완료 조건 "행 위치 보존"의 핵심 단언.
  • 결측/이상값: 이름만 있고 상태 없음, 상태만 있고 이름 없음, 4종 밖 오타 상태값(예: '활동중') 각각 → issues에 해당 rowIndex와 사유가 기록되고 players에서는 제외.
  • NFD 이름 정규화: normalize-status.test.ts:13-16과 같은 패턴으로 이름을 NFD로 분해한 픽스처 입력 → 결과 Player.name이 NFC로 정규화됐는지 확인.
