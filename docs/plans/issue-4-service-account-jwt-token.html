#4기능 개선 (신규 유틸)

P1 — 서비스계정 JWT 서명 유틸 + 액세스 토큰 캐시 (Workers Web Crypto)

damdam6/bernice 브랜치 feat/4-service-account-jwt-token ← main 라벨 없음 2026-07-13

1요약

Cloudflare Pages Functions(Workers 런타임)에는 Node crypto가 없어, 구글 서비스계정 인증에 필요한 RS256 JWT 서명을 crypto.subtle(Web Crypto)로 직접 구현해야 한다. 이 이슈는 서비스계정 키 JSON 파싱 → PKCS8 개인키 import → JWT 서명 → oauth2.googleapis.com/token 교환까지의 흐름을 functions/lib/googleAuth.ts에 getAccessToken(env, scope) 형태의 공용 유틸로 만들고, 발급된 액세스 토큰을 만료 전까지 인메모리로 재사용하는 캐시를 붙인다. 스코프를 파라미터로 받아 P1(읽기 전용)과 이후 P5(쓰기)에서 그대로 재사용할 수 있게 설계한다.

2현황 분석

functions/ 아래에는 현재 functions/api/health.ts:1 하나만 존재하며 단순 헬스체크(PagesFunction)만 제공한다. 서비스계정/구글 인증 관련 코드는 전혀 없다 — 신규 유틸.

기존 scripts/seed-sheet.mjs:69-84와 scripts/protect-sheet.mjs:30-42에 이미 "서비스계정 JWT → 액세스 토큰" 로직이 존재하지만, 둘 다 Node 전용 API인 node:crypto의 createSign('RSA-SHA256')을 사용한다. Cloudflare Pages Functions(Workers 런타임)에는 node:crypto가 없어 이 코드를 그대로 재사용할 수 없고, crypto.subtle 기반으로 새로 작성해야 한다 — 다만 JWT claim 구성(iss/scope/aud/iat/exp)과 base64url 인코딩 패턴은 그대로 참고할 수 있다.

시크릿 배선은 이미 준비돼 있다: wrangler.jsonc:12의 주석과 .dev.vars.example:5에 GOOGLE_SERVICE_ACCOUNT_KEY가 "서비스계정 키 JSON 전체를 한 줄로" 주입되는 형태로 문서화돼 있다. 로컬 secrets/sa-key.json(gitignore)이 이미 존재한다는 것도 스크립트 주석(scripts/seed-sheet.mjs:11, scripts/protect-sheet.mjs:8)에서 확인된다 — .dev.vars 생성 시 jq -c . secrets/sa-key.json 결과를 그대로 넣으면 된다.

functions/tsconfig.json:20은 "include": ["."]로 functions/ 아래 전체(서브디렉터리 포함)를 컴파일 대상으로 잡고, 루트 tsconfig.json:7이 이를 프로젝트 레퍼런스로 포함하므로 npm run typecheck(tsc -b)가 새 파일도 자동으로 검사한다.

Cloudflare Pages Functions는 onRequest*를 export하는 파일만 라우트로 등록한다 — 언더스코어 프리픽스 자체가 라우팅을 제외시키는 것은 아니고(특수 취급되는 건 _middleware 파일명뿐), 공용 유틸 파일은 onRequest*를 export하지 않으므로 어느 디렉터리에 두든 라우트가 생기지 않는다. 형제 PR(#14 등)이 이미 functions/lib/ 컨벤션을 쓰고 있어 그대로 맞춘다.

3설계

1. 파일 위치 — functions/lib/googleAuth.ts(형제 PR의 functions/lib/normalize-*.ts 컨벤션과 통일). Pages Functions는 onRequest*를 export하는 파일만 라우트로 등록하므로 — 언더스코어 프리픽스 여부와 무관하게 — 이 유틸은 애초에 라우트가 되지 않는다. 다음 이슈(Sheets API 읽기 래퍼)와 이후 P5(쓰기 유틸)에서 그대로 import한다.

2. JWT 서명 흐름(Web Crypto)

  • GOOGLE_SERVICE_ACCOUNT_KEY(JSON 문자열) → JSON.parse → { client_email, private_key } 추출, 누락 시 명확한 에러로 즉시 실패.
  • private_key(PEM, PKCS8)에서 헤더/푸터·개행 등 공백 문자 제거 → atob로 base64 디코드 → ArrayBuffer.
  • crypto.subtle.importKey('pkcs8', der, { name: 'RSASSA-PKCS1-v1_5', hash: 'SHA-256' }, false, ['sign'])로 서명키 import.
  • header({alg:'RS256',typ:'JWT'}) + claims({iss, scope, aud:'https://oauth2.googleapis.com/token', iat, exp: iat+3600})를 각각 JSON 직렬화 → UTF-8 바이트 → base64url 인코딩.
  • "{header}.{claims}" 서명 대상 문자열을 crypto.subtle.sign으로 서명 → base64url → 최종 JWT(header.claims.signature) 조립.

3. 액세스 토큰 교환 — POST https://oauth2.googleapis.com/token, body는 application/x-www-form-urlencoded로 grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer&assertion={jwt}. 실패 시 상태코드+응답 본문을 담은 에러를 throw.

4. 캐시 — 모듈 스코프의 Map<string, Promise<{ accessToken, expiresAt }>>를 scope별 키로 사용. getAccessToken(env, scope) 호출 시:

  • 캐시에 항목이 있으면 그 Promise를 await한 뒤 만료(여유분 적용) 전이면 그대로 재사용 — 동시에 들어온 호출도 같은 Promise로 수렴해 중복 발급을 막는다(단일 비행).
  • 없거나 만료됐으면 새로 발급하는 Promise를 즉시 캐시에 넣고(경합 시 단일 요청으로 수렴), 실패하면 캐시에서 제거해 다음 호출이 재시도할 수 있게 한다.
  • 만료 여유분(margin, 예: 5분)을 두고 실제 만료보다 일찍 갱신 — Google 액세스 토큰 수명은 보통 3600초(expires_in).

반려한 대안

  • Workers Cache API/KV로 인스턴스 간 캐시 공유 — 이슈 범위가 "인메모리 캐시"로 명시돼 있고, 초기 트래픽 규모(팀 내부용)에서는 인스턴스별 캐시로 충분해 과설계. 필요해지면 후속 이슈로 분리.
  • 서드파티 JWT 라이브러리 도입 — Workers 런타임 호환성 검증 비용과 의존성 추가 대비, RS256 서명 로직 자체가 짧아 직접 구현이 더 단순하고 기존 스크립트 패턴과도 일관됨.

4작업 단계

  1. base64url 인코딩 헬퍼(bytes/JSON), PEM→ArrayBuffer 변환, importPrivateKey, signServiceAccountJwt(RS256), issueAccessToken(oauth2 토큰 교환), getAccessToken(env, scope)(단일 비행 + 만료 캐시)를 담은 functions/lib/googleAuth.ts를 새로 작성한다. Env는 export해 후속 라우트가 재사용할 수 있게 한다. functions/lib/googleAuth.ts
  2. .dev.vars.example을 복사해 로컬 .dev.vars를 만들고, GOOGLE_SERVICE_ACCOUNT_KEY에 jq -c . secrets/sa-key.json 결과를 채운다(둘 다 gitignore돼 커밋되지 않음). .dev.vars (미커밋)
  3. 임시 검증 경로(functions/api/health.ts를 잠시 확장하거나 임시 디버그 라우트)에서 getAccessToken(env, 'https://www.googleapis.com/auth/spreadsheets.readonly')를 2회 연속 호출해 access_token 수신을 확인하고, console.log로 발급 시점을 찍어 두 번째 호출이 재발급 없이 캐시로 처리되는지 확인한다. 확인 후 임시 코드는 되돌린다. functions/api/health.ts (임시, 최종 커밋 대상 아님)
  4. npm run typecheck(tsc -b)로 신규 파일에 타입 오류가 없는지 확인한다. vitest 회귀 테스트(functions/lib/googleAuth.test.ts)도 함께 작성 — 특히 캐시 만료 시점의 동시 호출 레이스는 회귀 테스트로 고정해둔다. functions/lib/googleAuth.ts, functions/lib/googleAuth.test.ts

  • 스코프를 파라미터로 받아 캐시 키로 사용 — P1은 spreadsheets.readonly, 이후 P5 쓰기 스코프도 그대로 재사용 가능.
