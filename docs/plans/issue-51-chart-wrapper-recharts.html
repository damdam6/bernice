1요약

P3 공통 프론트 컴포넌트 4종(#48~#51) 중 하나로, recharts를 설치하고 이후 모든 화면(개인 프로필의 레이더·추이 라인, 전체 추이의 멀티라인)이 공유할 차트 프레젠테이션 계층을 만든다. 반응형 ResponsiveContainer 래퍼 하나, src/index.css의 디자인 토큰(브랜드 오렌지·액센트 블루·good/warn/bad)을 그대로 참조하는 공통 색·툴팁 테마, 그리고 라인·레이더 프리셋 2종을 구현한다. 의존성이 선언되지 않은 독립 이슈이고 실제 화면(개인 프로필 등)은 아직 없으므로, 이번 범위는 도메인 데이터에 묶이지 않는 재사용 가능한 제네릭 컴포넌트를 목데이터로 검증하는 것까지다 — 실제 /api/records 데이터를 이 프리셋에 꽂는 작업은 P4 개인 프로필·전체 추이 화면 이슈에서 진행한다.

2현황 분석

src/는 아직 P0 스캐폴딩 결과물뿐이다 — src/App.tsx:1(헬스체크 placeholder), src/main.tsx, src/index.css:1(Tailwind v4 @theme 토큰)뿐이고 src/components/ 디렉터리 자체가 없다. P3의 형제 이슈 #44(라우팅) · #45(레이아웃) · #46(TanStack Query) · #48~#50(카드/게이지/랭킹 테이블)에 대응하는 브랜치가 이미 로컬에 존재하지만(feat/44-routing-skeleton 등) 전부 main과 diff가 없는 빈 워크트리다 — 즉 참고할 기존 컴포넌트 컨벤션이 아직 없는 그린필드 상태다.

package.json:14-24에는 recharts가 없다. 최신 버전은 3.9.2이고 peerDependencies가 react ^19.0.0을 포함해 현재 React 19.2 스택과 호환된다.

디자인 토큰은 src/index.css:5-32의 Tailwind v4 @theme 블록에 있다 — 브랜드 오렌지 스케일(--color-brand-50~950), "차트 대비색"이라고 명시된 보조 블루(--color-accent-400/500/600), 시맨틱 --color-good/warn/bad. Tailwind v4의 @theme는 이 값들을 실제 CSS 커스텀 프로퍼티로도 방출하므로, var(--color-brand-500) 같은 참조를 SVG stroke/fill에 그대로 써도 브라우저에서 정상 동작한다 — 즉 차트 색상 상수를 새 hex로 중복 정의하지 않고 이 토큰을 그대로 가리키면 index.css가 단일 진실 소스로 유지된다.

실제 앱 UI 톤은 src/App.tsx:29처럼 밝은 카드(흰 배경, border-brand-200, rounded-2xl, shadow-sm)다 — docs/*.html 설계 문서들의 어두운 문서용 스타일(--bg:#0f1115 등)과는 무관하니 착각하지 않는다. 차트 테마는 App.tsx 쪽 밝은 카드 톤을 따라야 한다.

도메인 모델(shared/domain.ts)에는 라인 프리셋이 소비할 PlayerEventTrend.points (shared/domain.ts:92-104, 회차별 value/display/achieved)와, 레이더 프리셋이 훗날 조합할 종목별 값들이 있다. 다만 docs/index.html:234에 "랭킹·추이는 종목별로 원값 그대로 다룹니다(종합 점수·정규화 없음). 개인 프로필의 레이더 차트는 종목별 값을 축으로 표시"라고 명시돼 있다 — 셔틀런(초 단위 시간) · 골밑슛/자유투/패스캐치(개수)처럼 종목마다 단위가 다르므로, 레이더의 공유 반지름 축에 원값을 그대로 얹으면 시각적으로 비교가 안 된다(75초 vs 5/10개). 이 축별 정규화 방식(예: 목표 대비 퍼센트) 결정은 아직 존재하지 않는 개인 프로필 화면 이슈의 몫이다 — 이번 이슈는 그 결정을 내리지 않고, domain prop을 열어둔 제네릭 레이더 프리셋만 만든다.

테스트 인프라: vitest.config.ts:5가 environment: 'node'이고 package.json에 @testing-library/react·jsdom이 없다 — 지금까지의 모든 테스트(functions/lib/*.test.ts)는 순수 백엔드 로직이고, 프론트 컴포넌트 렌더 테스트 관례가 레포에 아직 없다.

3설계

  • 레이어 분리: src/components/charts/를 새 디렉터리로 만든다. 파일명은 functions/lib/ 전반(compute-rankings.ts, parse-session.ts 등)에서 이미 쓰는 케밥 케이스를 그대로 따른다 (App.tsx/main.tsx는 프레임워크 기본 진입점이라 예외로 둔다) — 레포에 통일된 파일명 관례를 만든다.
  • 컴포지션: ChartContainer(순수 ResponsiveContainer 래퍼, 기본 높이/마진만 담당) 위에 LineTrendChart·RadarProfileChart 두 프리셋을 얹는다. 이슈 제목의 세 항목(반응형 컨테이너 / 공통 테마 / 프리셋 2종)을 그대로 세 계층으로 대응시킨 것 — 프리셋이 각자 ResponsiveContainer를 중복 구현하지 않는다.
  • 제네릭 vs 도메인 결합: 두 프리셋 모두 shared/domain.ts 타입을 직접 import하지 않고 xKey/series[] 기반의 범용 props를 받는다. 개인 추이(라인 1개) · 전체 추이(멀티라인) · 레이더 모두 같은 프리셋으로 커버되게 하려는 목적 — 실제 PlayerEventTrend → 차트 데이터 변환은 그걸 쓰는 화면(P4) 쪽 책임으로 남긴다. 값 포맷팅(시간 1'15" vs 개수)도 yFormatter/ tooltipFormatter 콜백으로 호출자에게 위임 — 래퍼가 종목별 표시 규칙을 하드코딩하지 않는다.
  • 카드 크롬 제외: ChartContainer는 카드 테두리/배경을 그리지 않는다. #48(카드·뱃지·상태 pill)이 별도 이슈로 예정돼 있고 #51은 그 이슈와 "의존성 없음"으로 명시돼 있으므로, 카드 크롬은 소비하는 페이지가 #48의 Card로 감싸는 것으로 역할을 나눈다 — 두 이슈가 같은 책임을 중복 구현하지 않게 한다.
  • 공통 테마 소스: chart-theme.ts가 var(--color-brand-500) 등 CSS 변수 문자열과 카테고리 색 배열(멀티라인·멀티시리즈 레이더용), 그리드/축 스타일 상수를 export한다. 새 hex 상수를 만들지 않고 index.css 토큰만 참조한다.
  • 공통 툴팁: Recharts <Tooltip content={...}>에 꽂는 ChartTooltip 커스텀 컴포넌트 하나를 두 프리셋이 공유한다(흰 배경, border-brand-200, rounded-lg, 계열별 색 점 + 라벨 + 포맷된 값) — App.tsx 카드 톤과 통일.
  • 테스트 전략: 이슈의 완료 조건 자체가 "목데이터로 렌더" · "컨테이너 크기 변화에 반응"이라는 수동/시각 검증 문구다. 레포에 프론트 렌더 테스트 인프라(jsdom·RTL)가 아직 없고, Recharts의 ResponsiveContainer는 실제 ResizeObserver/레이아웃에 의존해 jsdom 목킹 비용이 크다. 이번 이슈에서 새 테스트 인프라를 부수적으로 들이지 않고, npm run dev 개발 서버에서 목데이터로 수동/시각 검증하는 쪽을 택한다 — 자동화 테스트 인프라 도입은 필요해지면 별도 이슈로 분리한다.

4작업 단계

  1. recharts 설치 (최신 3.x, React 19 peer 호환 확인됨). package.json, package-lock.json
  2. 공통 색/스타일 테마 모듈 작성 — index.css의 --color-brand-*· --color-accent-*·--color-good/warn/bad를 var(...) 문자열로 참조하는 카테고리 색 배열, 그리드/축/범례 스타일 상수 export. src/components/charts/chart-theme.ts
  3. 공통 툴팁 컴포넌트 — 라벨 + 계열별 색 점/이름/값(포맷 콜백 위임)을 흰 카드 톤으로 렌더. src/components/charts/chart-tooltip.tsx
  4. 반응형 컨테이너 래퍼 — ResponsiveContainer 얇은 래퍼, 기본 높이/마진 prop. src/components/charts/chart-container.tsx
  5. 라인 프리셋(추이용) — 단일/멀티 시리즈 라인, CartesianGrid·축·범례(2개 이상일 때만)· 공통 툴팁 적용, xFormatter/yFormatter 콜백 지원. src/components/charts/line-trend-chart.tsx
  6. 레이더 프리셋(개인 프로필용) — 축별 라벨(dataKey="axis") · PolarGrid/ PolarAngleAxis/PolarRadiusAxis(domain prop, 기본 'auto') · 단일/멀티 시리즈 Radar · 공통 툴팁. 단위가 다른 원값을 그대로 얹으면 비교가 안 된다는 점을 JSDoc 주석으로 명시(정규화는 호출자 책임). src/components/charts/radar-profile-chart.tsx
  7. 배럴 export. src/components/charts/index.ts
  8. 검증용 목데이터로 두 프리셋을 임시로 마운트 — docs/index.html의 더미 표(선수1~6 × 셔틀런/골밑슛/자유투/패스캐치)를 재사용해 라인(개인 추이 1명 + 멀티라인 전체 추이 각각) · 레이더(4축) 양쪽을 npm run dev로 렌더 확인 → 브라우저/디바이스 툴바로 컨테이너 리사이즈 반응 확인 → 검증 후 임시 마운트 코드는 커밋에서 제거(App.tsx는 P0 placeholder 상태 유지). 임시 검증 스니펫(커밋 대상 아님)
  9. npm run typecheck && npm test로 회귀 확인. tsconfig.app.json

5리스크

리스크영향완화책
레이더 축별 단위 불일치(시간 vs 개수) 미래의 개인 프로필 화면이 원값을 그대로 넣으면 시각적으로 왜곡된 스킬 프로필이 나옴 이번 프리셋은 정규화하지 않고 domain prop과 JSDoc으로 호출자에게 책임을 명시 — 실제 정규화 전략(목표 대비 %? 순위 percentile?)은 그 화면 이슈에서 결정
차트 완료 조건("렌더"·"리사이즈 반응")이 자동화 테스트가 아닌 수동 검증 문구 PR에 남는 자동 테스트가 없어 회귀 방지력이 약함 레포에 프론트 렌더 테스트 인프라가 전무한 상태에서 이번 이슈로 jsdom/RTL을 부수적으로 들이지 않기로 결정 — PR 설명에 수동 검증(스크린샷/리사이즈 확인) 근거를 남기고, 인프라 도입 필요성은 사용자에게 별도 판단 요청
var(--color-brand-500) 같은 CSS 변수 참조가 Tailwind v4 빌드 산출물에서 기대대로 방출되지 않을 가능성 차트 색이 비어보이거나(투명) 기본색으로 폴백 구현 중 npm run dev에서 실제 렌더 색을 눈으로 1차 확인 — 문제 시 hex 상수로 폴백하되 index.css 값과 동일하게 유지
#48(카드)·#50(랭킹 테이블) 등 형제 이슈가 이후 병렬로 진행되며 컴포넌트 디렉터리 관례가 갈릴 가능성 같은 src/components/ 하위에서 파일명·구조 스타일이 이슈마다 달라짐 이번 이슈에서 세운 케밥 케이스 파일명 · src/components/<도메인>/ 하위 배럴 export 구조를 향후 이슈에도 그대로 참고하도록 PR 설명에 명시

6테스트 계획

  • npm run typecheck — 새 컴포넌트들이 tsconfig.app.json의 strict 옵션들을 통과하는지 확인.
  • npm test — 기존 functions/lib/*.test.ts 전체가 이번 변경으로 깨지지 않는지 확인(프론트 전용 변경이라 영향 없어야 함).
  • npm run dev 개발 서버에서 목데이터(도메인 로직 없이 하드코딩) 기준으로 라인 차트(단일 시리즈 + 멀티 시리즈)와 레이더 차트(4축, 단일 시리즈)가 실제로 그려지는지 브라우저로 확인.
  • 브라우저 창 크기 조절 / 디바이스 툴바 폭 변경 시 두 차트가 즉시 리사이즈에 반응하는지(레이아웃 깨짐·오버플로 없는지) 확인.
  • 브랜드 오렌지·액센트 블루 색이 실제로 SVG에 적용되는지(투명/기본색 폴백 없는지) 육안 확인.
