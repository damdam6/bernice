1요약

src/pages/Rankings.tsx를 P4 플레이스홀더에서 확정 카드 행 리스트 화면으로 구현한다. #69가 만든 useRecords·buildPerformanceScale·PerformanceBar·AchievementBadge를 처음으로 실제 화면에 배선하고, 종목·회차 필터 칩(신규 공통 컴포넌트)과 순위 행 카드를 새로 만든다. 순위권(rankings[].events[].entries)은 그대로 쓰고, 면제·미측정·이상값은 sessions[].entries에서 활동 상태 선수만 보충해 하단에 붙인다 — API 계약 변경 없음. 구 테이블 컴포넌트(RankingTable.tsx, #50)는 사용처가 없어 새 카드 컴포넌트로 대체하며 삭제한다.

3설계

① 행 데이터 모델 — 구 RankingTableRow 판별 유니온을 확장해 invalid를 추가한다: { status: 'recorded', ...RankingEntry } | { status: 'exempt'|'unmeasured', playerId, name } | { status: 'invalid', playerId, name, display }.

② 병합 알고리즘 (buildRankingRows) — 입력은 선택된 종목의 EventRanking + 선택된 회차의 Session + 활동 선수 Set(players[]에서 파생, compute-home-summary.ts:18와 동일 방식):

  • 순위권 행 = eventRanking.entries 그대로(이미 rank 오름차순).
  • 보충 행 = session.entries를 순회하며 (활동 Set에 있고 AND 이미 순위권에 없는) 선수만, scores[eventKey].status(exempt/unmeasured/invalid)로 매핑. session.entries 기준으로 순회하므로 그 회차에 엔트리 자체가 없는 선수는 자동으로 제외된다(현황 분석 참고) — 별도 가드 불필요.
  • 보충 행 순서는 session.entries 원래 순서를 그대로 유지한다(재정렬하지 않음) — compute-rankings.ts의 "동점자는 입력 순서 유지" 원칙과 같은 결의 선택.
  • 동점 판정(findTiedRanks)은 구 RankingTable.tsx 로직을 그대로 이식 — recorded 행만 대상으로 rank별 개수를 세어 2개 이상이면 "공동" 처리.

③ 상태별 표시 규칙 — 아래 표가 RankingRow 컴포넌트의 전체 명세다. 뱃지 3종(면제/미측정/이상값)은 전부 neutral 톤이라 별도 컴포넌트 없이 RankingRow.tsx 안에서 라벨 맵 하나로 Pill을 직접 호출한다.

④ 종목 안내문 공식 — PRD §05의 두 예시("목표 1'17" 이내 · 낮을수록 좋음 ↓" / "목표 5개 이상 · / 10")를 maxScore 유무와 direction 두 축만으로 재현한다:

  • 목표 문구: valueKind==='count'면 목표 {target}개, 아니면(time) 목표 {target}.
  • 이내/이상: direction==='낮을수록' → "이내", '높을수록' → "이상".
  • 꼬리말: maxScore != null → · / {maxScore}, 없으면 → · {"낮을수록 좋음 ↓" 또는 "높을수록 좋음 ↑"}.
  • 검증: 드리블셔틀런(time·maxScore 없음·낮을수록·target "1:17") → "목표 1:17 이내 · 낮을수록 좋음 ↓". 골밑슛(count·maxScore 10·높을수록·target "5") → "목표 5개 이상 · / 10" — PRD §05 예시와 일치(목업의 아포스트로피 "1'17""는 스타일링으로 보고 원본 target 문자열을 그대로 사용).

⑤ "n차" 라벨 — sessions는 이미 날짜 오름차순이 보장되므로(domain.ts) sessions.map((s, i) => `${i+1}차`)를 Rankings.tsx 안에 인라인으로 둔다. 공유 유틸로 추출하지 않는 이유는 §02에 적었듯 현재 소비처가 하나뿐이라서다.

⑥ 기본 선택값 — 회차는 최신(sessions.at(-1)), 종목은 첫 번째(events[0]). useState<string | null>(null) + selected ?? default 폴백으로 구현해 데이터 로딩 완료 시점을 맞추는 useEffect를 따로 두지 않는다.

⑦ 새 파일 배치 — 기존 컨벤션을 그대로 따른다: FilterChip은 화면 전용이 아닌 공통 원자 컴포넌트라 Card.tsx/Pill.tsx와 같은 층위(src/components/)에, 순수 병합/파생 로직(buildRankingRows·findTiedRanks·buildEventGuidance)은 performance-scale.ts와 같은 층위(src/lib/)에 ranking-view.ts 한 파일로 묶는다. 카드 1행 프레젠테이션(RankingRow)은 구 RankingTable.tsx를 대체하므로 같은 자리(src/components/)에 둔다.

⑧ 에러·빈 상태 — useRecords 로딩 중엔 Spinner, 에러(401 포함)는 ErrorPanel(재시도=refetch), sessions.length===0 || events.length===0이면 칩·안내문을 렌더하지 않고 EmptyState, 선택된 회차·종목의 병합 행이 0건이면 구 RankingTable의 "표시할 기록이 없습니다" 카피를 그대로 계승해 보여준다.

기각안 — 구 RankingTable.tsx를 <table> 유지한 채 CSS만 카드처럼 패치하는 안: invalid 지원과 활동 필터 보강이 어차피 병합 로직 전체를 다시 요구해서 이득이 없고, "카드 행 리스트"라는 확정 디자인과 시맨틱(표 vs 리스트)이 어긋나 접근성 트리도 부자연스러워진다.
