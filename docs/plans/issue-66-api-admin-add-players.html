#66기능 개선

P5 — POST /api/admin/add-players (기존 회차에 참가자 추가)

damdam6/bernice 브랜치 feat/66-api-admin-add-players ← main 라벨 enhancement 2026-07-19

1요약

관리자가 이미 존재하는 회차 탭에 늦게 합류한 활동 선수들을 탭 맨 아래에 추가하는 쓰기 API를 신설한다. 각 행은 이름 참조 수식(='버니스명단'!A{행})만 채우고 점수 칸은 빈 채로 둔다 — 개명이 자동 반영되고 이후 점수 입력(#65 계열)의 전제인 "참가자 등록"을 완성한다. 이미 그 회차에 있는 선수는 후보 단계에서 제외되며, API도 요청에 섞여 들어오면 거부한다. 쓰기 성공 시 /api/records 캐시를 무효화해 팀 열람에 즉시 반영한다. 순수 검증·행 조립 로직은 functions/lib로 분리해 vitest로 단위 검증하고, 엔드포인트는 I/O 오케스트레이션만 맡는다.

2현황 분석

  • 인증·라우팅은 이미 준비됨. functions/api/_middleware.ts:37이 /api/admin/*를 admin 세션으로 강제하고, 함수 파일이 없어도 미들웨어가 먼저 걸려 401·403 계약이 성립한다 — 새 라우트 functions/api/admin/add-players.ts를 얹기만 하면 된다.
  • 회차 탭 A열 = 이름 참조 수식. 시딩 스크립트가 이름 열을 ='버니스명단'!A{n} 수식으로, valueInputOption=USER_ENTERED로 기록한다(scripts/seed-sheet.mjs:143). 점수 블록은 참여자만 B2부터 RAW로 채운다(scripts/seed-sheet.mjs:149). 즉 "참가자 추가"는 A열에 수식 1칸을 새 행으로 붙이는 일이다.
  • 선수 신원 = 명단 행 위치(id)와 이름. functions/lib/roster.ts:67에서 Player.id = 헤더 제외 행 위치이므로, 명단 시트 실제 행 = id + 1. 참조 수식은 ='버니스명단'!A{id+1}. 회차 참가 여부는 시스템 전반이 이름으로 식별한다(functions/lib/parse-session.ts:77의 이름 매칭, PRD §07 "id → 명단 이름 → 회차 탭 이름 매칭").
  • 쓰기 래퍼는 RAW 고정. functions/lib/sheetsWriteApi.ts:28 updateValues가 valueInputOption=RAW를 하드코딩한다. RAW로는 =로 시작하는 문자열이 수식이 아니라 리터럴 텍스트로 저장돼 참조가 죽는다 — 수식을 쓰려면 USER_ENTERED가 필수다. 429·5xx 재시도(멱등)·4xx 즉시 실패 로직은 그대로 재사용 가능.
  • 읽기 프리미티브도 준비됨. functions/lib/sheetsApi.ts의 getSpreadsheetTabTitles·batchGetValues·quoteSheetName, functions/lib/sheetTabs.ts의 classifySheetTabs로 탭 분류·A1 참조 조립이 이미 검증돼 있다.
  • 캐시 무효화 계약. functions/lib/records-cache.ts:7 RECORDS_CACHE_KEY를 caches.default.delete()로 지우면 다음 /api/records 조회가 미스→재조립된다(functions/api/records.ts:17와 같은 키).
  • 프론트는 아직 스텁. src/pages/admin/AddPlayers.tsx는 "#67에서 채워집니다" 플레이스홀더 — 이 이슈는 백엔드 API만이며 UI는 범위 밖. 요청 계약은 PRD의 POST /api/admin/records(sessionDate + playerId)를 그대로 확장해 정한다.

3설계

계약 (PRD /api/admin/records 스타일 확장):

POST /api/admin/add-players
{
  "sessionDate": "2025-08-16",   // 회차 탭 이름 (YYYY-MM-DD)
  "playerIds": [7, 8, 9]         // 추가할 활동 선수 id (버니스명단 행 위치)
}

→ 200
{
  "sessionDate": "2025-08-16",
  "added": [ { "playerId": 7, "name": "선수7" }, { "playerId": 8, "name": "선수8" } ]
}

에러 계약:

쓰기 파이프라인 (PRD §07 append-only 원칙): ① 탭 목록 조회 → classifySheetTabs → 명단 탭 + 대상 회차 탭 확인(없으면 404). ② 두 탭을 batchGetValues 1회로 읽어, 명단은 parseRoster로, 회차 탭은 A열로 기존 참가자 이름 집합 + 맨 아래 행 번호 산출. ③ 순수 함수 buildAddPlayersPlan이 검증(존재·활동·중복·이미 참가)을 수행하고 쓸 행/범위/added를 돌려주거나 타입 에러를 던짐. ④ updateValues(..., 'USER_ENTERED')로 '{탭}'!A{start}:A{end}에 수식 행 1회 쓰기(멱등). ⑤ 캐시 무효화 → 200.

핵심 결정 & 기각안:

  • 수식 쓰기 = USER_ENTERED. 시딩과 동일. RAW(기각)는 =를 리터럴로 저장해 참조가 죽는다. USER_ENTERED의 1:15 시각 자동해석 리스크는 점수 셀에만 해당하며 add-players는 점수를 안 쓰므로 무관.
  • append = update(계산된 바닥 행). values:append(기각)는 재시도 시 중복 append 위험(비멱등). PRD §07은 회차 탭이 append-only라 "조회한 행 번호로 그대로 update"가 안전·멱등하다고 명시 — 기존 래퍼의 재시도 정책과 정확히 맞물린다.
  • 참가 여부 판정 = 이름. 시스템 전반(parse-session, PRD §07)이 회차 참가를 이름으로 본다. A열 수식이 평가된 이름을 기존 참가자 집합으로 써서 요청 선수의 명단 이름과 대조한다.
  • 이미 참가자면 배치 전체 거부. "후보 제외/거부 계약"의 백스톱 — 하나라도 이미 있으면 아무것도 안 쓰고 409 + 충돌 id를 반환(부분 쓰기로 인한 모호함 제거).
  • 순수/불순 분리. 검증·행 조립은 functions/lib/add-players.ts 순수 함수로 빼 vitest로 촘촘히 검증하고, 엔드포인트는 fetch·write·cache I/O만 — 리포 관례(build-records-response↔records.ts)와 동일.

5리스크

리스크영향완화책
동시 add-players 2건이 같은 바닥 행을 읽고 같은 범위에 씀나중 요청이 앞 요청 행을 덮어써 일부 추가 유실PRD §07가 명시적으로 수용한 "마지막 승리" 모델(관리자 단일·순차 사용). 잠금은 범위 밖 — 리스크로만 기록.
명단 동명이인이 있고 그 중 한 명이 이미 회차에 있음이름 기반 판정이 다른 동명이인 추가를 already_participant로 오거부(보수적 false-positive)시스템 전반이 이름으로 식별(parse-session은 동명이인에 fail-loud). 안전한 방향(거부)이며 PRD는 이름 구분을 권고. 거부일 뿐 오염은 없음.
USER_ENTERED 오용으로 점수/헤더 칸까지 자동해석참조 수식 손상쓰는 범위는 A열 수식뿐. 점수·헤더에는 절대 안 씀(PRD §07 "이름 열·헤더 쓰기 금지"와 동일 규율).
회차 탭에 트레일링 공백 행이 섞여 바닥 행 오산중간 공백 행 위/아래 잘못된 위치에 append바닥 계산 시 트레일링 완전 공백 행을 트림한 뒤 lastRow+1. 단위 테스트로 고정.

6테스트 계획

  • 순수 로직(add-players.test.ts): 활동 선수 N명 추가 시 rows가 ='버니스명단'!A{id+1} 수식·startRow=바닥+1·added 이름 매핑이 맞는지.
  • 이미 참가한 id가 섞이면 already_participant로 던지고 충돌 id를 담는지(부분 쓰기 없음).
  • 탈퇴/비대상/휴식 id, 명단에 없는 id, 빈 배열, 중복·음수·비정수 id → 각각 validation_failed.
  • 회차 탭 끝에 공백 행이 있어도 startRow가 마지막 비공백 다음 행으로 계산되는지.
  • 쓰기 래퍼(sheetsWriteApi.test.ts): 기본 호출이 여전히 valueInputOption=RAW(회귀) + 명시 시 USER_ENTERED URL.
  • 엔드포인트(add-players.test.ts): 해피패스 200 — PUT이 대상 범위·USER_ENTERED·수식 바디로 1회, 캐시 delete 호출, 응답 added 정확. 없는 날짜 404, 이미 참가 409, 형식/상태 오류 400, Sheets API 실패 502.
  • 전체 스위트 회귀: npm test(vitest) 전건 그린 + npm run typecheck 통과.

7완료 조건

  • 선택한 활동 선수들이 대상 회차 탭 맨 아래에 이름 참조 수식 + 빈 점수로 추가된다.
  • 이미 그 회차에 있는 선수는 already_participant(409)로 거부되고, 하나라도 충돌 시 아무것도 쓰지 않는다(후보 제외/거부 계약).
  • 활동 아님·명단에 없음·형식 오류는 validation_failed(400), 없는 회차는 session_not_found(404)로 명확히 분기된다.
  • 쓰기 성공 시 /api/records 캐시가 무효화된다.
  • vitest 단위·통합 테스트가 추가되고 npm test + npm run typecheck가 그린이다.
