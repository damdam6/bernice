P2 — POST /api/refresh (records 캐시 무효화)

damdam6/bernice 브랜치 feat/43-api-refresh ← main 라벨 enhancement 2026-07-19

1요약

팀원이 회차 입력 직후 최신 랭킹을 보고 싶을 때 수동으로 캐시를 비울 수 있는 POST /api/refresh를 새로 만든다. 기존 쓰기 API(W3 POST /api/admin/records 등)와 같은 RECORDS_CACHE_KEY를 functions/lib/records-cache.ts에서 그대로 import해 caches.default.delete로 지우고, 삭제 여부를 JSON으로 응답한다. 인증은 /api/* 전체를 감싸는 _middleware.ts(#42)에 전부 위임하며, /api/admin/*이 아니므로 관리자뿐 아니라 로그인한 모든 세션(team·admin)이 호출할 수 있다.

2현황 분석

/api/records(functions/api/records.ts:16-30)는 caches.default를 RECORDS_CACHE_KEY(functions/lib/records-cache.ts:7, 'https://bernice-cache.internal/records/v1') 고정 URL로 조회하고, 미스 시 6시간 TTL로 cache.put한다. 무효화는 현재 쓰기 API 3곳에만 있다:

  • functions/api/admin/records.ts:119 — await caches.default.delete(RECORDS_CACHE_KEY) (문자열 그대로, 응답 전 await)
  • functions/api/admin/add-players.ts:86 — await caches.default.delete(new Request(RECORDS_CACHE_KEY))
  • functions/api/admin/create-sheet.ts:89 — context.waitUntil(caches.default.delete(new Request(RECORDS_CACHE_KEY)))

세 곳 모두 "쓰기 성공 후 무효화"라는 부수 효과이지 무효화 자체가 API 표면은 아니다. 이 이슈가 요구하는 것은 무효화만을 목적으로 하는 독립 엔드포인트다. PRD(docs/prd-record-input.html:104)도 "수동 새로고침 /api/refresh는 P2 예정"이라고 이 엔드포인트를 명시적으로 예고해 두었다.

인증 미들웨어(functions/api/_middleware.ts:23,37-42)는 /api/login·/api/health만 예외로 통과시키고 나머지 /api/*는 세션 검증(401), /api/admin/*만 추가로 role 검증(403)한다. /api/refresh는 admin 하위 경로가 아니므로 401 게이트만 적용되고, 이슈 본문 그대로 "자체 검증 없음"이 성립한다.

가장 단순한 기존 엔드포인트는 functions/api/health.ts:1-2 — Env 타입도 선언하지 않고 PagesFunction 하나로 끝나는 한 줄짜리 핸들러다. /api/refresh도 Sheets 접근이나 Env가 필요 없으므로 같은 최소 형태가 맞는다.

3설계

  • 파일 위치: functions/api/refresh.ts (이슈 지정 경로 그대로, admin 하위 아님).
  • 메서드: onRequestPost만 — 이슈 제목이 POST /api/refresh로 명시.
  • 구현: RECORDS_CACHE_KEY를 import해 await caches.default.delete(RECORDS_CACHE_KEY) 한 줄. 문자열 키 그대로 넘기는 형태를 택한다 — 가장 최근에 작성된 admin/records.ts:119(W3, 세 패턴 중 최신 커밋)가 이 형태이고, Cache.delete()는 RequestInfo | URL을 받으므로 new Request(...) 래핑은 불필요한 군더더기다.
  • 응답: 이슈 완료 조건 "삭제 여부 JSON 응답"에 맞춰 { deleted: boolean }. Cache.delete()가 그 자체로 "지울 게 있었는지"를 boolean으로 돌려주므로 별도 존재 확인 없이 반환값을 그대로 얹는다.
  • waitUntil로 응답을 안 막을지 여부: 막는다(await). create-sheet.ts는 무효화가 부수 효과라 waitUntil로 응답을 안 막지만, /api/refresh는 무효화 자체가 요청의 목적이자 응답 페이로드(deleted)의 근거이므로 완료를 기다려야 정확한 값을 응답할 수 있다. PRD §09의 "응답 전에 await" 원칙과도 일치.
  • 인증 자체 검증 안 함: 이슈 본문이 명시한 요구사항. 미들웨어가 이미 /api/refresh를 401 게이트에 포함하므로 핸들러 안에서 세션/역할을 다시 확인하지 않는다 — health.ts처럼 Env 타입도 선언하지 않는다.
  • 기각한 대안: 요청 바디 검증이나 rate limit 추가 — 이슈 범위(S) 밖이고 기존 3개 무효화 호출부에도 없는 패턴이라 이 엔드포인트만 새 관례를 만들 이유가 없다.

4작업 단계

  1. ✅ 새 핸들러 작성 — RECORDS_CACHE_KEY import, onRequestPost가 caches.default.delete(RECORDS_CACHE_KEY) 결과를 { deleted }로 응답. health.ts처럼 Env 선언 없이 최소 형태 유지. 계획대로 구현됨. functions/api/refresh.ts
  2. ✅ 단위 테스트 작성 — records.test.ts(캐시 전역 스텁)와 add-players.test.ts(delete 호출 인자 검증) 패턴을 재사용: vi.stubGlobal('caches', { default: { delete: vi.fn(...) } })로 true/false 두 경우 각각 { deleted: true }/{ deleted: false } 200 응답과 RECORDS_CACHE_KEY가 정확한 인자로 호출됐는지 검증. 계획대로 구현됨. functions/api/refresh.test.ts
  3. ✅ npm run typecheck·npm test·npm run build로 전체 회귀 확인. 실행 노트: 워크트리에 node_modules가 불완전(lucide-react 누락)해 최초 typecheck가 이 이슈와 무관한 이유로 실패 — npm install 후 재실행해 통과 확인. typecheck·463 vitest 테스트(신규 2건 포함)·build 전부 그린. package.json

5리스크

리스크영향완화책
Cache API는 콜로(데이터센터) 로컬다른 콜로에 남은 캐시는 /api/refresh 호출로도 안 지워지고 TTL(6시간)까지 생존 — PRD §09에 이미 문서화된 기존 한계이 이슈에서 새로 해결하지 않음(PRD 명시). 필요 시 별도 이슈로 분리
인증 자체 검증 없음team 세션도 캐시를 임의로 비울 수 있음(관리자 전용 아님)이슈 요구사항 그대로 — 캐시 삭제는 부작용이 크지 않은(재조립만 유발) 안전한 연산이라 team 허용이 의도된 설계

6테스트 계획

  • vitest: 캐시에 항목이 있어 delete가 true를 돌려줄 때 200 + { deleted: true } 확인.
  • vitest: 캐시가 비어 delete가 false를 돌려줄 때 200 + { deleted: false } 확인 (에러가 아님을 확인 — 무효화할 게 없어도 실패가 아니다).
  • vitest: caches.default.delete가 RECORDS_CACHE_KEY('https://bernice-cache.internal/records/v1') 값으로 정확히 호출됐는지 인자 검증 — 다른 쓰기 API와 같은 키를 지우는지가 이 이슈의 핵심 계약.
  • npm run typecheck — Env 미선언 최소 핸들러가 PagesFunction 타입과 충돌 없는지.
  • 회귀: 기존 캐시 관련 테스트(records.test.ts, admin/records.test.ts, admin/add-players.test.ts, admin/create-sheet.test.ts) 전부 그린 유지.

7완료 조건

  • ✅ 호출 시 RECORDS_CACHE_KEY 캐시를 삭제하고 삭제 여부({ deleted: boolean })를 JSON으로 응답한다.
  • ✅ vitest 테스트로 삭제 성공/실패(캐시 없음) 두 경우와 호출 인자를 검증한다.
  • ✅ 인증은 _middleware.ts(#42)에 전적으로 위임하며 핸들러 자체 검증이 없다 — /api/admin/*이 아니므로 team 세션도 401 없이 통과한다.
