1요약

이슈 #24(Sheets 읽기 래퍼)·#25~#27(명단/목표/회차 파서)·#28~#29(랭킹/추이·PB 계산)에서 이미 만들어져 main에 머지된 순수 함수들을 그대로 재사용해, GET /api/records가 #21(shared/domain.ts · docs/records-schema.html) 계약대로의 RecordsResponse JSON을 반환하도록 조립한다. 다만 HomeSummary(홈 요약: 최신 회차 참여 인원·종목별 달성률) 계산과, 탈퇴 선수를 sessions[].entries에서 제외하는 처리는 어느 선행 이슈에도 구현돼 있지 않아 이번 이슈에서 새로 작성해야 한다. 캐싱은 Cloudflare Cache API(caches.default)로 구현하고, 향후 P2의 /api/refresh가 같은 캐시 키로 무효화할 수 있도록 키를 별도 lib 모듈로 분리한다. 인증 게이트(P2)는 이 이슈 범위 밖이라 적용하지 않는다.

2현황 분석

  • 현재 functions/api/에는 health.ts(functions/api/health.ts:1-2) 하나뿐 — 캐싱도, env 사용도 없는 가장 단순한 형태. /api/records는 아직 없다.
  • shared/domain.ts:139-147의 RecordsResponse가 최종 응답 계약이고, docs/records-schema.html이 상태 4종 정책(§02)·점수 특수값(§03)·랭킹/달성/추이 규칙(§04)·예시 응답(§06)을 규정한다. §07은 "/api/records 엔드포인트 — 위 결과를 RecordsResponse 형태로 조립해 반환 + 캐싱"이 이 이슈의 몫이라고 명시한다.
  • 선행 이슈 산출물 확인 완료(functions/lib/): sheetsApi.ts(#24, fetchSheetBundle이 탭 분류 + batchGet까지 끝낸 SheetRawBundle을 반환), roster.ts(#25, parseRoster), parse-goals.ts(#26), parse-session.ts(#27), compute-rankings.ts(#28, computeSessionRankings), player-summary.ts(#29, computePlayerSummaries) — 전부 그대로 재사용 가능. 전부 순수 함수라 조립 로직도 순수 함수로 짜서 단위 테스트하기 좋은 조건.
  • 구멍 1 — HomeSummary 미구현: grep -rn "HomeSummary\|achievementRates\|latestSession"은 shared/domain.ts 정의 외에 어떤 구현도 없음을 확인했다. #28(랭킹)·#29(추이/PB)의 스코프는 각각 SessionRankings와 PlayerEventTrend/PlayerPersonalBest까지였고 HomeSummary는 남겨졌다 — 이번 이슈에서 새로 작성해야 한다.
  • 구멍 2 — 탈퇴 필터링이 이 계층 책임: functions/lib/parse-session.ts:25-27 주석이 "상태(탈퇴 등) 필터링은 이 파서의 책임이 아니다 ... 하류 소비자마다 포함 규칙이 달라 여기서 미리 걸러내면 복원할 수 없다"고 명시한다. 반면 docs/records-schema.html §02 표는 탈퇴 상태를 players[]·sessions[].entries·rankings/home.achievementRates "어디에도 등장하지 않음"으로 못박는다. computeEventRanking(compute-rankings.ts:32-36)과 computePlayerSummaries(player-summary.ts:98, isActivePlayer 필터)는 각자 내부적으로 탈퇴를 이미 걸러내지만, RecordsResponse.sessions 필드 자체(원시 회차별 기록)에는 탈퇴 엔트리를 걸러내는 코드가 어디에도 없다 — 이번 조립 단계가 명시적으로 처리해야 한다.
  • wrangler.jsonc:8-10의 SHEET_ID는 지금 어떤 함수 코드에서도 참조되지 않는다 — 이 이슈가 처음으로 소비한다. 시크릿(GOOGLE_SERVICE_ACCOUNT_KEY)은 .dev.vars(로컬)/Pages 시크릿(운영)이며 이미 googleAuth.ts가 그 계약대로 소비 중.
  • Cache API(caches.*) 사용처가 레포 전체에 전혀 없음(grep 확인) — 이 레포에서 처음 도입하는 패턴. Cloudflare 공식 문서 기준으로 caches.default는 문자열/커스텀 Request를 키로 받고(요청 자체와 무관하게 설계 가능), Cache-Control 헤더가 없는 응답은 상태코드 기반 기본 TTL로 저장되며, context.waitUntil(cache.put(...))로 응답을 먼저 반환하고 캐시 쓰기는 백그라운드에서 완료하는 것이 표준 패턴이다.
  • package.json의 dev:all(빌드 + wrangler pages dev --port 8788 동시 실행)이 완료 조건의 "로컬(npm run dev:all)에서 실제 시트 기반 응답" 검증 경로다. vitest.config.ts는 environment: 'node'라 caches.default는 테스트에서 전역이 아니다 — sheetsApi.test.ts가 fetch를 vi.stubGlobal로 흉내낸 것과 같은 방식으로 caches도 스텁해야 단위 테스트가 가능하다.
  • 깨끗한 워크트리 기준 npm install 후 npm run typecheck·npx vitest run 모두 그린(10 파일 · 177 테스트) — 회귀 기준선으로 사용.

3설계

모듈 분리 — 기존 컨벤션(compute-rankings.ts/player-summary.ts처럼 순수 계산은 functions/lib/에 테스트와 함께)을 따라 3개 모듈로 나눈다.

  • functions/lib/compute-home-summary.ts — (sessions, rankings, players) => HomeSummary. 새로 "활동+recorded" 필터를 다시 구현하지 않고, 이미 §04 규칙대로 계산된 최신 회차의 SessionRankings(rankings.at(-1), sessions와 같은 순서로 1:1 대응)를 그대로 집계한다 — eventRanking.entries.length = eligibleCount, .filter(e => e.achieved).length = achievedCount. 회차가 하나도 없으면 latestSession: null, achievementRates: [].
  • functions/lib/build-records-response.ts — SheetRawBundle + generatedAt을 받아 parseRoster/parseGoals/parseSession을 호출하고, 탈퇴 엔트리를 여기서 한 번만 걸러낸 뒤(그 결과를 sessions 필드에도, computeSessionRankings/computePlayerSummaries/computeHomeSummary 입력에도 동일하게 사용) RecordsResponse를 조립한다. 버니스명단/목표 탭이 없으면(bundle.roster/goals === null) RecordsAssemblyError(코드: missing_roster_tab/missing_goals_tab)를 던진다.
  • functions/lib/records-cache.ts — 캐시 키(버전 접미사 포함 고정 문자열, 예: https://bernice-cache.internal/records/v1)와 TTL 상수만 export. P2의 /api/refresh가 같은 상수를 import해 cache.delete(key)로 무효화할 수 있도록 이 이슈에서 소유권을 확정해 둔다. 응답 스키마가 나중에 바뀌면 v2로 올려 구버전 캐시를 자연스럽게 무효화할 수 있다.
  • functions/api/records.ts — onRequestGet: caches.default.match(cacheKey) 히트 시 그대로 반환. 미스 시 fetchSheetBundle → buildRecordsResponse → Cache-Control: public, max-age=… 헤더를 붙인 Response.json 생성 → context.waitUntil(cache.put(cacheKey, response.clone())) → 반환. try/catch로 SheetsApiError는 502({ error: 'sheets_api_error', message, upstreamStatus }), RecordsAssemblyError는 500({ error: err.code, message }), 그 외 Error는 500({ error: 'sheet_data_invalid', message })로 매핑.

기각한 대안

  • 탈퇴 필터링을 compute-rankings.ts/player-summary.ts 각자에 흩어 두기 — 기각: 그 두 함수는 이미 정확히 동작하고(활동 상태·비탈퇴 필터를 각자 갖고 있음) 손댈 이유가 없다. 문제는 오직 RecordsResponse.sessions 필드 자체이므로, 조립 단계에서 한 번만 걸러내는 편이 로직이 한 곳에만 있다.
  • context.request를 캐시 키로 그대로 사용 — 기각: 쿼리스트링·헤더 차이로 캐시가 쪼개질 위험이 있고, 별도 이슈(P2)의 /api/refresh가 같은 키를 재현하기 어려워진다. 고정된 합성 키가 "팀 전체가 같은 응답 하나를 공유"하는 이 엔드포인트의 의미와도 맞는다.
  • 홈 요약 달성률을 events/players/sessions로부터 처음부터 재계산 — 기각: computeEventRanking과 동일한 "활동+recorded" 필터를 중복 구현하게 되어 두 곳의 규칙이 어긋날 위험이 생긴다. 이미 계산된 rankings의 최신 회차 결과를 그대로 집계하는 편이 안전하고 코드도 짧다.
  • 에러 응답을 위해 shared/domain.ts에 정식 에러 타입 추가 — 기각: docs/records-schema.html §06이 인증(401)·에러 응답 형식은 P2 범위로 명시적으로 미루고 있고, 이 계약 파일은 이슈 #21이 소유한 강제 계약(§07)이라 이번 이슈에서 확장하지 않는다. 엔드포인트 레벨의 ad hoc JSON({ error, message })으로 충분.

4작업 단계

  1. 새 순수 함수 computeHomeSummary(sessions, rankings, players) 작성 — sessions.at(-1)/rankings.at(-1)로 최신 회차를 잡고, participantCount는 활동 상태 + participated: true만, achievementRates는 최신 회차 EventRanking.entries를 그대로 집계(eligibleCount 0 → rate 0). 회차 0건이면 latestSession: null, achievementRates: []. functions/lib/compute-home-summary.ts, functions/lib/compute-home-summary.test.ts
  2. functions/lib/records-cache.ts 작성 — 캐시 키 문자열 상수(버전 접미사 포함)와 TTL 초 상수 export. 트리비얼한 상수라 별도 테스트 없이 진행. functions/lib/records-cache.ts
  3. build-records-response.ts 작성 — SheetRawBundle + generatedAt → RecordsResponse. 순서: 탭 누락 가드(RecordsAssemblyError) → parseRoster/parseGoals → 각 회차 parseSession 후 탈퇴 엔트리 필터링 → computeSessionRankings/computePlayerSummaries/computeHomeSummary 호출 → 조립. 테스트: 탈퇴 선수가 sessions[].entries/players/rankings 어디에도 없음, 탭 누락 시 코드별 RecordsAssemblyError, 회차 0건일 때 home 기본값. functions/lib/build-records-response.ts, functions/lib/build-records-response.test.ts
  4. functions/api/records.ts 작성 — PagesFunction<Env>(Env는 googleAuth.Env + SHEET_ID: string 확장 정의). 캐시 히트/미스 분기, Cache-Control 헤더 부여, context.waitUntil(cache.put(...)), try/catch 에러 상태코드 매핑(502 vs 500×3코드). health.ts의 최소주의 스타일을 유지. functions/api/records.ts
  5. 엔드포인트 자체 테스트 — sheetsApi.test.ts가 fetch를 스텁하는 방식과 동일하게 caches 전역을 가짜 Map 기반 Cache로 스텁: (1) 첫 호출 캐시 미스 → 200 + Cache-Control 헤더 + cache.put 호출 확인, (2) 캐시 히트 시 fetchSheetBundle 미호출 확인, (3) SheetsApiError → 502, 파서 Error → 500 + 서로 다른 error 코드 확인. functions/api/records.test.ts
  6. npm run typecheck && npx vitest run로 회귀 없음 확인(기존 177개 + 신규 테스트).
  7. npm run dev:all로 실제 시트 기준 수동 스모크 — 완료 조건 항목(계약 형태 일치·연속 호출 캐시 적중·상태 4종 정책 반영)을 직접 확인.

5리스크

리스크영향완화책
"긴 캐시"의 구체적 TTL 값이 이슈 본문·설계 문서 어디에도 명시돼 있지 않음너무 길면 갱신 지연 체감, 너무 짧으면 시트 API 호출 절감 효과 감소6시간을 기본값으로 제안하고 단일 상수(records-cache.ts)로 관리 — 리뷰에서 쉽게 조정 가능
로컬 wrangler pages dev(Miniflare)의 Cache API 시뮬레이션이 실제 Cloudflare 엣지와 동작이 다를 수 있음로컬에서 본 캐시 적중 여부가 운영 환경과 다르게 보일 수 있음완료 조건의 캐시 적중 확인은 반드시 npm run dev:all로 직접 확인 + 단위 테스트로 캐시 히트/미스 분기 로직 자체를 별도 검증
로컬 반복 개발 중 캐시가 프로세스 재시작 전까지 남아 파서 수정이 응답에 안 보일 수 있음디버깅 중 "왜 안 바뀌지" 혼란wrangler pages dev 재시작으로 캐시 리셋 — 이번 이슈 범위에 캐시 우회 기능은 추가하지 않음(그 자체가 스코프 밖 기능)
RosterParseResult.issues(이름 없음/알 수 없는 상태값 같은 명단 행 이슈)는 RecordsResponse에 실을 필드가 없어 조립 단계에서 버려짐시트에 오류가 있어도 API 응답만으로는 감지 불가이 이슈 범위 밖 — 필요해지면 별도 로깅/관측 이슈로 분리
회차가 0건일 때 home.achievementRates: []로 반환하는 것은 계약 문서에 명시되지 않은 해석프론트(P3~P4)가 다른 형태(예: 종목별 0/0 항목)를 기대할 수 있음구현 시 이 판단을 PR 설명에 명시해 리뷰에서 확인받는다
