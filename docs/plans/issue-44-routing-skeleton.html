#44기능 개선

P3 — 라우팅 골격 (react-router + 빈 페이지 4개 + 404)

damdam6/bernice 브랜치 feat/44-routing-skeleton ← main 라벨 enhancement 2026-07-17

1요약

현재 앱은 src/App.tsx 하나로 고정된 헬스체크 화면만 렌더한다. 이번 이슈는 P3(프론트 공통 뼈대)의 첫 조각으로, react-router-dom을 설치해 /·/rankings·/players·/trends 4개의 빈 페이지와 404 라우트를 배선하고, 기존 헬스체크 UI를 홈 페이지로 옮긴다. 레이아웃·네비게이션(#45)과 TanStack Query(#46)는 후속 이슈이므로 이번 범위에서는 건드리지 않고, 순수 라우팅 골격만 놓는다.

2현황 분석

  • src/main.tsx:6-10 — createRoot가 <App /> 하나만 렌더. 라우터 프로바이더가 없다.
  • src/App.tsx:17-43 — App 컴포넌트가 헬스체크 상태 (useEffect로 /api/health fetch)와 화면 마크업을 한 번에 갖고 있다. 완료 조건 중 "기존 헬스체크 동작 유지"는 이 로직을 그대로 홈 페이지로 옮기기만 하면 충족된다.
  • package.json:15-16 — dependencies에 react· react-dom만 있고 react-router-dom은 미설치. npm 레지스트리 최신 버전은 7.18.1(2026-07-17 기준).
  • src/ 트리에 pages/나 라우트 관련 서브폴더가 아직 없다 — 지금까지는 단일 파일 구조.
  • 형제 이슈로 #45(공통 레이아웃+네비게이션), #46(TanStack Query 셋업)가 별도로 열려 있다 — 이번 이슈에서 네비게이션 바나 데이터 페칭 훅을 선점해서 만들 필요가 없다는 뜻이다.
  • 배포 대상은 Cloudflare Pages(wrangler.jsonc:5 — pages_build_output_dir: ./dist)이고, /api/*는 functions/api/*.ts Pages Functions가 처리한다 (vite.config.ts:9-11은 dev 서버에서만 /api를 8788로 프록시). 리포에는 public/ 디렉터리나 _redirects가 없다 — 처음엔 SPA 새로고침/직접 URL 진입을 위해 _redirects 추가가 필요하다고 가정했으나, wrangler pages dev로 직접 확인한 결과 Cloudflare "Classic Pages"는 404.html이 없으면 기본적으로 SPA로 간주해 미매칭 경로에 index.html을 200으로 서빙한다 — 별도 설정 없이 이미 충족되는 플랫폼 기본값이라 _redirects를 추가하지 않기로 했다(3장 설계 참고). dev 서버(Vite)도 SPA 폴백이 내장돼 있어 로컬 확인에는 영향이 없다.
  • vitest.config.ts의 테스트 환경은 node이고, jsdom/testing-library 류 의존성이 없다 — 지금까지 프론트 컴포넌트 단위 테스트가 존재하지 않는 상태 (functions/ 쪽 로직 테스트만 있음). 이번 S급 라우팅 골격에 새 테스트 인프라(jsdom 등)를 들이는 건 범위 밖으로 판단.

3설계

  • 라우터 구성: <BrowserRouter> + <Routes> 선언형 방식 채택. createBrowserRouter(데이터 라우터)는 loader/action 같은 기능이 필요할 때 쓰는데, 이번 이슈는 로더가 필요 없는 순수 골격이라 오버스펙. 선언형이 더 적고 읽기 쉬운 코드로 목적을 달성한다.
  • 파일 구조: src/pages/ 신설 — Home.tsx, Rankings.tsx, Players.tsx, Trends.tsx, NotFound.tsx. 기존 App.tsx의 헬스체크 마크업/로직을 Home.tsx로 그대로 이동하고, App.tsx는 <Routes> 선언만 담당하는 라우팅 진입점으로 역할을 바꾼다. main.tsx가 <BrowserRouter>로 <App />를 감싼다.
  • 빈 페이지 내용: #45(레이아웃+네비게이션)가 아직 없으므로 각 페이지는 헤더 내비게이션 없이 페이지명 + 한 줄 안내 텍스트만 렌더하는 최소 placeholder로 둔다 (기존 index.css의 brand 색상 토큰만 재사용해 스타일 붕 뜨지 않게). 레이아웃/네비게이션 UI를 여기서 선점하지 않는다.
  • 404 처리: <Route path="*" element={<NotFound />} />로 클라이언트 사이드 catch-all. 완료 조건의 "미정의 경로는 404"는 이 라우트로 충족.
  • 프로덕션 SPA 폴백 — public/_redirects 추가 안 함 (계획 수정): 최초 설계 때는 /* /index.html 200 리다이렉트 규칙을 추가하려 했으나, 구현 중 실제로 wrangler pages dev에 적용해보니 이 규칙 자체가 "Infinite loop detected" 경고로 무효 처리됐다(Parsed 0 valid redirect rules). 조사 결과 이는 Wrangler의 알려진 오탐 검증 버그이고 (cloudflare/workers-sdk#11824), 일부 설정에서는 오탐에 그치지 않고 실제 wrangler pages deploy를 에러 코드 10021로 막아버리는 사례도 보고돼 있다. 더 중요한 건 — 이 규칙 없이도(public/_redirects 자체를 지우고 재확인) /rankings·/no-such-path 등 모든 미매칭 경로가 이미 200으로 index.html을 서빙한다는 걸 로컬에서 직접 확인했다. Cloudflare Pages 공식 문서상 "Classic Pages"는 프로젝트에 404.html이 없으면 기본적으로 SPA로 간주해 미매칭 경로에 index.html을 200으로 서빙한다 — 즉 우리가 추가하려던 동작이 이미 플랫폼 기본값이었다. 실익 없이 배포 실패 리스크만 있는 파일이라 추가하지 않기로 결정.
  • 기각한 대안: createBrowserRouter 데이터 라우터 — 지금 당장 loader/action을 안 쓰므로 보일러플레이트만 늘어남. 나중에 TanStack Query(#46)가 라우터 loader가 아닌 훅 기반으로 명시돼 있어(빌드플랜: "TanStack Query 셋업 + useRecords 훅") 데이터 라우터로 미리 갈아탈 이유가 없다.

4작업 단계

  1. react-router-dom 설치 (^7) package.json, package-lock.json
  2. 기존 헬스체크 UI/로직을 App.tsx에서 src/pages/Home.tsx로 이동 (컴포넌트 이름만 Home으로 변경, 로직은 그대로) src/pages/Home.tsx (신규), src/App.tsx
  3. Rankings·Players·Trends 빈 페이지 컴포넌트 생성 — 페이지명 + 안내 문구만 있는 최소 placeholder src/pages/Rankings.tsx, src/pages/Players.tsx, src/pages/Trends.tsx (모두 신규)
  4. NotFound(404) 페이지 컴포넌트 생성 src/pages/NotFound.tsx (신규)
  5. App.tsx를 라우트 선언 전용으로 재작성: /→Home, /rankings→Rankings, /players→Players, /trends→Trends, *→NotFound src/App.tsx
  6. main.tsx에서 <App />를 <BrowserRouter>로 감싸기 src/main.tsx
  7. 타입체크 + 빌드 + (해당 시) 기존 vitest 스위트 통과 확인 npm run typecheck, npm run build, npm test

5리스크

리스크영향완화책
react-router-dom v7 API가 v6과 일부 다름(특히 데이터 라우터 관련) 선언형 BrowserRouter/Routes/Route API는 v6→v7 사이 안정적이라 실질 영향 낮음 설치 후 npm run typecheck로 즉시 검증
Cloudflare Pages의 "404.html 없으면 SPA로 간주" 기본 동작이 향후 플랫폼 변경으로 바뀔 가능성 배포 환경에서만 미매칭 경로가 실제 404로 보일 수 있음 (로컬에서는 안 보임) 이 동작은 Cloudflare Pages 공식 문서에 명시된 안정적 기본값 — 회귀 시 public/_redirects(/* /index.html 200) 재도입 검토, 단 재도입 전 wrangler pages deploy 실제 배포로 에러 코드 10021 발생 여부부터 확인할 것

6테스트 계획

  • npm run typecheck — 라우트/페이지 컴포넌트 타입 오류 없음 확인
  • npm run build — 프로덕션 빌드 성공
  • npm test — 기존 functions/ 단위 테스트 스위트가 이번 변경으로 깨지지 않는지 확인 (프론트 라우팅은 자동 테스트 대상 아님)
  • 수동 확인 (Vite dev 서버): / 진입 시 기존 헬스체크 UI("API 정상"/ "API 확인 중…" 등)가 그대로 뜨는지, /rankings·/players· /trends 각각 해당 placeholder가 렌더되는지, /no-such-path 같은 미정의 경로에서 404 페이지가 뜨는지, 404 페이지의 "홈으로 돌아가기" 링크가 /로 클라이언트 사이드 이동하는지 URL 바 직접 입력·클릭으로 확인
  • 수동 확인 (프로덕션 빌드 산출물, npm run build 후 npx wrangler pages dev dist --port <포트>): /rankings·/players·/trends·미정의 경로를 새로고침/직접 진입해도 200으로 정상 렌더되는지, /api/health가 여전히 정상 응답하는지, wrangler 로그에 _redirects 관련 경고가 없는지 확인
