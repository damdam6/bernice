P3 — 공통 레이아웃 + 네비게이션

1요약

#62가 만든 flat 라우트 테이블(메인 3탭 + 관리자 7라우트)을 두 개의 pathless 레이아웃 라우트로 감싼다. 메인 3탭(홈·랭킹·개인)은 MainLayout이 감싸 화면 하단에 3탭 BottomNav(활성=primary, 비활성=ink-muted)를 고정 노출하고, 관리자 스택(로그인·시트 관리·기록 입력 등 7라우트)은 AdminLayout이 감싸 BottomNav 대신 ‹ 뒤로 텍스트 버튼만 노출한다. 두 레이아웃 모두 Outlet 기반이라 탭 전환 시 레이아웃 자체는 리마운트되지 않는다. 아이콘은 PRD 권장대로 lucide-react를 신규 도입하고, 기존 8개 placeholder 페이지의 풀스크린 wrapper를 레이아웃에 맞게 조정한다.

2현황 분석

src/App.tsx:1-33 — <Routes> 하나에 12개 라우트가 평면 나열되어 있고 레이아웃 래퍼가 전혀 없다. docs/plans/issue-62-routing-three-tabs-admin-stack.html의 "설계" 절에 "Outlet 기반 레이아웃은 #45가 별도로 감쌀 예정이라 지금 만들면 곧 갈아엎힌다"고 명시되어 있어, 현재 flat 구조는 의도된 상태이며 이번 이슈가 그 감싸기 작업을 맡는다.

메인 3탭(src/pages/Home.tsx, Rankings.tsx, Players.tsx)과 관리자 6페이지(src/pages/admin/*.tsx) + NotFound.tsx 총 8개 파일이 각자 동일한 풀스크린 wrapper를 반복한다: <div className="flex min-h-dvh flex-col items-center justify-center bg-canvas px-6"> 안에 흰 카드 하나. 레이아웃 도입 시 min-h-dvh·bg-canvas를 레이아웃 루트로 옮기고 각 페이지는 남은 영역을 채우도록 조정해야 겹치지 않는다.

src/index.css:6-52 — Tailwind v4 @theme 토큰에 --color-primary: #3f51b5, --color-ink-muted: #b0b5bf, --color-line, --color-surface, --color-canvas가 이미 정의되어 있다(#61에서 완료). docs/prd-design.html §04의 네비 비활성 색 #aab0bc은 §03 토큰 표에 없는 인라인 참고값이며, 기존 ink-muted(#b0b5bf)과 사실상 동일해 별도 토큰 신설이 불필요하다.

아이콘 라이브러리가 프로젝트에 전무하다 — package.json dependencies에 아이콘 패키지가 없고, src/ 전체에서 lucide 검색 결과가 0건이다. 유일한 기존 아이콘형 렌더링은 src/components/common/Gauge.tsx:31-43의 손수 그린 도메인 전용 SVG(게이지 아크)뿐이라, 범용 아이콘 패턴을 재사용할 수 없다. docs/prd-design.html §06 각주는 "아이콘은 24px 그리드 스트로크(2px, round cap) … 구현은 lucide-react 권장"이라고 명시한다.

package.json:16 — react-router-dom@^7.18.1이 이미 설치되어 있어 <Route element={<Layout/>}>…</Route> 형태의 pathless 레이아웃 라우트(자식은 <Outlet/>로 렌더)를 추가 의존성 없이 쓸 수 있다.

테스트 관례: src/components/RankingTable.test.tsx·useRecords.test.tsx는 상단에 // @vitest-environment jsdom을 붙이고 @testing-library/react로 렌더/상호작용을 검증한다(파일 최상단 pragma 방식). 반면 PlayerStatusPill.test.tsx류 순수 표시 컴포넌트는 renderToStaticMarkup + src/test/html-utils.ts 헬퍼를 쓴다. src/pages/에는 테스트 파일이 아직 하나도 없다 — 페이지는 얇은 wrapper이고 로직은 컴포넌트에 있다는 이 저장소의 관행상, 이번 이슈의 테스트도 페이지가 아니라 BottomNav/AdminLayout 컴포넌트에 붙여야 한다.

3설계

선택한 설계 — src/components/layout/에 4개 컴포넌트 신설: MainLayout(메인 3탭용, Outlet + 하단 고정 BottomNav), AdminLayout(관리자 스택용, 상단 슬림 헤더의 BackButton + Outlet, 네비 없음), 그리고 그 안에서 쓰는 BottomNav·BackButton. App.tsx는 두 개의 pathless <Route element={...}>로 기존 자식 라우트들을 그룹화하기만 하고, #62가 확정한 URL 표(경로·컴포넌트 매핑)는 그대로 둔다. /trends 리다이렉트와 * 404는 두 레이아웃 밖에 남긴다 — 404는 "메인 3화면"이 아니므로 하단 네비를 노출하면 안 된다는 스펙(§04)을 문자 그대로 지키는 선택이다.

아이콘은 lucide-react를 신규 설치해 홈·랭킹(트로피)·개인(사람)·뒤로가기(쉐브론) 4개를 가져다 쓴다. PRD가 명시적으로 권장하고, 이후 #67/#68(잠금 아이콘, 스테퍼, 토글)에서도 같은 니즈가 반복될 것이므로 지금 손그림 SVG로 땜질하면 곧 교체될 코드를 만드는 셈이다.

색상은 신규 토큰 없이 기존 text-primary(활성)/text-ink-muted(비활성)를 그대로 재사용한다 — §03 토큰 표에 정식으로 있는 값이고 PRD §04의 인라인 참고색과 사실상 동일하다.

하단 네비는 position: fixed로 뷰포트 하단에 고정하고(env(safe-area-inset-bottom) 패딩 포함), 메인 컨텐츠 영역에 그 높이만큼 padding-bottom을 준다 — 콘텐츠가 길어져도 네비가 스크롤에 묻히지 않아야 앱 셸다운 지속성이 생긴다.

기각한 대안

  • BottomNav를 flex 정상 흐름에 놓기(고정 아님) — 기각: 콘텐츠가 뷰포트보다 길어지면 네비가 스크롤에 딸려 사라져 "탭 전환 시 레이아웃 유지"라는 완료 조건의 취지(항상 보이는 앱 셸)에 어긋난다.
  • Layout 하나 + variant prop으로 메인/관리자 분기 — 기각: 헤더 유무·네비 종류·정렬이 서로 달라 결국 컴포넌트 내부에서 다시 분기하게 된다. 라우트 선언부에서 바로 어떤 레이아웃인지 보이는 두 개의 작은 컴포넌트가 더 읽기 쉽다.
  • 손그림 인라인 SVG 아이콘 — 기각: PRD가 명시적으로 lucide-react를 권장했고, 곧 이어질 이슈(#67/#68)에서 같은 니즈가 반복돼 지금 서드파티를 들이는 편이 중복 작업을 줄인다.
  • NotFound도 MainLayout으로 감싸 시각적 일관성 확보 — 기각: §04 스펙이 "메인 3화면에서만" 하단 네비 노출을 명시하고, 404는 그 3화면에 속하지 않는다. 감싸면 404가 유효한 앱 섹션인 것처럼 오독될 수 있다.

4작업 단계

  1. 아이콘 라이브러리 추가 — npm install lucide-react(PRD §06 권장) package.json, package-lock.json
  2. BottomNav 신설 — 홈·랭킹·개인 3탭, NavLink로 활성 판정(/는 end), 활성=text-primary/비활성=text-ink-muted, 24px 아이콘 + 11px/700 라벨, 화면 하단 fixed + safe-area 패딩 src/components/layout/BottomNav.tsx
  3. BackButton 신설 — ChevronLeft 아이콘 + "뒤로" 텍스트, 클릭 시 useNavigate()(-1) src/components/layout/BackButton.tsx
  4. MainLayout·AdminLayout 신설 — 각각 <Outlet/> + 위 컴포넌트 조합. min-h-dvh·bg-canvas를 레이아웃 루트로 옮기고, MainLayout의 <main>에는 네비 높이만큼 padding-bottom 부여 src/components/layout/MainLayout.tsx, src/components/layout/AdminLayout.tsx
  5. App.tsx를 pathless 레이아웃 라우트 2개로 재구성 — 메인 3탭은 MainLayout 자식, 관리자 7라우트는 AdminLayout 자식으로 옮기고, /trends 리다이렉트·* 404는 그대로 최상위에 둠(경로·컴포넌트 매핑은 #62 표 그대로 유지) src/App.tsx
  6. 10개 placeholder 페이지의 풀스크린 wrapper(min-h-dvh … bg-canvas)를 레이아웃 내부에서 동작하도록 조정(flex-1 기반 센터링으로 교체) — NotFound.tsx는 레이아웃 밖이라 변경 없음 src/pages/Home.tsx, src/pages/Rankings.tsx, src/pages/Players.tsx, src/pages/admin/AdminLogin.tsx, src/pages/admin/SheetManagementHome.tsx, src/pages/admin/RecordsDateSelect.tsx, src/pages/admin/RecordsParticipants.tsx, src/pages/admin/RecordsPlayerInput.tsx, src/pages/admin/CreateSheet.tsx, src/pages/admin/AddPlayers.tsx 참고: 요약/리스크 절의 "8개"는 집계 오기 — 실제 wrapper 보유 페이지는 10개(NotFound 제외)였고, 파일 목록은 처음부터 정확했다.
  7. 테스트 작성 — BottomNav는 경로별로 대응 탭만 활성 스타일인지, AdminLayout은 하단 네비 미노출 + BackButton 클릭 시 이전 경로로 이동하는지 검증(MemoryRouter + @testing-library/react) src/components/layout/BottomNav.test.tsx, src/components/layout/AdminLayout.test.tsx
  8. npm run typecheck · npm run build · npm test 통과 확인 + claude-in-chrome으로 모바일 뷰포트(≈404×852)에서 3탭 전환·관리자 진입·뒤로가기 수동 확인 (파일 없음 · 커맨드 + 수동 확인) 참고: typecheck·build(266KB JS/gzip 84KB)·test(22개 파일 262개 테스트) 전부 통과. claude-in-chrome으로 로컬 vite 서버를 띄워 홈→랭킹 탭 전환(활성 강조 유지), /admin/login 진입(하단 네비 소실 + ‹ 뒤로 노출), 뒤로가기 클릭(이전 경로 복귀), 임의 경로 404(레이아웃 밖 단독 렌더)까지 확인함. 다만 브라우저 창 resize_window가 이 cmux 서피스에서 실제로 적용되지 않아 404×852 픽셀 단위 확인은 못했고, 넓은 뷰포트에서 동일 컴포넌트로 기능 동작만 검증했다 — CSS 자체는 모바일 우선(fixed 하단바 + flex-1 센터링)이라 좁은 폭에서도 동일하게 동작할 것으로 판단.

5리스크

리스크영향완화책
lucide-react 신규 의존성이 번들 크기에 영향낮음 — named import 4개뿐, PRD가 명시적으로 권장한 선택tree-shakable named import만 사용, npm run build 산출물 크기 확인
BackButton의 navigate(-1)이 히스토리 없이 관리자 라우트에 직접 진입(딥링크)했을 때 앱 밖으로 이탈낮음 — 내부 팀 전용 앱, 관리자 라우트를 외부 공유하지 않음이번 이슈 범위 밖으로 명시. 필요해지면 navigate('/admin') 폴백은 후속 이슈에서
8개 placeholder 페이지 wrapper 변경이 병행 진행 중인 #67/#68과 같은 파일을 건드려 충돌 가능중간 — 동일 파일을 다른 브랜치가 동시에 수정 중일 수 있음변경을 wrapper 클래스 한 줄 수준으로 최소화하고 내부 콘텐츠(h1·안내문)는 그대로 둬 병합 충돌 표면을 줄임

6테스트 계획

  • npm run typecheck — 신규 레이아웃 컴포넌트·재구성된 라우트 트리에 타입 오류 없는지 확인
  • npm run build — 빌드 통과, lucide-react 포함 정상 번들 확인
  • npm test — 기존 스위트 회귀 없음 + 신규 BottomNav/AdminLayout 테스트 통과
  • 신규 유닛: MemoryRouter로 /·/rankings·/players 각각 렌더 시 대응 탭만 활성 스타일(aria-current="page" 또는 text-primary)을 갖는지 확인
  • 신규 유닛: AdminLayout 렌더 시 BottomNav가 DOM에 없고, BackButton 클릭 시 history가 이전 경로로 이동하는지 확인
  • 수동(claude-in-chrome, 모바일 뷰포트): 홈→랭킹→개인 탭 전환 시 하단 네비가 리마운트 없이 유지되고 활성 탭이 바뀌는지, 홈 잠금 아이콘 대신 /admin/login 직접 진입 시 하단 네비가 사라지고 ‹ 뒤로가 보이는지, 뒤로가기 클릭 시 이전 화면으로 돌아가는지 확인

7완료 조건

  • 3개 탭(홈·랭킹·개인) 전환 시 BottomNav가 리마운트 없이 유지되고 활성 탭이 primary 색으로 강조된다
  • 관리자 라우트(/admin/* 7개) 전체에서 하단 네비가 노출되지 않고 ‹ 뒤로 텍스트 버튼이 대신 노출된다
  • 모바일 뷰포트에서 하단 네비·뒤로가기 버튼이 겹침·잘림 없이 정상 동작한다(safe-area 여백 포함)
  • npm run typecheck · npm run build · npm test 모두 통과한다
