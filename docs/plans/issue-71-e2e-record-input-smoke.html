기록 입력 W7 — E2E 마감 (측정일 시나리오 스모크 + 문서 정리)

damdam6/bernice 브랜치 feat/71-e2e-record-input-smoke ← main 라벨 enhancement 2026-07-19

1요약

"기록 입력" 기능(W0~W6, 이슈 #60·#41·#42·#63~#68)을 마감하는 마지막 이슈다. 세 가지를 한다: (1) 측정일 시나리오 — 관리자 로그인 → 기록지 만들기 → 참가자 선택/추가 → 전원 입력(수정 포함) → 저장 즉시 랭킹 반영 — 을 실제 라우팅·React Query 캐시를 통해 검증하는 E2E 스모크 테스트를 추가한다. (2) 코드 감사 중 발견한 에러 상태 버그 2건(create-sheet-api.ts·add-players-api.ts가 네트워크 오류 시 무한 "처리 중"에 멈추는 문제)을 고친다 — records-write-api.ts에 이미 한 번 적용된 수정과 동일한 패턴이다. (3) docs/build-plan.html P5 섹션과 docs/index.html (설계 개요)의 "아직 구현 시작 전" 서술을 실제 구현 완료 상태로 갱신한다.

2현황 분석

베이스라인은 안정적이다. npm run typecheck 클린, npm test 71개 파일·602개 테스트 전부 통과(수정 전 확인 완료) — 이번 변경은 이 위에 얹는다.

E2E/브라우저 테스트 프레임워크가 없다. package.json에 Playwright 등 브라우저 자동화 도구가 없고, 기존 관례는 vitest + @testing-library/react + jsdom, 그리고 vi.stubGlobal('fetch', ...)로 수동 스텁하는 방식이다(MSW 미사용). 14개 테스트 파일이 각자 로컬 jsonResponse 헬퍼를 중복 정의하는 것으로 봐서(예: src/pages/admin/RecordsPlayerInput.test.tsx:11-13) "테스트 파일마다 필요한 만큼 로컬로 만든다"가 이 repo의 확립된 관례다 — 새 공용 테스트 유틸을 만들지 않는다.

쓰기 화면 전체가 쿼리 무효화 하나로 묶여 있다. AdminLogin·SheetManagementHome· CreateSheet·AddPlayers·RecordsParticipants·RecordsPlayerInput가 전부 같은 쿼리 키 RECORDS_QUERY_KEY = ['records'](src/hooks/useRecords.ts:49)를 쓰고, 저장 성공마다 invalidateQueries를 호출한다 (RecordsPlayerInput.tsx:117, CreateSheet.tsx:57, AddPlayers.tsx:90). "저장 즉시 랭킹 반영"은 이 무효화→재조회 사이클에 의존하는 동작이라, 페이지 단위 테스트(각자 독립된 QueryClient로 렌더)로는 원천적으로 검증할 수 없고 여러 화면을 실제로 넘나드는 통합 테스트가 필요하다 — 이번 이슈가 그 유일한 검증 지점이 된다.

발견 1 — 네트워크 오류 미방어 버그(에러 상태 정리 대상). 가장 최근 커밋(101cf74, "W6 리뷰 반영")이 src/lib/records-write-api.ts의 saveRecord를 고쳤다: fetch가 네트워크 단절로 reject하면 Promise<SaveRecordResult> 계약(항상 ok:true|false로 resolve)이 깨져 호출부에서 unhandled rejection이 나고 submitting 상태가 영구히 true로 남는 버그였다(수정: records-write-api.ts:33-46, try/catch로 network_error 반환). 그런데 같은 패턴의 다른 두 파일은 이 수정이 적용되지 않았다:

  • src/lib/create-sheet-api.ts:19-24 — fetch 호출에 try/catch 없음. 호출부 CreateSheet.tsx:45(handleConfirm)도 try/catch 없이 결과만 분기.
  • src/lib/add-players-api.ts:22-27 — 동일 구조. 호출부 AddPlayers.tsx:78(handleConfirm)도 동일.

네트워크가 끊긴 채로 "기록지 만들기"/"참가자 추가"를 누르면 버튼이 "만드는 중…"/"추가하는 중…"에 멈춘 채 에러 문구도 없이 복구 불가능해진다 — 이슈의 "에러·빈 상태 정리" 항목이 가리키는 구체적 결함이다. 코드 감사 결과 이 두 곳 외의 관리자 쓰기 핸들러(로그인·로그아웃·저장)는 이미 호출부가 자체 try/catch로 감싸고 있어 안전함을 확인했다(AdminLogin.tsx:27-40, LoginGate.tsx:55-67, SheetManagementHome.tsx:17-24) — login-api.ts·logout-api.ts의 "네트워크 실패를 여기서 안 잡는다"는 설계 주석대로다.

빈 상태(empty state)는 이미 충분하다. CreateSheet·AddPlayers· RecordsDateSelect·RecordsParticipants·RecordsPlayerInput 5개 화면 모두 공용 EmptyState/ErrorPanel/Spinner로 로딩·에러·빈 목록을 처리하고 있다(각 파일 확인). 추가로 손댈 빈 상태 결함은 발견하지 못했다 — 이번 "정리"는 위 에러 처리 버그 2건에 집중한다.

발견 2 — "추이" 화면은 실제로 존재하지 않는다. src/pages/Players.tsx(개인/추이 탭, /players)와 src/pages/Home.tsx(홈, /)는 각각 "선수별 추이 화면은 P4에서 채워집니다", "P0 스캐폴딩 · 화면은 P3~P4에서 채워집니다"라는 플레이스홀더 그대로이고, 둘 다 useRecords()를 아예 호출하지 않는다. 차트 컴포넌트(radar-chart.tsx 등, 이슈 #69에서 완료)와 트렌드 계산(functions/lib/player-summary.ts)은 준비돼 있지만 화면에 연결되지 않았다. 이슈 #71의 의존 목록(#60·#41·#42·#63~#68)에는 이 두 화면을 채우는 이슈가 없다 — "기록 입력" W-트랙과 무관하게 이미 존재하던 별개의 gap이며, 이번 이슈로 새로 만들지 않는다(3장에서 처리 방식을 설명).

tsconfig 경계. tsconfig.app.json:23의 include: ["src", "shared"]는 functions/를 포함하지 않는다 — 그래서 src/lib/seoul-date.ts:1-3, src/lib/korean-sort.ts:1-2가 "functions/lib의 동일 로직을 import할 수 없어 프론트 사본을 둔다"는 주석과 함께 이미 의도적으로 중복 보유 중임을 확인했다. 반대로 shared/build-event-score.ts· shared/normalize-score.ts는 "Sheets·행/열 개념에 의존하지 않는 순수 함수만 shared/로 둔다"(build-event-score.ts:1-3)는 원칙대로 양쪽에서 안전하게 import 가능하다. 이 경계는 E2E 테스트의 목(mock) 백엔드 설계에 그대로 적용된다(3장).

3설계

E2E 스모크 테스트 — src/App.e2e.test.tsx 신설

실제 <App /> 전체를 MemoryRouter + 전용 QueryClientProvider로 감싸고, vi.stubGlobal('fetch', ...)에 상태를 갖는(stateful) 목 핸들러를 연결한다. 기존 페이지별 테스트(예: RecordsPlayerInput.test.tsx)는 매 테스트가 정적 픽스처 하나만 돌려주면 충분했지만, 이번엔 "로그인 → 생성 → 여러 번 저장 → 조회"가 한 세션 안에서 앞선 요청의 결과를 뒤의 응답에 반영해야 하므로 상태가 필요하다 — 이 부분만 기존 관례에서 확장한다.

목 백엔드가 무엇을 재사용하는가. 2장에서 확인한 tsconfig 경계 때문에 functions/lib/compute-rankings.ts 등은 src/ 테스트에서 직접 import할 수 없다(tsc -b가 깨진다) — 그리고 그 경로를 억지로 열기 위해 랭킹/트렌드 계산 함수를 shared/로 승격하는 것은 이번 이슈 범위를 넘는 리팩터라 하지 않는다. 대신:

  • 재사용: shared/build-event-score.ts의 buildEventScore — 검증의 단일 원천(PRD §08)이므로, 목의 "저장" 핸들러가 실제 프론트·백엔드와 정확히 같은 규칙으로 셀 값을 정규화하게 한다. src/lib/seoul-date.ts의 formatSeoulDate, src/lib/korean-sort.ts의 compareKorean — 둘 다 이미 프론트 전용으로 존재하는 진짜 함수라 목의 "오늘 날짜 부여"·"가나다 정렬"에 그대로 쓴다.
  • 직접 구현(테스트 파일 내부, import 아님): 랭킹 계산(방향별 정렬 + 공동순위)은 functions/lib/compute-rankings.ts와 같은 규칙을 손으로 재현한다 — 단, 픽스처를 선수 3명· 종목 4개로 작게 고정해 사람이 직접 검산할 수 있는 범위로 제한한다(이미 functions/lib/compute-rankings.test.ts가 이 알고리즘 자체의 전체 커버리지를 갖고 있으므로, 여기서 다시 전면 검증할 필요는 없다).

시나리오(단일 it — 스모크는 하나의 연속 흐름을 보는 것):

  1. /admin/login에서 관리자 코드 입력 → 로그인 성공 → /admin 진입.
  2. "기록지 만들기" → 활동 선수 3명 중 2명만 선택("참가자 선택"을 의미 있게 만들기 위한 부분 선택) → 생성 → 참가자 목록 화면 도달 + 생성 토스트 확인.
  3. "참가자 추가"로 나머지 1명 마저 추가 — 같은 참가자 목록에 두 진입 경로(기록지 만들기의 참가자 선택 vs. 참가자 추가)가 합쳐짐을 함께 증명하고, 동시에 발견 1의 회귀 대상인 add-players 경로도 스모크가 지나가게 한다.
  4. 참가자 목록에서 선수1 → 입력 화면 → 4종목(시간 1·개수 2·면제 가능 개수 1) 전부 입력 → 저장 → 목록 복귀 + "완료" 뱃지 확인.
  5. 선수2 입력 — 면제 가능 종목은 면제 토글 사용 → 저장.
  6. 선수3 입력 → 저장. (이로써 "전원 입력" 충족.)
  7. 선수1을 다시 선택 → 기존 값 프리필 확인 → 개수 종목 값 하나를 변경("수정 포함") → 재저장.
  8. /rankings로 이동 — 관리자 세션도 GET /api/records를 통과하므로 팀 패스코드 게이트를 다시 거치지 않는다(근거: AdminGate.tsx:6-9 주석 "team 세션도 GET /api/records는 통과"의 역, 그리고 두 게이트가 같은 쿼리 캐시를 공유). 수정한 종목의 필터 칩을 선택 → 수정 전 순위와 다른 순위가 반영됐는지 확인(예: 수정으로 1위가 바뀜) → 면제 처리한 선수가 "면제" 뱃지로 표시되는지 확인(근거: ranking-view.ts의 buildRankingRows가 면제·미측정 참가자를 보충 행으로 포함시킴).
  9. "추이" 관련 확인(화면 없이): 목 핸들러가 캡처해 둔 마지막 GET /api/records 응답 바디에서, 방금 수정한 선수·종목의 점수 값이 최신값으로 갱신됐는지 직접 assert한다 — players[].trends를 소비하는 실제 화면이 아직 없으므로(2장), 그 데이터 계약이 저장 직후 올바르게 갱신된다는 사실만 확인하는 임시 검증 지점이다. 테스트 파일 상단 주석에 이 한계를 남긴다.

"폰 뷰포트" 처리. jsdom은 실제 CSS 레이아웃을 계산하지 않으므로 픽셀 단위 반응형 검증은 애초에 불가능하다. window.innerWidth/innerHeight를 모바일 크기로 맞춰 의도만 표시하고, 실제 반응형 검증은 하지 않는다는 한계를 테스트 상단 주석과 리스크 표에 명시한다 — 거짓 안심을 주지 않기 위해서다.

에러 상태 버그 수정

create-sheet-api.ts·add-players-api.ts의 fetch 호출을 records-write-api.ts:33-46과 정확히 같은 모양으로 try/catch해, 네트워크 실패 시 { ok: false, error: 'network_error', message: '네트워크 오류로 ...' }를 resolve하게 한다. 호출부(CreateSheet.tsx·AddPlayers.tsx)는 이미 if (!result.ok) 분기가 있으므로 컴포넌트 쪽 변경은 없다 — 수정이 API 래퍼 레이어 안에서 끝난다.

문서 갱신 — 범위를 좁게 잡는다

docs/build-plan.html은 P5 섹션(build-plan.html:119-127)에만 손댄다 — 현재 목록엔 관리자 코드 검증·create-sheet·add-players·UI·캐시 무효화만 있고 점수 입력 (POST /api/admin/records)과 입력 화면 3종이 빠져 있다(P5 계획이 2026-07-12에 쓰였고, 점수 입력 PRD는 2026-07-17에 나왔으므로). 이 누락 항목만 추가하고, 문서 상단 배지·P0~P4/P6의 다른 섹션 상태는 이번 이슈가 검증할 수 없는 내용(배포 여부 등)이라 손대지 않는다.

docs/index.html("설계 개요")은 이슈가 문서 전체를 지목했으므로 더 넓게 본다 — 상단 배너 (index.html:141 "아직 구현(스캐폴딩)은 시작하지 않았습니다")가 지금은 명백히 사실이 아니다 (71개 테스트 파일·602개 테스트가 통과하는 실제 동작하는 앱이 있다). 이 배너와 §05 ③ 라벨 (index.html:307 "PRD 확정" → 구현 완료로), §06 표의 "기록 입력" 행 (index.html:334 상태 "확정" → "완료")과 footer 업데이트일(index.html:342)을 갱신한다. 다만 §06의 "사이트 구축: 대기"(index.html:336) 행은 개인 프로필/홈 화면·배포 등 이번 이슈로 검증 못하는 항목이 여전히 남아 있어 "완료"로 못 박지 않고, "핵심 기능(기록 조회·기록 입력) 구현 완료 · 개인 프로필 화면 등 일부 남음"처럼 사실만 정확히 반영한다.
