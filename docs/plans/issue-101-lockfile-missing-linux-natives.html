#101버그 수정

CF Pages 빌드 실패 — package-lock.json에 리눅스 네이티브 바이너리(@rollup/rollup-linux-x64-gnu 등) 누락

damdam6/bernice 브랜치 fix/101-lockfile-missing-linux-natives ← main 라벨 bug 2026-07-19

1요약

커밋된 package-lock.json이 macOS(darwin-arm64)에서 만들어져, 네이티브 바이너리를 쓰는 4개 계열(@rollup/rollup-*, @esbuild/*, @tailwindcss/oxide-*, lightningcss-*)의 설치 대상 패키지 노드가 darwin-arm64 하나씩만 기록돼 있다. CF Pages(Linux x64)의 npm ci는 락파일에 실제로 기록된 패키지 노드만 엄격 설치하므로 linux-x64 바이너리가 빠지고, vite build가 @rollup/rollup-linux-x64-gnu를 못 찾아 첫 번째로 터진다(고쳐도 esbuild·oxide·lightningcss 연쇄). 해결책은 락파일을 클린 재생성해 전 플랫폼 optional 패키지 노드를 락파일에 심는 것이며, 재발 방지로 Linux npm ci && npm run build를 돌리는 최소 CI를 추가한다.

2현황 분석

  • 스택 / 빌드 경로. package.json의 build = tsc -b && vite build. Vite 7 + Tailwind v4 + Cloudflare Pages 조합이라 rollup·esbuild·@tailwindcss/oxide·lightningcss 네 가지 네이티브 바이너리를 모두 사용한다.
  • 락파일 형식. package-lock.json의 lockfileVersion: 3 (npm 7+). 로컬 개발 npm 11.6.1 / Node v24, CF는 nodejs@22.16.0.
  • 핵심 증상 — 선언은 있지만 설치 노드가 없음. linux-x64 변형은 부모 패키지의 optionalDependencies 블록에 선언돼 있다: package-lock.json:2137(@rollup/rollup-linux-x64-gnu), :1505(@esbuild/linux-x64), :785(@tailwindcss/oxide-linux-x64-gnu), :1752(lightningcss-linux-x64-gnu). 그러나 실제 설치 단위인 "node_modules/@rollup/rollup-linux-x64-gnu": { resolved, integrity } 패키지 노드는 0개다(darwin-arm64 노드만 존재).
  • 확인 방법. grep '"node_modules/@rollup/rollup-linux-x64-gnu":' package-lock.json → 0. 반면 grep '"node_modules/@rollup/rollup-darwin-arm64":' → 1. 나머지 세 계열도 동일 패턴.
  • 배제된 원인. Node 버전 아님(CF가 nodejs@22.16.0 정상 감지, Vite 7 요구 충족) · .npmrc 오염 아님(레포에 .npmrc 없음).

3원인 분석

근본 원인: 이 락파일은 darwin-arm64 머신에서 생성/갱신되면서 현재 플랫폼용 optional 패키지 노드만 기록됐다. npm ci는 락파일에 resolved/integrity가 박힌 패키지 노드 목록을 그대로 설치하되, 각 노드의 os/cpu 제약에 맞는 것만 reify한다. Linux에서는 (a) darwin-arm64 노드가 플랫폼 불일치로 건너뛰어지고 (b) linux-x64 노드는 애초에 락파일에 없어 설치 자체가 불가 → rollup이 자기 네이티브 모듈을 못 찾고 Cannot find module @rollup/rollup-linux-x64-gnu로 종료. 이것이 npm/cli#4828로 알려진 optional dependency 락파일 프루닝 문제다.

고칠 지점: optional 패키지 노드는 락파일의 ideal tree 단계에서 os/cpu 무관하게 전부 기록돼야 한다. 락파일을 깨끗한 상태에서 재생성하면(기존 락파일·node_modules 제거 후 npm install) npm 11이 전 플랫폼 optional 노드를 다시 써 넣는다 — 이슈 제보자가 --package-lock-only 재생성으로 이미 검증한 동작.

채택안: 기존 해석 버전을 보존한 락파일을 node_modules가 없는 깨끗한 임시 디렉터리에서 npm install --package-lock-only --ignore-scripts로 보충 → 락파일에 4개 linux-x64 노드 존재 및 기존 패키지 버전 변경 0건 검증 → npm ci/build/test로 정합성·회귀 확인. 완전 신규 재생성은 semver 범위 안의 전이 의존성 15개를 함께 올려, 기능 변화 위험을 줄이기 위해 버전 보존안을 택했다.

재발 벡터 메모: 이슈가 경계한 #93(zod 추가)은 최종적으로 CLOSED되고 #100(손수 조립형 런타임 검증)으로 대체돼 zod 의존성이 실제로 추가되지 않았다(grep -c zod package-lock.json → 0). 따라서 그 특정 재발 경로는 현재 소멸. 다만 “macOS에서 npm install이 락파일을 다시 프루닝” 하는 일반 위험은 그대로라, 아래 4단계의 CI 가드가 필요하다.

4작업 단계

  1. 기존 해석 버전을 보존한 채 깨끗한 환경에서 락파일 보충 — npm install --package-lock-only --ignore-scripts. package.json과 기존 패키지 버전은 건드리지 않는다(의존성 추가/변경 아님). package-lock.json (재생성)
  2. 재생성 락파일에 전 플랫폼 optional 노드가 심겼는지 검증 — 4개 linux-x64 패키지 노드가 각각 1개씩 존재하는지 grep으로 단언(@rollup/rollup-linux-x64-gnu·@esbuild/linux-x64·@tailwindcss/oxide-linux-x64-gnu·lightningcss-linux-x64-gnu). 기존 패키지 버전 변경 0건도 확인. package-lock.json
  3. 로컬 정합성·빌드·테스트 확인 — npm ci(락파일↔package.json 동기 검증) → npm run build(tsc -b && vite build) → npm test(vitest run) 순으로 그린 확인. package.json (scripts)
  4. 재발 방지 CI 추가 — ubuntu-latest에서 npm ci && npm run build(+ npm test)를 PR마다 돌리는 최소 워크플로. CF Linux 환경을 그대로 재현해 어떤 단일-플랫폼 락파일 회귀든 머지 전에 잡는다. 기존 ai-review.yml은 그대로 두고 별도 ci.yml 신설. .github/workflows/ci.yml (신규)
  5. 문서화 — README에 “락파일은 전 플랫폼 재생성; 단일 플랫폼 락파일 커밋 금지” 짧은 주의문 추가(CI 가드의 인간용 보완). README.md
  6. 플랜 체크오프 후 커밋 — 락파일·CI·문서 변경을 레포 스타일대로 정리. PR 본문에는 Closes #101을 기재한다. package-lock.json, .github/workflows/ci.yml, README.md

5리스크

리스크영향완화책
락파일 재생성으로 transitive 패치/마이너 버전이 함께 올라감 의도치 않은 의존성 변경, 빌드/동작 미묘한 변화 가능 package.json 범위는 불변이므로 major 점프는 없음. diff 리뷰 + npm run build·npm test 그린으로 확인. 큰 변화면 범위 좁혀 재생성.
로컬(macOS)에서는 Linux npm ci를 진짜로 못 돌림 “고쳤다”가 실제 CF 통과와 어긋날 위험 4단계 ubuntu-latest CI가 머지 전 Linux npm ci && build로 직접 증명. 최종 확증은 CF Pages 배포 성공(완료 조건).
로컬 npm 11.6.1 vs CF 번들 npm 버전 차이 재생성 락파일이 CF에서 다르게 해석될 여지 lockfileVersion 3은 npm 7+ 공통 호환. CI가 CF와 같은 계열 Node/Linux로 검증.
CI 워크플로 신설 = 새 표면(러너 시간·비용) PR마다 빌드 잡 추가 단일 ubuntu 잡으로 최소화. 부담되면 완료 조건상 “문서화”만으로 대체 가능(완료 조건 4는 “검토” 항목).

6테스트 계획

  • 락파일 완전성(회귀 방지 단언). grep -c '"node_modules/@rollup/rollup-linux-x64-gnu":' package-lock.json 등 4개 계열이 각각 1 — 재생성 전(모두 0) 대비 변화 확인.
  • 락파일 정합성. npm ci가 에러 없이 완료(package.json↔락파일 동기).
  • 빌드 회귀. npm run build(tsc -b && vite build) 성공, dist 산출.
  • 단위 테스트 회귀. npm test(vitest run) 기존 통과 유지.
  • Linux 재현(핵심). 신설 CI가 ubuntu-latest에서 npm ci && npm run build 그린 — 이슈의 원 실패(Cannot find module @rollup/rollup-linux-x64-gnu)가 더는 재현되지 않음을 증명.
  • 최종 확증. CF Pages 빌드/배포 성공(Linux) — merge 후 실제 파이프라인.

7완료 조건

  • 로컬 npm ci && npm run build 통과, npm test 회귀 없음.
  • CF Pages 빌드/배포 성공(Linux).
  • 단일-플랫폼 회귀 방지책 반영 — ubuntu-latest CI의 npm ci && npm run build 완전성 체크(주) + README 주의문(보). #93은 zod 미추가로 소멸했으나 일반 재발 위험은 CI로 상시 차단.
