1요약

현재 /trends(전체 추이) 라우트를 삭제하고, 메인 화면을 하단 네비 3탭(홈·랭킹·개인) 체계로 정리한다. 동시에 관리자 전용 스택 — 로그인 → 시트 관리 홈 → (기록 입력: 날짜 선택 → 참가자 목록 → 선수별 입력) / 기록지 만들기 / 참가자 추가 — 의 라우트와 스텁 페이지 7개를 신설한다. 이번 이슈는 URL 구조와 빈 페이지 골격만 만들며, 실제 화면 내용(네비 UI·인증 가드·폼 로직)은 #45·#47·#67·#68 등 후속 이슈가 채운다. 삭제된 /trends 접근은 개인 탭으로 리다이렉트해 UX 단절을 막는다.

2현황 분석

src/App.tsx:1-18은 #44에서 만든 flat 라우트 5개: /(Home) · /rankings(Rankings) · /players(Players) · /trends(Trends) · *(NotFound). 레이아웃 래퍼나 중첩 라우트 없이 <Routes> 하나에 평면적으로 나열되어 있다.

src/pages/Players.tsx는 이미 "개인 프로필"로 라벨링되어 있고 안내문("선수별 추이 화면은 P4에서 채워집니다")이 추이 통합을 이미 전제하고 있어, 이번 이슈에서 이 파일 자체는 수정할 필요가 없다 — 삭제 대상은 src/pages/Trends.tsx뿐이다.

src/·functions/ 전체를 검색해도 admin/session 관련 프론트엔드 코드가 전혀 없다(관리자 로그인·세션·가드 컴포넌트 모두 미착수). 즉 관리자 스택은 완전한 그린필드이며, 이번 이슈가 만드는 라우트에 연결할 기존 가드 컴포넌트가 없다.

이슈 본문이 참고 문서로 지목한 docs/prd-design.html은 모든 브랜치·워크트리를 검색해도 존재하지 않는다 — 같은 시각(2026-07-17 11:21)에 한꺼번에 열린 디자인 개편 이슈군(#61·#62·#69·#70·#67·#68 등)이 공통으로 이 문서를 참조하지만 아직 작성되지 않은 상태다. 대신 docs/index.html:255-281(현재 살아있는 IA 문서)과 관련 이슈 본문들이 실질적 스펙 역할을 한다. 단, docs/index.html:273-280은 "📈 전체 추이"를 여전히 최상위 페이지로 그리는 구버전 IA라 이번 결정을 반영하지 못한 상태다 — 이 문서 갱신은 이번 이슈 범위 밖이지만 후속 문서화 시 참고.

gh issue view 45 확인 결과, #45(공통 레이아웃 + 네비게이션 — 하단 3탭 UI, 관리자 스택 뒤로가기, Outlet 레이아웃)가 "선행: #44, #62, #61"로 명시되어 있다 — 즉 실제 네비게이션 UI/레이아웃은 이번 이슈의 범위가 아니라 #45가 이 라우트들을 감싸는 방식으로 후속 작업한다. #67·#68(기록 입력 W5/W6)도 "선행 … #62(라우팅)"으로 명시되어 있어, 관리자 스택 화면들의 실제 내용(로그인 폼, 시트 관리 홈의 버튼 3개, 날짜/참가자 카드, 타입별 입력기)은 이번 이슈에서 구현하지 않고 스텁으로만 남긴다.

package.json:16 — react-router-dom@^7.18.1 이미 설치·사용 중이므로 추가 의존성 없이 <Route> 추가/삭제만으로 처리 가능하다.

3설계

선택한 설계

  • 기존 flat <Routes> 스타일을 그대로 유지한다(레이아웃 래퍼 도입 안 함) — Outlet 기반 레이아웃은 #45가 별도로 감쌀 예정이라 지금 만들면 곧 갈아엎힌다.
  • 관리자 라우트는 /admin 접두사로 묶는다 — 이미 확정된 백엔드 접두사 /api/admin/*(#64~#66)와 대칭을 맞춘다.
  • 기록 입력 3단계만 이슈 본문이 명시적으로 "날짜 선택 → 참가자 목록 → 선수별 입력"으로 분해했으므로, 이 흐름만 동적 세그먼트로 중첩한다. 기록지 만들기·참가자 추가는 이슈 본문에서 단일 항목으로 남아 있어 평면 라우트로 유지한다(과설계 방지).
  • 동적 세그먼트를 쓰는 두 스텁 페이지(참가자 목록·선수별 입력)는 useParams()로 캡처된 값을 화면에 그대로 표시한다 — 실제 UI는 #67/#68 몫이지만, 파라미터가 제대로 파싱되는지는 이번 이슈가 증명해야 할 부분이다.
  • /trends는 완전 삭제 대신 <Navigate to="/players" replace />로 리다이렉트한다 — 이슈가 "구현 시 결정"으로 열어둔 부분에 대한 결정이며, 근거는 아래 "기각한 대안" 참고.

기각한 대안

  • 관리자 라우트를 별도 서브 라우터로 완전히 분리 — react-router v7 한 트리 안에서 flat하게 처리 가능하고, 어차피 #45가 Outlet 레이아웃으로 다시 감쌀 예정이라 지금 분리해봐야 곧 재작업된다.
  • /trends를 삭제만 하고 * 404로 자연스럽게 떨어뜨림 — 오늘 막 생긴 페이지라 외부 링크 리스크는 낮지만, 내용이 개인 탭으로 이동했다는 사실을 아는 사용자가 없으므로 리다이렉트 쪽이 혼란을 원천 차단하고 비용은 <Route> 한 줄뿐이다.
  • 기록 입력 흐름의 상태를 query string(?date=&player=)으로 전달 — path segment가 react-router 관용에 더 맞고 딥링크·뒤로가기 시맨틱이 명확하다.

4작업 단계

  1. 전체 추이 페이지 삭제 src/pages/Trends.tsx
  2. App.tsx에서 Trends import·라우트를 제거하고, /trends를 <Navigate to="/players" replace />로 대체 src/App.tsx
  3. 관리자 스택 스텁 페이지 7개 신설 — 기존 Players.tsx/Rankings.tsx와 동일한 카드+h1+안내문 패턴, 안내문에 실제 구현 담당 이슈 번호 명시. RecordsParticipants·RecordsPlayerInput은 useParams() 값을 함께 표시 src/pages/admin/AdminLogin.tsx, src/pages/admin/SheetManagementHome.tsx, src/pages/admin/RecordsDateSelect.tsx, src/pages/admin/RecordsParticipants.tsx, src/pages/admin/RecordsPlayerInput.tsx, src/pages/admin/CreateSheet.tsx, src/pages/admin/AddPlayers.tsx
  4. 위 표대로 App.tsx에 관리자 라우트 7개 등록(3탭 라우트·404는 그대로 유지) src/App.tsx
  5. npm run typecheck · npm run build로 라우트 트리·미사용 import 여부 확인 — 순수 골격 이슈라 신규 vitest 케이스보다 타입체크·빌드가 더 유효한 검증(#44도 동일 방식). 실행 결과: 타입체크 통과, 빌드 통과(100 modules), npm test 19개 파일 249개 테스트 전부 통과 (파일 없음 · 커맨드)
  6. 브라우저로 전체 경로 수동 확인: /·/rankings·/players, /trends(→ /players 리다이렉트 확인), /admin/login·/admin·/admin/records·/admin/records/2025-05-16·/admin/records/2025-05-16/abc123(파라미터 표시 확인)·/admin/create-sheet·/admin/add-players, 임의 경로(404 확인). claude-in-chrome으로 12개 경로 전부 확인 — /trends는 tabs_context_mcp로 최종 URL이 /players임을 직접 확인, 동적 세그먼트 두 페이지 모두 파라미터가 화면에 정확히 표시됨 (수동 확인 완료)

5리스크

리스크영향완화책
/admin/*에 인증 가드가 아직 없어 로그인 없이 스텁 화면이 노출됨낮음 — 스텁이라 실제 데이터·쓰기 기능 없음안내문만 노출하고 실 로직은 넣지 않음. 가드는 #41·#42·#47·#67에서 별도 처리
/trends 리다이렉트가 이슈에서 "구현 시 결정"으로 열려있던 항목이라 의도와 다를 수 있음낮음 — 내부용 앱, 실사용자 없음플랜에 근거 명시, 승인 단계에서 조정 가능
#45가 이 라우트들을 감싸는 Outlet 레이아웃을 준비 중이라 경로 설계가 어긋나면 후속 이슈에서 재작업 필요중간#67·#68에 나온 화면 분해에 맞춰 경로 설계, flat <Route>라 감싸기 쉬움

6테스트 계획

  • npm run typecheck — 라우트·컴포넌트 타입 오류 없는지 확인
  • npm run build — 빌드 통과 및 Trends 등 미사용 import 없는지 확인
  • npm test(vitest) — 기존 스위트 회귀 없는지 확인(라우팅 골격 자체엔 신규 유닛 테스트 없음 — #44와 동일)
  • 수동: 브라우저에서 3탭 + 관리자 스택 각 경로 전환, /trends 리다이렉트, 동적 세그먼트 파라미터 표시, 임의 경로 404 확인

7완료 조건

  • 3탭(홈·랭킹·개인) + 관리자 스택 7개 라우트가 각각 대응 페이지를 렌더한다
  • /trends 라우트·페이지가 삭제되고, 접근 시 /players로 리다이렉트된다
  • 존재하지 않는 경로는 여전히 404로 처리된다
  • npm run typecheck · npm run build · npm test 모두 통과한다
