P2 — /api/login 로그인 시도 rate limiting (무차별 대입 방어)

damdam6/bernice 브랜치 feat/80-login-rate-limiting ← main 라벨 enhancement 2026-07-18

1요약

PR #77 코드리뷰의 🟠 major(security) 지적에 대한 후속 작업. /api/login에 시도 제한이 없어 패스코드 무차별 대입이 가능하다. 이슈의 방안 A~D 중 B안(KV 카운터)으로 확정한다 — 이 앱은 무료 *.pages.dev 도메인이라 WAF 규칙(A)을 붙일 존이 없고, Rate Limiting 바인딩(D)은 Workers 전용이라 Pages Functions에서 못 쓰며, DO(C)는 별도 Worker 배포가 필요해 팀 앱 규모에 과하다. IP당 실패 5회를 넘으면 10분 슬라이딩 윈도우 동안 429 + Retry-After를 반환하고, 성공 로그인은 카운터를 지운다. KV 바인딩이 없으면 기존 동작 그대로(fail-open) — 배포 순서에 안전하다.

2현황 분석

  • functions/api/login.ts:16-40 — onRequestPost는 JSON 파싱 → 빈/비문자열 code 거르기 → resolveRole 고정시간 비교 → 성공 시 세션 쿠키 발급, 실패 시 즉시 401. 시도 횟수 제한·지연·차단이 전혀 없다.
  • functions/api/login.ts:53-56 — constantTimeEqual은 타이밍 오라클만 막는다. 반복 시도는 자유.
  • wrangler.jsonc — 바인딩은 vars.SHEET_ID뿐. KV·DO·Rate Limiting 등 상태 저장 바인딩이 없어 애플리케이션 레벨 카운팅 인프라가 아직 없다.
  • docs/sheet-integration.html — 호스팅은 "우선 무료 *.pages.dev → 이후 커스텀 도메인" 확정. 현재 커스텀 도메인(존)이 없다.
  • functions/api/login.test.ts:11-26 — 테스트는 makeContext로 request/env만 채워 핸들러를 직접 호출하는 패턴. KV는 in-memory 모의 객체로 같은 패턴에 얹을 수 있다.
  • 클라이언트 IP는 Cloudflare가 모든 요청에 붙이는 CF-Connecting-IP 헤더로 얻을 수 있다(로컬 wrangler pages dev에서도 세팅됨).

3설계

선정: B안 — KV 실패 카운터. 탈락 사유:

  • A. WAF Rate Limiting 규칙 — 존(커스텀 도메인) 단위 기능. 현재 *.pages.dev만 쓰므로 적용할 존이 없고, 코드/레포 밖 설정이라 vitest 검증도 불가.
  • C. Durable Object — Pages Functions는 DO를 직접 정의 못 해 별도 Worker 배포가 필요. 소규모 팀 앱 로그인에 강한 일관성은 과투자.
  • D. Rate Limiting 바인딩 — unsafe.bindings 기반 Workers 전용 API. Pages Functions(wrangler.jsonc의 pages_build_output_dir 구성)에서 지원되지 않음.

동작 설계 (새 모듈 functions/lib/login-rate-limit.ts):

  • 상수: MAX_FAILURES = 5, WINDOW_SECONDS = 600(10분). KV 키 login-fail:{ip}, 값 { count, resetAt } JSON.
  • 차단 판정: 핸들러 진입 시 count ≥ MAX_FAILURES이고 now < resetAt이면 429 + Retry-After: resetAt−now + { error: 'too_many_attempts' }. 차단 중에는 카운터를 갱신하지 않는다(차단 연장 없음 → 마지막 실패로부터 10분 뒤 자동 해제).
  • 실패 기록: 401(패스코드 불일치)마다 count+1, resetAt = now + WINDOW, expirationTtl = WINDOW로 put — 슬라이딩 윈도우(연속 실패가 이어지면 창이 밀린다). TTL이 항상 600초라 KV 최소 TTL(60초) 제약과 충돌 없음.
  • 성공/무관 요청: 200 성공 시 키 delete(정상 이용자 오입력 흔적 초기화). 400(비-JSON)은 패스코드 추측이 아니므로 카운트하지 않음. 리뷰 반영(PR #81): 차단 검사가 돌려주는 hasFailures로 실패 기록이 있을 때만 delete — 이력 없는 로그인의 불필요한 KV 쓰기 제거.
  • 키 식별자: CF-Connecting-IP, 없으면 'unknown'(로컬 개발 공용 버킷 — 무해).
  • fail-open: env.LOGIN_RATE_LIMIT(KVNamespace)이 미바인딩이면 제한 로직 전체를 건너뛴다. 바인딩 생성 전에도 배포가 깨지지 않고, 로컬 dev는 wrangler가 KV를 로컬 시뮬레이션해준다.
  • 테스트 주입을 위해 시간은 now 인자(기본 Date.now())로 받는다.

5리스크

리스크영향완화책
KV eventual consistency — 여러 엣지 PoP에서 동시 시도하면 임계(5회)를 몇 회 초과해 통과 가능낮음 — 차단이 수 초~수십 초 늦어질 뿐, 무차별 대입 자체는 여전히 무력화이슈에 이미 명시된 KV의 알려진 한계로 수용하고 PR에 기록. 정밀 보장이 필요해지면 C안(DO)으로 승격
공유 IP(NAT·사내망)에서 한 명의 오입력 5회가 팀 전체를 10분 차단중간 — 정상 이용자 일시 차단임계 5회/10분은 보수적 값 + 성공 로그인 시 카운터 즉시 삭제로 오입력 흔적 초기화. 필요 시 상수만 조정
LOGIN_RATE_LIMIT 바인딩을 프로덕션에 붙이지 않으면 fail-open이라 방어가 비활성중간 — 기능이 조용히 꺼진 상태 유지PR 본문에 네임스페이스 생성·바인딩을 완료 체크리스트로 명시, wrangler.jsonc 주석에도 기록
429 신설로 프론트 로그인 폼(후속 이슈)의 에러 처리 계약 변화낮음응답 스키마를 기존 401과 동형({ error, message })으로 맞추고 Retry-After 헤더 제공

6테스트 계획

  • 모듈 테스트 (login-rate-limit.test.ts): 실패 1~4회 누적 시 비차단 → 5회째부터 차단, Retry-After = resetAt − now 정확성, 윈도우 경과(now 전진) 후 해제, clearFailures 후 재카운트 0부터.
  • 통합 테스트 (login.test.ts 확장): 틀린 코드 5회 → 6번째 요청 429 + Retry-After 헤더 + error: 'too_many_attempts'; 차단 중 올바른 코드도 429·Set-Cookie 없음; 4회 실패 후 성공 → 200이고 카운터 삭제돼 다음 실패는 1회부터; IP가 다르면 서로 독립; 비-JSON 400은 카운터 미증가; LOGIN_RATE_LIMIT 미바인딩이면 무제한 시도에도 기존 401.
  • 회귀: 기존 login.test.ts 9개 케이스가 수정 없이(또는 KV 모의만 추가된 채) 전부 통과 — 성공/정상 이용 영향 없음 확인.
  • 전체: npm run typecheck && npm test 그린.
