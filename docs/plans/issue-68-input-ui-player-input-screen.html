1요약

선수별 입력 화면(/admin/records/:sessionDate/:playerId)은 현재 라우팅만 연결된 플레이스홀더(src/pages/admin/RecordsPlayerInput.tsx:3-16)다. 이번 이슈는 이 화면을 종목 타입별 입력기(시간 분·초 2필드, 개수 스테퍼+직접입력, 면제 가능 종목 전용 토글)로 채우고, 기존 값 프리필·인라인 검증·저장(POST /api/admin/records)·성공 토스트/실패 재시도까지 완성한다. 검증 규칙이 클라이언트·서버 두 곳으로 갈라지는 걸 막기 위해 normalize-score뿐 아니라 서버가 이미 쓰는 buildEventScore(값 정규화 + valueKind 교차검증)까지 함께 shared/로 승격해, 클라이언트가 서버와 완전히 동일한 함수로 인라인 검증을 수행하게 한다.

2현황 분석

  • 라우팅은 이미 연결돼 있다 — src/App.tsx:33의 /admin/records/:sessionDate/:playerId가 AdminGate+AdminLayout(src/App.tsx:27-35) 하위라 인증 가드와 상단 뒤로가기 버튼(src/components/layout/AdminLayout.tsx:1-11)이 이미 갖춰져 있다 — 이 화면이 새로 만들 게 없다.
  • 현재 RecordsPlayerInput.tsx(src/pages/admin/RecordsPlayerInput.tsx:1-16)는 useParams로 sessionDate·playerId만 읽고 고정 문구를 렌더하는 플레이스홀더 — 데이터 조회·폼·저장 로직이 전혀 없다.
  • 상류 화면 RecordsParticipants(src/pages/admin/RecordsParticipants.tsx:51-63)가 navigate(\`/admin/records/\${session.date}/\${playerId}\`)(59행)로 진입시키고, 저장 성공 후 navigate(..., {state:{toast}})로 돌아오면 이미 useToast(76-85행)로 그 토스트를 표시하도록 완성돼 있다 — 이 화면은 돌아갈 때 그 계약(state.toast 문자열)만 지키면 된다.
  • 도메인 타입: RecordsResponse.sessions[].entries[](shared/domain.ts:54-60) — SessionEntry.scores는 events[] 전체 key가 항상 존재함이 타입 계약(57행 주석)이라 프리필 루프에서 null 체크가 필요 없다.
  • 서버 계약은 이미 구현·머지돼 있다 — POST /api/admin/records(functions/api/admin/records.ts:1-197)는 {sessionDate, playerId, scores: Record<string,string>}을 받아 200에 scores: Record<string, EventScore>(recorded/exempt/unmeasured/invalid)를 돌려준다. 에러: 400 validation_failed, 404 session_not_found/not_participant, 502 sheets_api_error(164-197행).
  • 429·5xx 재시도는 이미 서버 안에서 끝난다 — functions/lib/sheetsWriteApi.ts:21-43의 백오프 재시도 루프(PRD §07 설계 근거: "재시도는 서버 안에서"). 클라이언트는 한 번만 호출하고, 실패하면 에러 문구 + 입력값 보존만 하면 된다(재시도 버튼은 곧 저장 버튼 재클릭).
  • 검증 규칙의 단일 원천은 두 함수다: normalizeScore(functions/lib/normalize-score.ts:12-55)와 이를 감싸 EventDefinition.valueKind 교차검증까지 하는 buildEventScore(functions/lib/parse-session.ts:169-198). 둘 다 순수 함수이고 Sheets·행/열 개념에 전혀 의존하지 않는데, 지금은 functions/lib(서버 전용 디렉터리)에만 있어 프론트가 재사용할 수 없다.
  • 이 화면이 필요로 하는 모든 UI 패턴은 이미 형제 화면들이 확립해뒀다 — 하단 고정 액션 바 + 성공 시 토스트(src/pages/admin/CreateSheet.tsx:45-93, src/pages/admin/AddPlayers.tsx:78-131), {ok:true|false} 클라이언트 API 래퍼 모양(src/lib/create-sheet-api.ts:5-39, src/lib/add-players-api.ts:3-42), 저장 성공 시 navigate 전에 queryClient.invalidateQueries({queryKey: RECORDS_QUERY_KEY, exact:true}) await(CreateSheet.tsx:57-60).
  • 면제 가능 종목 플래그는 프론트 어디에도 아직 없다 — PRD가 "프론트 종목 상수로 시작"을 권장(디자인 PRD §09 열린 항목)했고, 지금은 45도패스캐치 하나만 해당(docs/sheet-integration.html:133, scripts/seed-sheet.mjs:43).
  • shared/엔 지금 domain.ts 하나뿐이지만, tsconfig.app.json:16(include에 "shared")과 functions/tsconfig.json:15(include에 "../shared")가 이미 양쪽 다 포함하고 있어 새 파일을 shared/에 두는 데 설정 변경이 필요 없다.

3설계

  • 상태 모델 — 이벤트별 로컬 폼 상태를 RecordsPlayerInput이 소유하는 판별 유니언 FieldState(시간: minutes/seconds, 개수: count, 공통: exempt 불리언)로 둔다. EventDefinition.valueKind가 필드 모양을 결정하므로 타입 오분기가 애초에 불가능하다. (대안: 필드별로 원시 문자열 하나만 들고 있다가 파싱 — mm/ss 두 입력을 합성 문자열 하나로 관리해야 해 UI 상태와 어긋나 기각)
  • 검증 재사용 범위 확장 — 이슈 본문은 "normalize-score의 shared/ 승격 검토"만 언급하지만, buildEventScore까지 함께 옮긴다. 이유: buildEventScore는 normalizeScore + valueKind 교차검증만 하는 순수 함수라 승격 비용이 없고, 옮기지 않으면 클라이언트가 valueKind 교차검증 규칙(예: 개수 종목에 시간형 입력)을 별도로 재구현해야 해 PRD §08 "검증 규칙의 원천은 하나"가 두 곳으로 갈라진다. (대안: normalize-score만 옮기고 교차검증은 클라이언트에서 재작성 — 드리프트 위험이 있어 기각)
  • 저장 payload 조립 — buildFieldRaw(field)가 FieldState를 서버가 기대하는 원본 문자열("mm:ss" / 숫자 문자열 / "면제" / "")로 변환한다. 조립된 문자열을 다시 buildEventScore(서버와 동일 함수)에 통과시켜 인라인 에러 문구와 저장 버튼 활성/비활성을 판단한다 — 별도 규칙을 새로 만들지 않는다.
  • 개수 필드 클램프 비대칭 — 스테퍼(±)는 maxScore에서 상한 클램프(PRD §05 명시), 직접 입력은 만점 초과를 허용(§08 "만점 초과는 막지 않는다" 그대로). 하한 0은 스테퍼·직접입력 모두에 적용하는데, 이는 normalizeScore의 parseCount(shared/normalize-score.ts 이동 후, 정수만 허용하는 INTEGER_RE)가 애초에 음수 표기를 받지 않는 것과 일관된 해석이다.
  • 재시도 UX — 서버가 이미 429/5xx 백오프를 담당하므로, 클라이언트는 CreateSheet·AddPlayers와 동일하게 실패 시 에러 문구 + 기존 입력값 보존만 하고 "다시 저장"은 같은 저장 버튼 재클릭으로 처리한다. 별도 재시도 버튼·로직을 새로 만들지 않는다.
  • 컴포넌트 분해 — 재사용 가치가 있는 원자 단위(TimeScoreInput/CountScoreInput/ExemptToggle)만 컴포넌트로 분리한다. 종목 카드 레이아웃 자체는 이 화면 하나에서만 쓰이는 조합이라 RecordsPlayerInput.tsx 안에서 인라인으로 조립한다 — SelectablePlayerList처럼 실제로 2개 이상 화면이 공유할 때만 별도 컴포넌트로 분리하는 기존 관례를 따른다.

5리스크

리스크영향완화책
shared/ 이동 시 import 경로 누락서버 빌드·테스트 깨짐(기능 회귀는 아님)이동 대상 임포터 사전 전수 조사 완료(grep 기준 정확히 2개 파일) + npm run typecheck가 즉시 드러냄
mm/ss 부분 입력(분만 채움 등)의 raw 조립 규칙이 서버 판정과 어긋날 가능성정상값이 invalid로 오판되거나, 반대로 잘못된 값이 저장 시도됨조립된 raw 문자열을 그대로 buildEventScore(서버와 동일 함수)에 통과시켜 판정 — 별도 규칙 재구현 없음
개수 필드 하한(0) 클램프가 PRD에 명시적으로 언급되지 않음(상한 클램프만 명시)스테퍼로 음수를 만들 수 있으면 저장 시 invalid로 막히긴 하지만 UX가 어색함normalizeScore의 정수 전용 규칙과 일관되게 스테퍼·직접입력 모두 하한 0으로 통일해 구현(§3 설계에 근거 명시)
저장 성공 후 invalidateQueries와 navigate 순서가 뒤바뀌면 참가자 목록이 옛 상태 뱃지를 보여줄 수 있음저장했는데 "미입력" 뱃지가 그대로 보임CreateSheet.tsx:57-60·AddPlayers.tsx:90-93과 동일하게 invalidate를 await한 뒤 navigate
