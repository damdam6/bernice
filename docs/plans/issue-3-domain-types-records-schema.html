#3기능 개선 (설계 · 리뷰 필수)

P1 — 도메인 타입 & /api/records JSON 응답 계약 설계

1요약

P1(백엔드 데이터 레이어)의 크리티컬 패스 시작점 — 파서·랭킹 계산·/api/records 엔드포인트 이슈와 P3~P4 프론트 전체가 이 계약에 의존한다. 이번 이슈는 구현이 아니라 설계: 명단(4종 상태)·목표(종목·방향)·회차(점수, 면제/미측정/시간형)를 표현하는 공용 TypeScript 도메인 타입과, 프론트가 추가 가공 없이 바로 쓸 수 있는 /api/records JSON 응답 스키마를 확정한다. 아래 3. 설계 섹션이 이슈가 요구하는 "설계안"이며, 사용자 리뷰·승인 후 4. 작업 단계를 실행해 shared/domain.ts와 스키마 문서를 커밋한다. 실제 시트 파서·랭킹 계산·엔드포인트 구현은 이 계약이 확정된 뒤 별도 이슈에서 진행한다(범위 밖).

2현황 분석

  • 도메인 타입 없음 — 레포에 TS 소스는 src/(프론트)와 functions/(Pages Functions) 두 갈래뿐이고, 실제 로직은 functions/api/health.ts:1-2 헬스체크가 전부. /api/records도, 이를 뒷받침할 타입도 아직 없다.
  • 두 TS 프로그램이 분리돼 있음 — tsconfig.app.json:16의 include: ["src"]와 functions/tsconfig.json:12의 include: ["."]는 서로 다른 루트를 본다. 공용 타입 파일을 src/ 안에 두면 functions/가 프론트 폴더를 상대경로로 끌어와야 하는 역전이 생기고, 반대로 functions/ 안에 두면 프론트가 백엔드 폴더를 참조해야 한다 — 둘 다 소유권이 애매해진다.
  • 시트 스키마는 이미 확정·시딩까지 완료 — docs/sheet-integration.html §02가 탭 3종(버니스명단/목표/회차)의 정확한 컬럼 구조를 정의하고, scripts/seed-sheet.mjs:29-44에 그 구조가 실제 값으로 시딩돼 있다: 버니스명단은 이름·상태 2열(21행), 목표는 종목·목표·만점·방향 4열, 회차는 이름(참조 수식) + 종목별 점수 열.
  • 선수 상태 4종 드롭다운이 코드로 이미 고정됨 — scripts/seed-sheet.mjs:176의 ONE_OF_LIST가 활동/탈퇴/비대상/휴식 정확히 그 4개 문자열로 시트 드롭다운을 제한한다. 타입 쪽 리터럴 유니온이 이 4개 값과 정확히 일치해야 드리프트가 없다.
  • 점수 특수값 규칙은 docs/sheet-integration.html §03에 이미 표로 정리됨 — 면제(면제 문자열, 랭킹 제외·0점 아님) · 빈칸(그 종목만 미측정) · 점수 전부 빈칸(회차 전체 미참여) · 셔틀런의 1:15(분:초) 형식. docs/sheet-rules.html:174-178은 이 1:15가 구글 시트에서 시각으로 자동 변환되는 함정까지 별도 경고한다.
  • 랭킹·달성 규칙도 확정됨 — docs/sheet-integration.html:161-168: 종목별 랭킹만(종합 없음), 방향은 목표 탭 방향 열 기준(셔틀런만 낮을수록), 값은 정규화 없이 원값 그대로 표시, 달성 여부는 뱃지로만 표현.
  • 선수 식별은 id 없이 행 위치 — docs/sheet-integration.html:107,170-177: 버니스명단의 행 위치가 곧 신원이고 회차 이름 열은 그 행을 참조하는 수식이라, 개명해도 추이가 끊기지 않는다. API가 프론트에 내려줄 안정적인 id는 이 행 위치에서 파생해야 한다(시트에 별도 id 열은 없음).
  • 남은 블로커 — docs/build-plan.html:142,144가 "비대상·휴식의 랭킹·표시 동작"과 "JSON 응답 계약 스키마"를 미정 블로커로 남겨뒀는데, 이번 이슈 본문이 상태 정책은 이미 확정해서 던져줬고(탈퇴=전체 숨김 / 비대상·휴식=랭킹·달성률 제외하되 프로필엔 뱃지와 함께 노출) JSON 계약은 이번 이슈가 직접 확정하는 대상이다.

3설계

3-1. 공용 타입 위치 — shared/domain.ts

src/도 functions/도 소유하지 않는 새 최상위 폴더 shared/에 타입 전용 파일 하나(shared/domain.ts)를 둔다. tsconfig.app.json의 include에 "shared"를, functions/tsconfig.json의 include에 "../shared"를 추가해 두 프로그램이 대칭적으로 같은 파일을 본다. 값은 거의 없고(작은 상수 배열 몇 개) 대부분 타입/인터페이스라 번들 크기에 영향 없음.

3-2. 상태 4종 정책의 타입 반영

PlayerStatus는 시트 드롭다운 문자열을 그대로 리터럴 유니온으로 사용한다(번역 매핑 레이어를 두지 않음 — 아래 기각안 참고). 정책은 타입이 아니라 응답 구성 규칙으로 반영한다:

  • 탈퇴 = 사이트 전체 숨김 → /api/records 응답 어디에도(players, sessions[].entries, rankings) 등장하지 않는다. 파서 단계에서 완전히 걸러낸다.
  • 비대상·휴식 = players[]에는 포함(상태 뱃지용 status 필드 포함), 과거 sessions[].entries에도 정상 포함(프로필에서 과거 기록 열람 가능) — 하지만 rankings와 home.achievementRates 집계 대상에서는 제외.
  • 활동 = 전 영역에 포함되고 랭킹·달성률 집계 대상.
⚠️ 확인 요청 — "탈퇴 = 사이트 전체 숨김"의 해석 범위
이슈 문구를 문자 그대로 읽어 과거 회차 기록(sessions)까지 완전히 제외하는 것으로 설계했다. docs/sheet-rules.html:84의 기존 문구("과거 기록은 보존되고 현재 랭킹에서만 빠진다")보다 더 엄격한 해석인데, 이슈 본문이 더 최신 결정이라 우선했다. 만약 "탈퇴자도 과거 회차 표에는 남기고 목록·랭킹에서만 뺀다"는 의도였다면 SessionEntry 필터링 지점만 바꾸면 되는 국소적인 변경이다 — 리뷰에서 확정해달라.

3-3. 점수 특수값 표현 — ScoreStatus + 정규화

모든 점수 칸을 { status, value, display } 3필드로 통일한다. status가 recorded일 때만 value(정렬·판정용 숫자 — 시간형은 총 초, 개수형은 그대로)와 display(원본 표시 문자열 — "1:12" 또는 "6")가 채워진다. 면제는 exempt, 빈칸(그 종목만 미측정)은 unmeasured. 회차 전체 미참여는 별도 상태값이 아니라 SessionEntry.participated(모든 종목이 unmeasured면 false)로 파생 표현 — "면제/빈칸/미참여"를 표에서 그대로 구분해 보여줘야 하는 개인 프로필·회차 상세 화면과, 깨끗한 숫자만 필요한 랭킹·추이 화면 양쪽을 하나의 필드셋으로 커버한다.

3-4. 종목 정의 — EventValueKind

목표 탭에는 "이 종목이 시간형인지 개수형인지"를 나타내는 열이 따로 없다(방향 열만 있음). 콜론 포함 여부로 파서가 valueKind('time' | 'count')를 판별해 EventDefinition에 명시적으로 싣는다 — 방향(낮을수록/높을수록)과 값 형식(시간/개수)이 우연히 지금은 1:1(셔틀런만 낮을수록·시간형)이지만 개념적으로 독립이라 나중에 "낮을수록 좋은 개수형 종목"이 추가돼도 깨지지 않는다.

3-5. 타입 정의 전문 (shared/domain.ts)

// shared/domain.ts
// 버니스 실력 기록 — 공용 도메인 타입. 파서·계산(functions)과 프론트(src)가 함께 import.
// 근거: docs/sheet-integration.html(시트 스키마) · docs/sheet-rules.html(운영 규칙)

// ── 선수 상태 — 버니스명단 상태 드롭다운 4종 (scripts/seed-sheet.mjs:176과 동일해야 함) ──
export const PLAYER_STATUSES = ['활동', '탈퇴', '비대상', '휴식'] as const
export type PlayerStatus = (typeof PLAYER_STATUSES)[number]

// ── 종목 정렬 방향 — 목표 탭 방향 드롭다운 ──────────────────────────────
export const RANK_DIRECTIONS = ['낮을수록', '높을수록'] as const
export type RankDirection = (typeof RANK_DIRECTIONS)[number]

// ── 점수 값 형식 — 셀 표기로 파서가 판별(콜론 포함 = 시간) ────────────────
export const EVENT_VALUE_KINDS = ['count', 'time'] as const
export type EventValueKind = (typeof EVENT_VALUE_KINDS)[number]

// ── 개별 점수 상태 — 정상 기록 · 면제 · 미측정 ────────────────────────────
export const SCORE_STATUSES = ['recorded', 'exempt', 'unmeasured'] as const
export type ScoreStatus = (typeof SCORE_STATUSES)[number]

/** 선수 — 버니스명단 1행 = 선수 1명.
 *  id는 시트 열이 아니라 행 위치 기반 파생값(1부터, 헤더 제외).
 *  행 순서 = 신원(정렬·삭제 금지, docs/sheet-rules.html §01)이라 재계산해도 항상 안정적. */
export interface Player {
  id: number
  name: string
  status: PlayerStatus
}

/** 종목 정의 — 목표 탭 1행 = 종목 1개. key가 회차 점수 컬럼·랭킹·추이를 잇는 유일한 식별자. */
export interface EventDefinition {
  key: string
  valueKind: EventValueKind
  target: string            // 목표치 원본 표시값 ("1:17" | "5")
  targetValue: number        // 정규화 목표치(시간=초, 개수=그대로) — 달성 판정 기준
  maxScore: number | null    // 만점(표시용 분모). 없으면 null (예: 셔틀런)
  direction: RankDirection
}

/** 회차 1건, 종목 1개의 점수 한 칸. */
export interface EventScore {
  status: ScoreStatus
  value: number | null    // 정규화 값. status가 'recorded'가 아니면 null
  display: string | null  // 원본 표시값 그대로("1:12" | "6"). status가 'recorded'가 아니면 null
}

/** 회차 1건에서 선수 1명의 기록 행. */
export interface SessionEntry {
  playerId: number
  name: string                          // 그 시점 이름(참조 수식이라 개명 시 항상 최신과 동일)
  scores: Record    // key = EventDefinition.key
  participated: boolean                 // 전 종목 unmeasured면 false (회차 전체 미참여)
}

/** 회차 1개 = 날짜 탭 1개. */
export interface Session {
  date: string              // YYYY-MM-DD, 탭 이름 원본
  entries: SessionEntry[]   // 그 회차 탭에 실제로 존재하는 명단 행만
                             // (이후 가입자는 과거 회차에 엔트리 자체가 없음)
}

/** 종목별 랭킹 1행. */
export interface RankingEntry {
  playerId: number
  name: string
  value: number
  display: string
  rank: number        // 1부터. 동점은 공동순위(예: 1,1,3) — 다음 등수는 동점자 수만큼 건너뜀
  achieved: boolean
}

export interface EventRanking {
  event: string              // EventDefinition.key
  entries: RankingEntry[]    // rank 오름차순. 면제·미측정·활동 외 상태 제외
}

/** 회차 1개에 대한 종목별 랭킹 묶음 — 랭킹 화면의 "회차 선택 → 종목 탭" 흐름과 그대로 대응. */
export interface SessionRankings {
  sessionDate: string
  events: EventRanking[]
}

/** 개인 추이 — 종목 1개의 회차별 기록 흐름. */
export interface TrendPoint {
  sessionDate: string
  value: number
  display: string
  achieved: boolean
  deltaFromPrevious: number | null   // 직전 유효 기록 대비 변화. 첫 기록이면 null
}

export interface PlayerEventTrend {
  event: string
  points: TrendPoint[]    // 유효 기록을 남긴 회차만(면제·미측정 제외), 날짜 오름차순
}

export interface PlayerPersonalBest {
  event: string
  value: number
  display: string
  sessionDate: string
  achieved: boolean
}

/** 선수 1명의 프로필 요약(추이·PB 포함). 탈퇴 상태는 응답 어디에도 등장하지 않음. */
export interface PlayerSummary extends Player {
  trends: PlayerEventTrend[]
  personalBests: PlayerPersonalBest[]
}

export interface EventAchievementRate {
  event: string
  achievedCount: number
  eligibleCount: number   // 면제·미측정 제외한 분모, 활동 상태만 집계
  rate: number             // 0~1
}

export interface HomeSummary {
  latestSession: { date: string; participantCount: number } | null
  achievementRates: EventAchievementRate[]
}

/** GET /api/records 최종 응답. */
export interface RecordsResponse {
  generatedAt: string          // ISO — 캐시 확인용
  events: EventDefinition[]    // 목표 탭 행 순서
  players: PlayerSummary[]     // 활동·비대상·휴식만(탈퇴 제외). 버니스명단 행 순서
  sessions: Session[]          // 날짜 오름차순
  rankings: SessionRankings[]  // 날짜 오름차순, 활동 상태만 집계 대상
  home: HomeSummary
}

⚠️ 확인 요청 — 동점자 랭킹 처리
어느 문서에도 동점 처리 규칙이 없어 표준 공동순위(1,1,3 — 동점자는 같은 등수, 다음 등수는 동점 인원수만큼 건너뜀)를 기본값으로 제안한다. RankingEntry.rank의 계산 방식만 바뀌는 문제라 다른 필드에 영향 없음 — 다른 방식(예: 1,2,2)을 원하면 리뷰에서 알려달라.

3-7. 기각한 대안

타입을 src/types/에만 두고 functions가 상대경로로 가져오기 — 기각: functions/tsconfig.json의 include가 "."뿐이라 프론트 폴더를 참조하면 백엔드가 프론트 구조에 종속되는 레이어 역전이 생김.
랭킹·추이 계산을 프론트에서 sessions 원본으로부터 수행 — 기각: 이슈가 명시한 "프론트가 추가 가공 없이 소비 가능한 형태" 요건 위반. 정렬·판정 로직이 프론트에도 중복 구현되면 백엔드와 결과가 어긋날 위험.
점수를 원본 문자열만 내려주고 프론트가 시간(콜론) 파싱 — 기각: 셔틀런 정렬·PB 비교 로직이 프론트마다 재구현되며 랭킹 기준이 어긋날 위험. 백엔드가 한 번만 정규화하는 편이 안전.
상태값을 영문 enum(ACTIVE/WITHDRAWN 등)으로 변환 — 기각: 시트 드롭다운 문구가 곧 화면 뱃지 텍스트라, 변환 레이어를 두면 드롭다운 문구가 바뀔 때 코드도 별도로 바꿔야 하는 드리프트 지점이 생김. 한글 리터럴 그대로가 파서를 단순 패스스루로 유지.
Zod 등 런타임 스키마 검증 라이브러리 도입 — 기각: 이번 이슈 범위는 타입·계약 설계이며 런타임 검증은 실제 파서 구현 이슈에서 필요 시 판단. 신규 의존성 추가는 범위 밖.
