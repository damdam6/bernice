타입 안정성 — API 응답 as 타입 단언 제거 + 런타임 검증 도입

damdam6/bernice 브랜치 feat/93-type-safety-remove-as-assertions ← main 라벨 enhancement 2026-07-19

1요약

프론트가 자체 백엔드 응답을 as로 단언해 쓰는 5개 파일(useRecords·login-api·add-players-api·create-sheet-api·records-write-api)에서 단언을 제거하고, 레포 관용구에 맞는 손수 조립형 파서("parse, don't validate")로 런타임 검증을 도입한다. zod는 도입하지 않는다(사유는 §3). 서버측은 완료 조건이 명시한 request.json() 단언 3곳을 이미 모범인 records.ts의 raw: unknown 관용구로 통일한다. 검증 실패 시 에러 경로는 읽기(useRecords)=ApiError throw, 쓰기 래퍼=기존 기본값 폴백(UX 불변)으로 정의한다.

2현황 분석

클라이언트 — 자체 API 응답 as 단언 5곳

  • src/hooks/useRecords.ts:33 — (await res.json()) as RecordsResponse. 방어 로직 없는 날것이며, 이 응답은 7개 화면(Rankings·AddPlayers·CreateSheet·RecordsDateSelect·RecordsParticipants·RecordsPlayerInput 등)이 깊게 소비한다. src/hooks/useRecords.ts:28의 에러 바디 as ErrorBody | null도 동일 파일.
  • src/lib/login-api.ts:23 — as { role?: LoginRole; message?: string } | null. role은 AdminLogin.tsx:29가 result.role !== 'admin'으로 분기하는 보안 관련 값인데 런타임 검증이 없다.
  • src/lib/add-players-api.ts:42 · src/lib/create-sheet-api.ts:39 · src/lib/records-write-api.ts:48 — .catch(() => null) as {…} | null 후 body?.x ?? 기본값 방어형. 성공 필드 소비는 얕다(토스트 문구·네비게이션: AddPlayers.tsx:91-92, CreateSheet.tsx:58-59, RecordsPlayerInput.tsx:111-112는 ok/message만 사용).

서버 — request.json() 단언 3곳 + 사내 모범 선례 1곳

  • functions/api/login.ts:35 · functions/api/admin/add-players.ts:34 · functions/api/admin/create-sheet.ts:34 — as { x?: unknown } | null 컨테이너 단언 후 값은 typeof로 검증. 값 수준에선 안전하지만 관용구가 갈라져 있다.
  • functions/api/admin/records.ts:44-48, 134-162 — 이미 모범: raw: unknown으로 받고 parseBody(+isPlainObject, records.ts:130)가 값을 검증하며 새 객체를 조립해 반환. 이 관용구가 이번 작업의 표준이다.

리뷰 스레드(미해결) 및 테스트 기반

  • PR #54(useRecords, "zod/io-ts 도입" 제안) · #91(add-players/create-sheet) · #92(records-write, "타입 단언 제거") — 모두 미해결. #91 스레드의 CSRF 지적은 이 이슈 범위가 아니다(별도 처리 대상).
  • 테스트 픽스처는 전부 RecordsResponse로 타입돼 있어(예: src/hooks/useRecords.test.tsx:18, src/App.e2e.test.tsx:124) 완전한 형태다 — 엄격 파서를 도입해도 기존 픽스처가 그대로 통과하며, E2E가 실제 fetchRecords 경로로 파서를 검증하게 된다.

3설계

결정 ① 도입 방식 — zod 기각, 손수 조립형 파서 채택

  • 레포 관용구: 이 레포는 constant-time-equal(functions/lib/constant-time-equal.ts), 세션 쿠키 HMAC(session-cookie.ts), Google 서비스 계정 JWT 서명(googleAuth.ts)까지 전부 무의존 손수 구현이고, 런타임 의존성은 React 계열뿐이다. records.ts의 parseBody가 정확히 같은 문제의 사내 선례.
  • 계약의 원천 유지: shared/domain.ts가 서버·프론트 공용 계약이다. zod를 쓰면 스키마·타입 이중 관리 또는 domain.ts의 z.infer 전면 개편이 필요 — 이번 이슈 규모(M)를 넘는 전역 개편이라 기각.
  • 조립형("parse, don't validate")의 강점: raw is T 단언형 가드는 domain.ts에 필수 필드가 추가돼도 조용히 통과하지만, 값을 꺼내 T를 새로 조립해 반환하면 필수 필드 누락이 컴파일 에러로 강제된다.

결정 ② 검증 실패 시 에러 처리 경로

  • 읽기(useRecords): 200 + 계약 위반 바디 → ApiError('records fetch failed (invalid body)', res.status) throw. 기존 invalid json 경로(useRecords.ts:36)와 같은 분류라 재시도 규칙(shouldRetryRecordsQuery: 200은 4xx가 아니므로 재시도 허용)도 그대로 맞아떨어진다.
  • 쓰기 래퍼 3종 + login: 필드 단위 검증 추출 + 기존 기본값 폴백 유지(UX 불변). 2xx인데 바디가 깨진 경우 서버 쓰기는 이미 성공한 상태라, 실패로 표시해 재시도를 유도하면 add-players는 409 충돌을 만든다 — 관대한 폴백이 옳다. 에러 경로(비-2xx)는 기존 unknown_error + 일반 메시지 폴백 그대로. login-api의 role은 'team' | 'admin' 화이트리스트 검증(그 외 값 → undefined → AdminLogin이 실패로 처리, 기존 분기 재사용).

결정 ③ 서버측 범위 — request.json() 3곳만, 나머지는 제외(사유 기록)

  • 반영: functions/api/login.ts · admin/add-players.ts · admin/create-sheet.ts의 컨테이너 단언을 records.ts 관용구(raw: unknown + isPlainObject 좁히기)로 통일. 동작·응답 계약 불변(기존 핸들러 테스트가 회귀 방어).
  • 제외: sheetsApi.ts:46,75,105(외부 Google API 메타데이터 — 다운스트림 파서가 방어, 전면 스키마 검증은 ROI 낮음) · googleAuth.ts:67·session-cookie.ts:113(값을 unknown으로 받아 이미 런타임 검증하는 안전 패턴) · login-rate-limit.ts:39(자체 직렬화 KV 값) · 테스트 파일의 as(목 조립용) · location.state as(라우터 상태 — API 응답 아님, 이슈 제목 범위 밖).

새 모듈 배치

  • shared/is-plain-object.ts — records.ts:130의 isPlainObject를 승격(서버·프론트 공용, 중복 제거).
  • src/lib/parse-records-response.ts — parseRecordsResponse(raw: unknown): RecordsResponse | null 전체 깊이 조립 파서. parseEventScore도 export해 records-write-api의 성공 scores 검증과 단일 소스로 공유. 소비자가 프론트뿐이라 shared/가 아닌 src/lib에 둔다.

5리스크

리스크영향완화책
파서가 domain.ts와 어긋나 정상 응답을 거부 전 화면 데이터 로드 실패(가장 큰 리스크) 조립형 파서라 신규 필수 필드는 컴파일 에러로 강제. App.e2e.test.tsx가 실제 fetchRecords 경로로 완전 픽스처를 통과시켜 회귀 감지. 파서 실패는 재시도 가능한 ApiError로 화면 에러 UI에 안착.
엄격 검증이 기존에 통과하던 관대한 응답을 거부(동작 변화) 쓰기 성공이 실패로 표시 → 재시도 유도 → add-players 409 혼란 엄격 모드는 읽기(useRecords)만. 쓰기 래퍼·login은 필드 단위 폴백으로 기존 UX 완전 보존.
서버 바디 파싱 개편의 회귀 400/응답 계약 변화 동작 불변 원칙(컨테이너 단언 → unknown 좁히기만 교체). 각 핸들러의 기존 테스트(비-JSON 400, 검증 실패 케이스)가 커버.
EventScore 검증 로직 이중화로 인한 불일치 읽기/쓰기 경로 간 판정 상이 parseEventScore를 단일 소스로 두고 parse-records-response와 records-write-api가 공유.
