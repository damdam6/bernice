1요약

회차 탭(3개월마다 추가되는 날짜 이름 탭)의 원시 2D 배열을 shared/domain.ts의 Session 도메인 타입으로 변환하는 순수 함수 parseSession을 functions/lib/parse-session.ts에 신규 작성한다. 점수 셀은 이미 병합된 normalizeScore(functions/lib/normalize-score.ts, 이슈 #6 → #20)로 정규화하고, 선수 식별은 이슈 #3(→ #21)이 확정한 "행 위치가 신원" 원칙을 그대로 적용해 회차 탭의 데이터 행 위치로 playerId를 도출하며 회차 탭 자체의 이름 셀(참조 수식 결과)을 명단과 대조하는 무결성 검증으로 쓴다. 시트 연동 없이 픽스처 기반 테스트만으로 완결한다.

2현황 분석

functions/lib/에는 이미 세 조각의 기반이 준비돼 있고, 회차 파서 자체는 아직 없다.

  • 도메인 계약 — shared/domain.ts:24-31의 Player는 "id는 시트 행 위치 기반 파생값(1부터, 헤더 제외)"이라고 명시한다. shared/domain.ts:53-60 SessionEntry는 playerId · name(그 시점 이름) · scores: Record<EventDefinition.key, EventScore> · participated를 요구하고, shared/domain.ts:62-67 Session은 { date, entries[] }다. EventScore(shared/domain.ts:45-51)는 status로 분기하는 판별 유니온이다.
  • 값 정규화 — functions/lib/normalize-score.ts:12-22 normalizeScore(raw)는 ScoreValue(kind: 'count' | 'seconds' | 'exempt' | 'blank' | 'invalid')를 돌려준다. NFKC 정규화, 분 상한(999), 초 범위(0-59), 3파트(h:mm:ss) 자동 변환 의심 케이스까지 이미 처리돼 있다(functions/lib/normalize-score.test.ts).
  • 탭 분류 — functions/lib/sheetTabs.ts:53-86 classifySheetTabs가 탭 이름을 YYYY-MM-DD 정규식 + 캘린더 왕복 검증으로 걸러 RoundTab { name, date }를 만든다. 이 파서는 그 결과물(유효성 검증이 끝난 탭 이름)을 그대로 Session.date로 재사용하면 되고, 날짜 형식을 다시 검증할 필요가 없다.
  • 시트 스키마(docs/sheet-integration.html §02, docs/sheet-rules.html §01/§04) — 회차 탭 헤더는 이름 | 종목1 | 종목2 | … 고정이고, 종목 헤더 셀은 ='목표'!A2 같은 참조 수식이라 목표 탭 행 순서와 항상 같은 순서로 나타난다. 데이터 행의 이름 셀은 ='버니스명단'!A2 참조 수식이고, 회차 탭은 명단 전원을 행 순서 그대로 미러링한다(중간 삽입·재배치·삭제 금지 — 이 규칙이 깨지면 "이름은 밀려 바뀌는데 점수는 그대로"인 사고가 난다).
  • 의존 이슈 상태 — 이슈 본문의 #3/#6은 각각 #21(도메인 타입, main 병합됨)·#20(값 정규화, main 병합됨)를 가리키며 그대로 재사용한다. 반면 #25(명단 파서)·#26(목표 파서)는 아직 미구현(OPEN)이다 — 이 파서는 그 둘의 실제 구현이 아니라 이미 확정된 Player[]/EventDefinition[] 타입 계약(shared/domain.ts)만 입력으로 받아 픽스처로 개발한다. 타입 계약만 맞으면 #25/#26이 나중에 구현돼도 이 파서는 그대로 재사용된다.
  • 탈퇴 필터링 책임 소재 — docs/records-schema.html §02는 "탈퇴 = 응답 어디에도 등장하지 않음, 파서 단계에서 제외"라고 적혀 있지만, Session.entries 주석(shared/domain.ts:65-66)은 "그 회차 탭에 실제로 존재하는 명단 행만"이라고만 하고 상태 필터링은 언급하지 않는다. 다운스트림 이슈(#28 랭킹은 활동만, #29 프로필 추이는 비대상·휴식도 포함)는 PlayerStatus별로 서로 다른 포함 규칙을 Session.entries 전체에 대해 적용해야 하므로, 이 파서가 상태로 미리 걸러내면 하류에서 필요한 정보를 복원할 수 없다. 따라서 이 이슈는 상태 필터링을 하지 않는 것으로 설계한다(§3 참고).

3설계

3-1. 시그니처

// functions/lib/parse-session.ts
export function parseSession(
  tabName: string,               // 이미 검증된 회차 탭 이름 (예: sheetTabs.RoundTab.name) → Session.date로 그대로 사용
  rows: string[][],              // Sheets API 원시 2D 배열, rows[0] = 헤더
  players: Player[],             // 명단 파서(#25) 출력과 같은 계약. id = 1-based 행 위치
  events: EventDefinition[],     // 목표 파서(#26) 출력과 같은 계약. key로 헤더 매칭
): Session

날짜 형식 재검증은 하지 않는다 — classifySheetTabs가 이미 캘린더 왕복 검증까지 끝낸 값을 넘겨받는다는 계약으로 둔다(중복 검증 대신 호출부 책임으로 명시).

3-2. 선수 매칭 — 행 위치가 1차 키, 이름은 무결성 검증

완료 조건 "명단 행 위치/이름 기준 매칭 전략이 명단 파서(#25)와 일관됨"을 다음과 같이 해석한다: 회차 탭의 데이터 행 N(1부터, 헤더 제외)은 항상 ='버니스명단'!A(N+1)을 참조하므로(전원 미러링 + append-only 규칙), playerId = N을 행 위치로 직접 도출하는 것이 Player.id(shared/domain.ts:25 "행 위치 기반 파생값")와 동일한 컨벤션이다. 이름 셀은 수식이 정확히 해석됐는지 확인하는 2차 무결성 검증으로 쓴다: players[N-1]이 존재해야 하고, 그 .name이 회차 탭의 이름 셀(NFC 정규화 후)과 일치해야 한다. 불일치·범위 초과는 시트 편집 규칙(§01) 위반이 실제로 일어났다는 뜻이므로 조용히 무시하지 않고 Error를 던진다 — 잘못 매칭된 점수가 다른 사람 기록으로 섞이는 사고보다, 새로고침 시 에러로 드러나 관리자가 시트를 고치는 편이 훨씬 안전하다.

이름만으로 매칭하지 않는 이유: 회차 탭 이름 셀은 참조 수식이라 항상 "현재" 명단 이름과 같아야 하지만(개명 반영), 이것만으로 매칭하면 이론상 동명이인이나 공백/정규화 차이에 취약하다. 행 위치는 시트의 물리적 구조(참조 수식이 가리키는 행)에서 직접 나오는 값이라 더 견고하고, Player.id 정의와 정확히 같은 축을 쓴다.

3-3. 종목 헤더 매칭

헤더 행(rows[0])의 첫 칸은 "이름"과 NFC 일치해야 한다. 이후 칸들은 각각 NFC 정규화 후 events[].key와 매칭한다(위치가 아니라 텍스트 매칭 — 헤더가 ='목표'!A2:A5 참조라 목표 탭과 같은 순서로 나오는 게 보장되지만, 매칭 로직 자체는 순서에 의존하지 않게 해 더 견고하게 만든다). 다음은 즉시 Error:

  • 헤더 칸 텍스트가 어떤 events[].key와도 안 맞음(알 수 없는 종목)
  • 같은 종목 키가 헤더에 중복
  • events[]의 어떤 키가 헤더 어디에도 없음(종목 컬럼 누락)

3-4. 셀 → EventScore 조립 (valueKind 교차검증 포함)

docs/records-schema.html §03은 normalizeScore의 ScoreValue.kind → 이 계약의 ScoreStatus 매핑과, "종목 형식(valueKind)과 셀에서 판별된 형식이 다르면 invalid로 승격"하는 교차검증을 "조립 단계(파서/엔드포인트 이슈)의 책임"이라고만 적어두고 어느 이슈인지 못박지 않았다. 이 파서가 이미 events[]를 받아 헤더를 매칭하므로 이 교차검증을 이 파서가 흡수한다 — 그러면 Session이 최종 EventScore를 담은 채로 나가고, 하류(#28~#30)가 같은 정규화 로직을 다시 구현하거나 재검증할 필요가 없다.

normalizeScore 결과EventScore
exempt{ status: 'exempt', value: null, display: null }
blank{ status: 'unmeasured', value: null, display: null }
invalid{ status: 'invalid', value: null, display: raw.trim(), reason }
count · event.valueKind==='count'{ status: 'recorded', value, display: raw.trim() }
seconds · event.valueKind==='time'{ status: 'recorded', value, display: raw.trim() }
count/seconds인데 event.valueKind와 불일치{ status: 'invalid', value: null, display: raw.trim(), reason: '종목 형식과 값 형식이 다름' }

row[columnIndex]가 undefined(Sheets API가 행 끝의 빈 셀을 생략해 배열이 짧아진 경우)면 normalizeScore(undefined) → blank → unmeasured로 자연스럽게 처리된다(별도 분기 불필요).

3-5. participated 파생 및 빈 행 처리

participated = Object.values(scores).some(s => s.status !== 'unmeasured') — 전 종목 unmeasured면만 false, exempt·invalid도 "뭔가 입력됨"으로 참여로 친다(records-schema.html §03과 일치).

데이터 행의 모든 칸(이름 포함)이 공백이면 식별할 이름 자체가 없으므로 엔트리를 만들지 않고 조용히 스킵한다(Sheets 범위 조회가 실제 데이터보다 한두 행 더 가져왔을 때 생기는 흔한 아티팩트로 취급) — 이름은 있는데 매칭이 안 되거나 점수만 있고 이름이 없는 부분적으로만 빈 행은 진짜 이상 상태이므로 Error로 처리한다. 스킵된 행이 있어도 이후 행의 playerId는 원본 배열의 절대 위치를 그대로 쓰므로(반복문 인덱스를 필터링 전 배열에서 얻음) 밀림이 생기지 않는다.

데이터 행 수가 players.length보다 많으면(존재하지 않는 명단 위치를 참조) Error. 반대로 적은 것은 정상이다 — 신규 가입자는 과거 회차에 엔트리 자체가 없다(이슈 본문 그대로).

3-6. 상태(탈퇴 등) 필터링은 여기서 하지 않는다

§2에서 분석한 대로, PlayerStatus별 포함 규칙은 소비자마다 다르다(활동만/비대상·휴식 포함/탈퇴 항상 제외). 이 파서는 players[]를 오직 이름 무결성 검증에만 쓰고 status는 들여다보지 않는다 — 필터링은 랭킹(#28)·추이(#29)·엔드포인트 조립(#30)의 책임으로 남긴다. PR 설명에 이 결정을 명시해 향후 리뷰에서 혼동을 막는다.

5리스크

리스크영향완화책
#25/#26(명단·목표 파서)이 아직 미구현실제 시트 출력과의 통합 테스트는 이 이슈 범위 밖shared/domain.ts 타입 계약만으로 픽스처 작성 — 순수 함수라 #25/#26이 나중에 그 타입을 만들어내기만 하면 그대로 연결됨
valueKind 교차검증 책임을 이 이슈가 흡수하기로 임의 결정#30(엔드포인트) 설계 시 중복 검증 또는 책임 혼동 가능성PR 설명에 "Session이 이미 최종 EventScore를 담아 나간다"고 명시, #30 플랜 작성 시 이 문서 링크
탈퇴 등 상태 필터링을 이 파서가 하지 않기로 결정후속 이슈(#28~#30)가 필터링을 빠뜨리면 탈퇴 회원이 응답에 노출도메인 주석·PR 설명에 "필터링은 하류 책임"임을 명시, 완료 조건에도 반영
Sheets API가 트레일링 빈 셀/행을 생략해 배열 길이가 들쭉날쭉할 수 있음(실제 동작은 #24 미구현으로 아직 확인 불가)인덱스 접근 시 undefined를 못 다루면 컬럼이 밀리거나 예외 발생짧은 행은 normalizeScore(undefined) → blank로 자연 처리, 완전 빈 행은 스킵 테스트로 커버 — #24 구현 후 실제 응답 형태로 재검증 필요하면 별도 이슈

6테스트 계획

  • npx vitest run functions/lib/parse-session.test.ts — 아래 케이스가 모두 통과해야 한다.
  • 정상: 개수형·시간형 혼합 회차에서 recorded value/display가 정확히 계산됨(예: 1:15 → 75초).
  • 면제: 면제 셀 → { status: 'exempt', value: null, display: null }, 그 종목은 랭킹 제외 대상이지만 participated는 true로 남음.
  • 미참여: 전 종목 빈칸인 행 → 모든 칸 unmeasured, participated: false. 한 종목만 빈칸인 행은 그 칸만 unmeasured고 participated는 true.
  • 이상값 승격: 개수 종목 셀에 1:15 입력 → invalid(reason에 형식 불일치 명시), normalizeScore 자체가 invalid로 판별하는 값(예: 1:75)도 그대로 invalid 전달.
  • 이름 무결성: 회차 탭 이름 셀이 players[N-1].name과 다르면 Error, NFD로 들어와도 NFC 정규화 후 일치하면 정상 통과.
  • 헤더 매칭: 헤더 열 순서를 events[] 순서와 다르게 섞어도 텍스트 매칭으로 정상 처리, 알 수 없는 헤더 텍스트·종목 컬럼 누락·중복 헤더는 각각 Error.
  • 행 수: rows 데이터 행 수가 players.length보다 많으면 Error, 적은 것(신규 가입자 미포함)은 정상적으로 그만큼만 entries에 담김.
  • 빈 행: 완전히 빈 행(이름·점수 전부 공백)은 스킵되고, 그 다음 실제 행의 playerId는 원래 위치를 유지함(밀리지 않음).
  • npm run typecheck — shared/domain.ts import, Record<string, EventScore> 조립이 strict 옵션 하에서 타입 에러 없이 통과.
