#63기능 추가

기록 입력 W2 — Sheets 쓰기 래퍼 (write scope · values.update RAW · 재시도)

damdam6/bernice 브랜치 feat/63-sheets-write-wrapper ← main 라벨 enhancement 2026-07-17

1요약

이슈 #24(읽기 래퍼) 이후 처음으로 시트에 값을 쓰는 경로를 연다. 기존 googleAuth.getAccessToken(env, scope)를 쓰기 scope(https://www.googleapis.com/auth/spreadsheets)로 재호출해 읽기와 별도로 캐시되는 토큰을 발급받고, 신규 모듈 functions/lib/sheetsWriteApi.ts에 단일 범위 values.update(valueInputOption=RAW) 래퍼 updateValues(env, sheetId, range, values)를 추가한다.

429·5xx 응답만 짧은 백오프로 최대 2회 자동 재시도하고(대상이 항상 행 전체 교체 update라 재시도가 멱등), 4xx는 재시도 없이 즉시 SheetsApiError(기존 sheetsApi.ts의 에러 타입을 그대로 재사용)로 실패시킨다. 기존 sheetsApi.ts는 한 줄도 수정하지 않는다 — 회차 탭에서 대상 행 찾기·normalize-score 값 검증·실제 API 엔드포인트(POST /api/admin/records)는 전부 후속 이슈(PRD §11의 W3)의 몫이다. 이 이슈는 독립적으로 병행 가능하다고 명시돼 있다(의존성 없음).

2현황 분석

  • functions/lib/googleAuth.ts:25 — getAccessToken(env, scope)가 이미 scope-agnostic하다. scope 문자열이 tokenCache(모듈 전역 Map<string, Promise<CachedToken>>)의 키라서, 읽기 scope와 다른 값을 넘기기만 하면 완전히 독립된 토큰이 별도 발급·캐시된다 — googleAuth.ts 변경이 전혀 필요 없다.
  • functions/lib/sheetsApi.ts:10-39 — 읽기 래퍼의 형태(SHEETS_BASE_URL 상수 · SheetsApiError · callSheetsApi 내부 헬퍼)가 이번 모듈이 그대로 따라야 할 참조 패턴이다. 다만 callSheetsApi는 READONLY_SCOPE·GET·재시도 없음에 고정된 비공개 함수라 그대로 재사용할 수 없다 — 게다가 sheetsApi.ts:11-12에 이미 "쓰기가 필요한 관리자 기능(P5)은 별도 scope로 자체 처리한다"는 주석이 있어, 별도 모듈로 가는 방향이 코드에 선반영돼 있다.
  • functions/lib/sheetsApi.ts:14-22 — SheetsApiError(message, status)는 읽기·쓰기 어느 쪽 실패든 같은 형태로 호출자에게 상태코드를 전달하면 되므로, 새 에러 타입을 만들지 않고 그대로 import해 재사용한다.
  • scripts/seed-sheet.mjs:86-93,130-135 — 과거 1회성 스크립트가 이미 PUT .../values/{range}?valueInputOption=RAW + body: { range, majorDimension: 'ROWS', values } 형태로 이 시트에 실제로 쓰기에 성공한 전례가 있다. node:crypto 기반이라 Workers에는 그대로 못 쓰지만, 요청 형태(URL·쿼리·바디 스키마)는 그대로 검증된 참고 자료다.
  • docs/prd-record-input.html §07(줄 297-321)이 이 이슈의 사실상 스펙이다 — 쓰기 파이프라인 6단계 중 이 이슈가 맡는 부분은 1(토큰)·4(values.update RAW)·5(429·5xx 재시도)뿐이고, 2(행 찾기)·3(값 검증)·6(캐시 무효화)은 §11 표에서 W3으로 명시적으로 분리돼 있다. "재시도는 서버 안에서, 429·5xx에만, update가 항상 멱등이라 안전"이라는 설계 근거도 이미 §07에 확정돼 있어 이 이슈에서 새로 정할 것은 없다.
  • 저장소 어디에도 재시도/백오프나 vi.useFakeTimers() 패턴이 아직 없다 — 이 이슈가 처음 도입한다. vitest@4.1.10(devDependencies)는 vi.useFakeTimers()/vi.runAllTimersAsync()를 지원하므로 새 의존성 없이 바로 쓸 수 있다.
  • functions/lib/sheetsApi.test.ts:11-88 — 테스트용 RSA 키페어 생성(beforeAll) + vi.stubGlobal('fetch', ...)로 oauth2 토큰 엔드포인트와 sheets.googleapis.com을 함께 스텁하는 패턴이 이미 있다. 새 테스트 파일이 이 패턴을 그대로 확장(PUT 스텁 + 실패 시퀀스 스텁 추가)하면 된다.
  • wrangler.jsonc — SHEET_ID는 이미 평문 vars로 설정돼 있고, 그 시트에 쓰기 권한을 가진 서비스계정 자격증명은 seed-sheet.mjs가 같은 scope(.../auth/spreadsheets)로 이미 성공적으로 썼던 바로 그 계정이라 권한 문제는 없다.

3설계

  1. 파일 위치: functions/lib/sheetsWriteApi.ts (+ sheetsWriteApi.test.ts) — sheetsApi.ts와 동일 계층의 새 형제 모듈. sheetsApi.ts는 전혀 수정하지 않는다.
  2. scope 상수: WRITE_SCOPE = 'https://www.googleapis.com/auth/spreadsheets'. import { getAccessToken, type Env } from './googleAuth'로 토큰 발급 재사용, export type { Env }도 sheetsApi.ts와 동일하게 재노출(다음 이슈가 이 모듈만 import해도 Env를 바로 쓸 수 있게).
  3. 에러 재사용: import { SheetsApiError } from './sheetsApi' — 새 에러 클래스를 만들지 않는다.
  4. 단일 진입점 updateValues(env: Env, sheetId: string, range: string, values: string[][]): Promise<void>. range는 호출자가 이미 조립한 A1 표기 전체 문자열(예: "'2025-08-16'!B7:E7") — 탭 이름 quoting은 기존 sheetsApi.ts의 quoteSheetName을 호출자(W3)가 그대로 재사용하면 되므로 이 모듈이 다시 구현하지 않는다(읽기 래퍼의 batchGetValues(env, sheetId, ranges: string[])와 같은 "range는 이미 완성된 문자열" 계약).
  5. 요청 형태 (scripts/seed-sheet.mjs 전례 그대로): PUT {SHEETS_BASE_URL}/{encodeURIComponent(sheetId)}/values/{encodeURIComponent(range)}?valueInputOption=RAW, 바디 { range, majorDimension: 'ROWS', values }. sheetId·range 둘 다 인코딩하는 이유는 읽기 래퍼가 이미 하는 것과 동일한 방어(특수문자가 경로를 벗어나지 못하게).
  6. 재시도 정책: RETRY_DELAYS_MS = [300, 900] 같은 명시적 배열 상수로 "최대 2회 재시도"를 표현(지수적으로 벌어지는 간격, 총 대기 1.2초 이내). 배열 길이가 곧 재시도 횟수라 테스트에서 정확한 시퀀스를 검증하기 쉽고, 나중에 정책을 바꿀 때도 상수 하나만 건드리면 된다 — 이슈 문구의 "1~2회"를 "최대 2회"로 확정해서 구현한다(정확히 1회로 좁히고 싶다면 배열을 [500]로 바꾸기만 하면 되는 구조).
    function isRetryableStatus(status: number): boolean { return status === 429 || status >= 500 } export async function updateValues(env: Env, sheetId: string, range: string, values: string[][]): Promise<void> { const accessToken = await getAccessToken(env, WRITE_SCOPE) const url = `${SHEETS_BASE_URL}/${encodeURIComponent(sheetId)}/values/${encodeURIComponent(range)}?valueInputOption=RAW` const body = JSON.stringify({ range, majorDimension: 'ROWS', values }) for (let attempt = 0; ; attempt++) { const response = await fetch(url, { method: 'PUT', headers: { Authorization: `Bearer ${accessToken}`, 'Content-Type': 'application/json' }, body, }) if (response.ok) return const error = new SheetsApiError( `Sheets API 쓰기 실패 (${response.status}): ${await response.text()}`, response.status, ) if (!isRetryableStatus(response.status) || attempt >= RETRY_DELAYS_MS.length) throw error await sleep(RETRY_DELAYS_MS[attempt]) } }
    토큰은 루프 진입 전 1회만 발급(재시도 구간이 최대 ~1.2초라 만료 걱정 없음 — 읽기 래퍼의 callSheetsApi와 동일한 전제).
  7. 지연 구현: sleep(ms) = new Promise(resolve => setTimeout(resolve, ms)). 테스트에서 vi.useFakeTimers() + vi.runAllTimersAsync()로 실제 대기 없이 재시도 시퀀스를 즉시 진행시킨다 — 프로덕션 코드에 테스트 전용 훅(주입 가능한 delay 파라미터 등)을 추가하지 않는다.
  8. 기각한 대안 1: sheetsApi.ts의 callSheetsApi를 확장해 메서드·재시도까지 처리 — 기각. 이슈가 "기존 sheetsApi.ts는 읽기 전용 유지"를 명시하고, 재시도는 읽기 경로엔 없는 책임이라 섞으면 읽기 헬퍼가 불필요하게 복잡해진다.
  9. 기각한 대안 2: 클라이언트(브라우저) 쪽 재전송 — 기각. PRD §07이 "재시도는 서버(Pages Function) 안에서"로 이미 확정한 결정이라 이 이슈에서 재론하지 않는다.
  10. 기각한 대안 3: fetch 자체가 던지는 네트워크 예외(타임아웃 등)까지 잡아서 재시도 — 기각. 이슈 문구가 "429·5xx 한정"으로 명시하고, 읽기 래퍼도 이런 예외를 별도 처리하지 않는 동일 선례가 있어 일관성을 위해 그대로 전파시킨다.

4작업 단계

  1. functions/lib/sheetsWriteApi.ts 신규 생성 — SHEETS_BASE_URL·WRITE_SCOPE·RETRY_DELAYS_MS 상수, SheetsApiError import, isRetryableStatus·sleep 내부 헬퍼 작성. functions/lib/sheetsWriteApi.ts
  2. updateValues(env, sheetId, range, values) 구현 — §3의 pseudo-code대로 토큰 발급 → PUT 요청 → 성공 시 반환, 실패 시 재시도 가능 여부 판단 → 백오프 후 루프 또는 SheetsApiError throw. functions/lib/sheetsWriteApi.ts
  3. 단위 테스트 작성 — sheetsApi.test.ts의 스텁 패턴(RSA 키페어 + vi.stubGlobal('fetch', ...))을 확장해 oauth2 토큰과 PUT .../values/... 엔드포인트를 함께 스텁하고, 실패를 순서대로 흘려주는 응답 큐를 추가. vi.useFakeTimers() + vi.runAllTimersAsync()로 재시도 대기를 즉시 진행. 커버 케이스:
    • 정상 응답 → 1회 호출로 성공, 요청이 PUT·valueInputOption=RAW 쿼리·{range, majorDimension: 'ROWS', values} 바디로 나감을 검증
    • 429 1회 → 성공: 재시도 후 정상 완료
    • 500 2회 → 성공: 재시도 상한(2회)까지 써도 복구되면 성공
    • 500 3회 연속: 재시도 소진 후 SheetsApiError(status 500)로 실패, fetch가 정확히 3회(초기 1 + 재시도 2) 호출됐는지 카운트 검증
    • 400 같은 4xx: 재시도 없이 즉시 SheetsApiError, fetch가 정확히 1회만 호출됨
    • 쓰기 scope 토큰이 읽기 scope와 별도로 발급됨(oauth2 요청의 assertion JWT를 디코드해 scope 클레임이 WRITE_SCOPE인지 확인)
    • sheetId·range에 경로 구분자 등 특수문자가 있어도 encodeURIComponent로 안전하게 인코딩됨(읽기 래퍼의 동일 테스트 패턴)
    구현 노트: 계획은 7개 케이스를 별도 it()로 나열했지만, "정상 응답" 케이스와 "쓰기 scope 검증" 케이스를 하나로 합쳐 6개 it()로 구현했다 — scope는 고정 상수라 파일 내 모든 테스트가 googleAuth의 모듈 전역 tokenCache를 공유하고, 첫 테스트 이후로는 캐시 히트라 토큰이 재발급되지 않는다. JWT scope 클레임을 안정적으로 검증하려면 캐시가 비어 있는 파일의 "첫 테스트"에서만 가능해, 별도 테스트로 분리하면 실행 순서에 암묵적으로 의존하게 된다. 커버리지는 7개 항목 그대로 유지된다. 또한 fake timers 사용 시 expect(promise).rejects...를 vi.runAllTimersAsync() 이전에 먼저 attach해야 unhandled rejection 경고가 나지 않는다는 점을 재시도-소진 테스트에서 확인·반영했다. functions/lib/sheetsWriteApi.test.ts
  4. npm run typecheck · npm test 전체 통과 확인. 실제 시트에 대한 라이브 쓰기 스모크 테스트는 이 이슈에서 하지 않는다(§5 리스크 참고) — 유닛 테스트(fetch 모킹)만으로 동작을 검증한다. 결과: npm run typecheck 통과(에러 0). npm test 전체 20개 파일 · 255개 테스트 전부 통과(신규 6개 포함), 회귀 없음. (검증만)

5리스크

리스크영향완화책
실제 시트에 대한 라이브 쓰기 확인은 팀 실데이터(회차 점수)를 덮어쓸 수 있어 파괴적잘못 실행하면 실제 기록이 훼손될 수 있음이 이슈는 유닛 테스트(fetch 모킹)로만 검증하고 라이브 쓰기 확인을 하지 않는다. 실제 API 왕복 확인은 행 찾기·검증까지 갖춰진 W3(POST /api/admin/records) 통합 시점, 필요하면 그때 디스포저블 시트 사본으로
백오프 파라미터([300, 900]ms, 최대 2회)는 이슈 문구 "1~2회"에 대한 이 계획의 해석 — PRD가 정확한 숫자를 못박지 않음실제 운영에서 기대와 다른 재시도 횟수/타이밍일 수 있음상수 하나(RETRY_DELAYS_MS)로 정의해 리뷰에서 바로 조정 가능하게 설계. 계획 승인 시 이 숫자에 이견이 있으면 지금 조정
fetch 자체의 네트워크 예외(타임아웃 등)는 429/5xx 재시도 대상 밖일시적 네트워크 문제도 재시도 없이 즉시 실패로 전파이슈 문구가 "429·5xx 한정"으로 명시하고 읽기 래퍼도 동일 — 의도된 범위. 필요해지면 별도 이슈
재시도가 "행 전체 교체라 멱등"이라는 PRD 전제에 의존호출자가 부분 업데이트 용도로 이 함수를 오용하면 재시도가 더 이상 안전하지 않음이 모듈 자체는 range/values를 그대로 전달만 할 뿐이고, "행 전체 교체로만 쓸 것"은 다음 이슈(W3)가 지켜야 할 호출 계약 — 함수 설계 의도를 §3 주석과 이 문서에 명시

7완료 조건

  • RAW 쓰기 계약 — valueInputOption=RAW가 모든 요청에 고정돼 1:15 같은 문자열이 시각으로 자동 변환되지 않음(테스트로 쿼리 파라미터 검증).

  • 테스트(vitest) — functions/lib/sheetsWriteApi.test.ts가 §4의 커버 케이스를 모두 포함해 통과.
  • functions/lib/sheetsApi.ts는 한 글자도 수정되지 않음 — 읽기 전용 유지.
