P1 — 종목별 랭킹 계산 (방향·면제/빈칸 제외·상태 정책)

damdam6/bernice 브랜치 feat/28-event-ranking-calc ← main 라벨 없음 2026-07-13

1요약

회차·종목별 랭킹을 계산하는 순수 함수를 functions/lib/에 추가한다. 입력·출력 타입(Player·EventDefinition·Session → EventRanking·SessionRankings)과 동점·달성·상태 정책 규칙은 이미 shared/domain.ts와 docs/records-schema.html §02·§04에 확정되어 있다(이슈 #21, PR #16 리뷰 승인). 이슈 본문의 의존성 #3은 그 도메인 타입 이슈를 가리키며, 실제로는 #21로 머지되어 이미 main의 shared/domain.ts에 들어와 있다 — 재구현하지 않고 그대로 import해 쓴다. 따라서 이번 작업은 규칙을 새로 정의하는 게 아니라 이미 문서화된 계약을 구현하고 픽스처로 검증하는 것이다.

2현황 분석

  • 출력 타입은 이미 정의돼 있다 — shared/domain.ts:70-89의 RankingEntry·EventRanking·SessionRankings. 아직 어떤 파일도 이 타입을 만들어내지 않는다(구현 없음, 신규).
  • 입력 타입도 이미 정의돼 있다 — Player(shared/domain.ts:27-31, status는 4종 전체를 그대로 갖는 좁혀지지 않은 타입), EventDefinition(shared/domain.ts:36-43, direction·targetValue 포함), SessionEntry/EventScore(shared/domain.ts:48-60, scores는 종목 key 전체가 항상 존재).
  • 동점·달성·상태 정책 규칙은 코드가 아니라 이미 문서에 확정돼 있다 — docs/records-schema.html:151-171(§04: 표준 공동순위 1,1,3 · 경계값 포함 달성 · 0/0 → rate 0)과 docs/records-schema.html:96-103(§02: 탈퇴=전체 제외, 비대상·휴식=랭킹만 제외). 이 이슈가 처음부터 규칙을 정할 필요가 없다는 뜻.
  • docs/records-schema.html:388-441에 rankings 예시 JSON이 선수1(활동)·선수3(활동, invalid 포함)·선수5(휴식)·선수7(활동, 미참여)까지 상태 혼합 케이스로 이미 나와 있어, 그대로 회귀 테스트 픽스처로 재사용 가능.
  • 기존 functions/lib/의 계산 유틸(normalize-score.ts, normalize-status.ts, sheetTabs.ts)은 모두 "순수 함수 + 동일 디렉터리 *.test.ts" 형태이고 파서·네트워크에 의존하지 않는다 — 이 이슈도 같은 모양을 따른다.
  • functions/tsconfig.json:1-19의 include에 이미 "../shared"가 들어 있어 functions/lib/에서 shared/domain.ts를 바로 import할 수 있다. 다만 grep -rn "from '.*shared" functions/ src/ 결과가 0건 — 아직 아무도 이 타입을 소비하지 않으므로 이 이슈가 첫 소비처가 된다.

3설계

  • 파일: functions/lib/compute-rankings.ts + functions/lib/compute-rankings.test.ts(기존 normalize-*.ts와 동일한 kebab-case 네이밍).
  • 함수 시그니처:
    • computeEventRanking(event: EventDefinition, entries: SessionEntry[], players: Player[]): EventRanking — 종목 1개에 대한 랭킹.
    • computeSessionRankings(session: Session, events: EventDefinition[], players: Player[]): SessionRankings — events.map(e => computeEventRanking(e, session.entries, players))로 조립.
    여러 회차를 한 번에 도는 함수는 만들지 않는다 — SessionRankings[] 배열 조립은 /api/records 엔드포인트(별도 이슈)의 책임이고, 이 이슈는 "회차 1개 × 종목별" 단위 순수 계산만 맡는 게 범위에 맞다.
  • 포함 조건(한 줄로 상태 4종 + 면제/빈칸/이상값을 동시에 처리): player.status === '활동' && entry.scores[event.key].status === 'recorded'. 첫 조건 하나로 탈퇴·비대상·휴식이 모두 배제되고(§02 표와 동일 결과), 두 번째 조건으로 면제·미측정·이상값이 배제된다(§04와 동일 결과). 파서가 이미 탈퇴를 걸러줬을 거라고 가정하지 않는다 — 이 함수는 픽스처로 독립 테스트되므로 방어적으로 직접 재확인한다.
  • 빈 결과 처리: 종목에 유효 엔트리가 0건이어도 EventRanking 자체는 생략하지 않고 entries: []로 포함한다(모든 events[]에 대해 1개씩) — EventAchievementRate의 "0/0 → rate 0" 패턴(§04)과 동일한 결로, 프론트의 "회차 선택 → 종목 탭" 흐름에서 종목 탭 자체가 사라지지 않게 한다.
  • 정렬·동점 처리: 정규화된 value 기준으로 direction에 따라 정렬(낮을수록=오름차순, 높을수록=내림차순) 후, 표준 공동순위(1,1,3)를 값 비교로 직접 구현 — 이전 엔트리와 값이 같으면 이전 rank를 재사용, 다르면 index+1. display 문자열은 비교에 쓰지 않는다(§04 명시 규칙). 이 규칙은 이미 PR #16에서 승인된 것이라 이 이슈에서 새로 결정하지 않고 그대로 구현·테스트만 한다.
  • 달성(achieved) 판정: 경계값 포함 — 낮을수록는 value <= event.targetValue, 높을수록는 value >= event.targetValue(§04와 동일, RankingEntry.achieved).
  • 기각한 대안: (a) PlayerStatus를 타입 레벨에서 좁혀 받는 방식 — Player는 이미 4종 그대로인 타입이라 실질적 방어가 안 되고 런타임 필터가 필요해 의미 없음. (b) 이벤트당 엔트리 0건이면 EventRanking 자체를 배열에서 생략 — 프론트가 "이 회차엔 이 종목 데이터가 없음" 상태를 표현할 근거가 없어져 기각.
