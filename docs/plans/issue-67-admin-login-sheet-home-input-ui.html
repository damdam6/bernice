1요약

관리자 로그인부터 참가자 목록까지, 기록 입력 플로우의 앞부분 4화면을 채운다. 화면 라우트와 도메인 타입(RecordsResponse)은 이미 있고, 백엔드 API(#64 #65 #66)도 이미 구현·배포돼 있다 — 이번 이슈는 순수 프론트엔드 UI 작업이다. 6개 페이지 컴포넌트(현재 전부 "#67에서 채워집니다" placeholder)를 디자인 PRD §05 스펙대로 구현하고, 관리자 스택을 보호하는 게이트와 로그아웃 기능(신규 POST /api/logout)을 추가한다. GET /api/records 응답 하나로 날짜 선택·참가자 목록·기록지 만들기·참가자 추가 4화면의 데이터를 전부 충당할 수 있어 신규 조회 API는 없다.

2현황 분석

라우팅 · 현재 placeholder

src/App.tsx:26-34에 관리자 스택 라우트 7개가 이미 등록돼 있다 — /admin/login·/admin·/admin/records·/admin/records/:sessionDate·/admin/records/:sessionDate/:playerId·/admin/create-sheet·/admin/add-players. 이 중 /admin/records/:sessionDate/:playerId(RecordsPlayerInput.tsx)는 #68 몫 — 본문에 "#68에서 채워집니다"라고 명시돼 있어 이번 이슈 범위에서 제외한다. 나머지 6개 페이지(AdminLogin·SheetManagementHome·RecordsDateSelect·RecordsParticipants·CreateSheet·AddPlayers, 전부 src/pages/admin/*.tsx)는 10~16줄짜리 카드 placeholder뿐이다.

src/components/layout/AdminLayout.tsx:4-15는 관리자 스택 공통 레이아웃(뒤로가기 헤더 + Outlet)이며, 인증 검사가 전혀 없다 — 로그아웃 상태에서 URL을 직접 쳐도 그대로 렌더된다. 열람 게이트(src/components/auth/LoginGate.tsx, #47)가 메인 3탭을 useRecords()의 401로 게이팅하는 것과 대칭되는 장치가 관리자 스택엔 없다.

백엔드 — 이미 구현된 부분

functions/api/login.ts(#41)는 ADMIN_CODE·TEAM_PASSCODE 양쪽을 검사해 {ok:true, role:'admin'|'team'} + Set-Cookie(HttpOnly)를 내려준다. 관리자 코드가 우선이라 두 코드가 같아도 admin이 이긴다(functions/api/login.ts:60-69). functions/api/_middleware.ts(#42)는 /api/admin/*에 admin role을 강제(403)하고, 그 외 /api/*는 유효한 세션만 요구(401) — GET /api/records는 team 세션으로도 통과한다(isAdminPath가 /api/admin 접두만 좁혀 판정, _middleware.ts:48-50).

쓰기 API 3종은 이미 완성·병합돼 있다: POST /api/admin/records(#64, 선수 단위 점수 upsert — #68에서 소비), POST /api/admin/create-sheet(#65, {participantIds:number[]} → 201 + 회차 탭 생성, 오늘 이미 있으면 409), POST /api/admin/add-players(#66, {sessionDate, playerIds:number[]} → 200, 이미 참가자면 409). 세 endpoint 모두 성공 시 RECORDS_CACHE_KEY(Cache API)를 무효화하므로, 프론트는 성공 후 TanStack Query 캐시(RECORDS_QUERY_KEY, 별개 캐시)만 직접 무효화하면 된다.

로그아웃 API는 없다. functions/lib/session-cookie.ts:145-149의 SessionCookieOptions.maxAgeSeconds 주석이 "0이면 즉시 만료 — 세션 삭제(로그아웃)용"이라고 이미 이 용도를 예견해뒀지만, 이를 쓰는 POST /api/logout 핸들러 자체는 아직 없다(functions/api/에 login.ts만 있고 logout.ts 없음, _middleware.ts:23의 PUBLIC_PATHS에도 없음). PRD(docs/prd-design.html:164,214)는 시트 관리 홈에 "로그아웃 텍스트 버튼"을 명시하므로 이 화면이 동작하려면 신규 endpoint가 필요하다.

도메인 타입 — 신규 화면이 필요로 하는 필드는 이미 계약에 있다

shared/domain.ts의 RecordsResponse(140-147행)가 4화면의 데이터를 전부 충당한다: sessions[](52-67행, 날짜 오름차순 · entries[].scores로 입력 상태 파생) + players[](활동/비대상/휴식만, 탈퇴 제외)로 날짜 선택·참가자 목록·기록지 만들기·참가자 추가 4화면 모두를 신규 조회 API 없이 만들 수 있다. "n차" 라벨은 계약 필드가 아니라 프론트 파생값 — docs/prd-design.html:292가 규칙을 명시: "sessions 날짜 오름차순 인덱스+1".

가나다 정렬 비교자는 새로 만들 필요 없이 기존 백엔드 구현을 그대로 옮겨 쓴다: functions/lib/create-sheet.ts:72 — (a,b) => (a.name < b.name ? -1 : a.name > b.name ? 1 : a.id - b.id). 다만 functions/lib는 tsconfig.app.json의 include(src, shared)에 없어 프론트에서 직접 import할 수 없다(functions/tsconfig.json이 별도 프로젝트) — 로직만 프론트 유틸로 복제한다.

같은 경계 문제가 functions/lib/seoul-date.ts(오늘 KST 날짜 계산, UTC+9 고정 오프셋 방식)에도 적용된다 — 기록지 만들기 화면의 "오늘 · YYYY-MM-DD" 헤더 표시용으로 프론트에 동일 로직을 복제해야 한다.

3설계

관리자 스택 라우트 구조 — 로그인 화면과 데이터 화면을 분리 게이팅

선택: App.tsx의 관리자 라우트를 /admin/login(게이트 없음)과 나머지 5개(신규 AdminGate로 감쌈)로 나눈다. AdminGate는 LoginGate와 동일하게 useRecords()의 401을 신호로 써서 <Navigate to="/admin/login" replace />로 돌려보낸다.

기각안: 각 데이터 페이지가 개별적으로 401을 처리. → 5곳에 동일 리다이렉트 로직이 중복되고, 화면 진입 시 "잠깐 빈 화면 → 리다이렉트"가 페이지마다 따로 깜빡여 UX가 나빠진다. AdminGate 한 곳에서 처리하면 LoginGate와 대칭적인 구조가 되고 App.tsx diff도 한 줄짜리 중첩 라우트 추가로 끝난다.

알려진 한계(비스코프로 남김): team 세션으로 /admin URL에 직접 진입하면 GET /api/records가 200을 반환하므로(팀 세션도 통과) AdminGate를 그대로 통과해 시트 관리 화면이 보인다 — 실제 쓰기 시도는 /api/admin/* 미들웨어가 403으로 막는다. 프론트가 role을 알 방법이 없다(GET /api/records 응답엔 role 필드가 없음, #67 의존성에 "whoami" 성격 엔드포인트가 없음). 관리자 로그인 성공 응답의 role은 로그인 그 순간에만 알 수 있으므로 로그인 화면 자체의 오분기 방지에만 쓴다(아래).

관리자 로그인 — role 불일치 처리

POST /api/login은 admin/team 코드를 구분 없이 같은 200으로 받아들이고 role만 다르게 준다(오라클 방지가 목적, functions/api/login.ts:1-7 주석). 관리자 로그인 화면에서 팀 패스코드를 넣으면 서버는 200 + role:'team'을 준다 — 이 경우도 화면은 "관리자 코드가 올바르지 않습니다"로 실패 취급하고 시트 관리로 넘어가지 않는다(코드 자체는 유효해도 관리자 권한은 아니므로). src/lib/login-api.ts의 LoginResult에 role을 추가해 호출부가 분기할 수 있게 한다 — 기존 소비자(LoginGate)는 role을 안 보므로 하위 호환.

로그아웃 — 신규 POST /api/logout

세션 쿠키가 HttpOnly라 프론트 JS로 지울 수 없다 — 서버가 Max-Age=0 Set-Cookie를 내려줘야 한다. buildSessionSetCookie(token, {maxAgeSeconds:0})가 이미 이 용도로 준비돼 있으므로(session-cookie.ts:145-149), 토큰 값은 무엇이든(빈 문자열도) 상관없다 — 브라우저는 Max-Age=0을 받으면 즉시 쿠키를 삭제한다. 인증 여부와 무관하게 항상 200을 반환하는 멱등 엔드포인트로 만든다(로그아웃은 실패할 이유가 없다 — 이미 로그아웃 상태에서 다시 호출해도 200). _middleware.ts의 PUBLIC_PATHS에 /api/logout을 추가할 필요는 없다 — 로그아웃은 세션이 있는 상태에서만 의미 있는 동작이므로 기존 인증 게이트(401)를 그대로 통과 조건으로 둔다. /api/admin/logout이 아니라 /api/logout인 이유: team 세션도 언젠가 로그아웃이 필요할 수 있고(현재 UI엔 없지만), admin 전용으로 좁힐 이유가 없다.

입력 상태 파생 규칙 (참가자 목록 뱃지 · 날짜 선택 완료 n/N)

docs/prd-design.html:225 · docs/prd-record-input.html:230,237 계약을 그대로 코드화: entry 하나의 unmeasuredCount = events 중 entry.scores[event.key].status === 'unmeasured'인 개수. unmeasuredCount === events.length → 미입력, === 0 → 완료, 그 외 → 일부. 날짜 선택의 "완료 n/N"은 그 회차 entries 중 완료(unmeasuredCount===0)인 개수 / entries.length. SessionEntry.scores는 "events[] 전체 key가 항상 존재(누락 없음)"가 타입 계약(shared/domain.ts:57)이므로 null 체크 없이 순회 가능.

참가자 추가 진입점 — 회차 사전 선택

참가자 목록(/admin/records/:sessionDate)의 [참가자 추가] 버튼은 navigate('/admin/add-players', {state:{sessionDate}})로 현재 회차를 넘긴다. AddPlayers는 location.state?.sessionDate가 있으면 그 회차 칩을 초기 선택, 없으면(시트 관리 홈에서 바로 진입) 최신 회차를 기본값으로 쓴다 — router state는 새로고침 시 사라지므로 실패해도 "최신 회차 기본 선택"으로 자연히 폴백된다(별도 에러 처리 불필요).

토스트 — 공용 훅

src/hooks/useToast.ts 신규: {message, show(text)}를 반환하는 작은 훅. 내부에서 setTimeout 1.9초로 자동 소멸, show() 재호출 시 타이머 리셋. src/components/common/Toast.tsx는 프레젠테이션만 — message가 있을 때만 하단 중앙 다크 필 렌더. 기록지 만들기·참가자 추가 두 화면이 공유한다.

리스크영향완화책
team 세션으로 /admin 직접 진입 시 AdminGate를 통과(§3 설계에서 확인한 알려진 한계)낮음 — 팀 멤버가 시트 관리 화면을 "볼" 수는 있으나 실제 쓰기(create-sheet/add-players/records)는 서버 미들웨어가 403으로 차단비스코프로 명시 처리. 화면 하단에서 403 응답 시 일반 에러 메시지로 표시(별도 처리 불필요) — 서버가 최종 방어선(시트가 SoT 철학과 일치)
신규 POST /api/logout이 #67 의존성 목록(#64/#65/#66/#61/#62)에 없던 범위 추가중간 — PRD가 로그아웃 버튼을 명시하지만 이를 동작시킬 API가 없어 새로 만들어야 함기존 session-cookie.ts의 maxAgeSeconds:0이 이미 이 용도로 설계돼 있어 구현이 5줄 내외로 작음 · login.ts 테스트 패턴을 그대로 재사용해 리스크 낮춤
토스트·가나다 정렬·완료 상태 파생 로직을 여러 화면이 공유 — 유틸 경계를 잘못 나누면 화면마다 규칙이 미묘하게 어긋날 수 있음중간 — 특히 "완료 n/N"과 참가자 뱃지가 같은 원본 규칙에서 나와야 함entry-status.ts 단일 모듈에서 두 파생값을 함께 export해 소스가 하나임을 강제(작업 단계 3)
기록지 만들기·참가자 추가 성공 후 TanStack Query 캐시(RECORDS_QUERY_KEY)를 무효화하지 않으면, 참가자 목록으로 복귀해도 옛 데이터가 보임중간 — 서버 Cache API는 각 endpoint가 이미 지우지만, 프론트 쿼리 캐시는 별개두 화면 모두 성공 후 queryClient.invalidateQueries(RECORDS_QUERY_KEY, exact) 호출을 작업 단계에 명시(11·12단계)
