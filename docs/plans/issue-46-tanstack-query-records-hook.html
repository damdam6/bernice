P3 — TanStack Query 셋업 + useRecords 훅 (401 구분)

damdam6/bernice 브랜치 feat/46-tanstack-query-records-hook ← main 라벨 enhancement 2026-07-17

1요약

@tanstack/react-query를 설치해 src/main.tsx에 QueryClientProvider를 장착하고, GET /api/records를 호출해 shared/domain.ts의 RecordsResponse로 타입이 적용된 데이터를 반환하는 useRecords 훅을 새로 만든다. 핵심은 401을 일반 에러와 타입 수준에서 구분하는 것 — 이슈 #42가 확정한 401 {"error":"unauthorized"} 계약을 UnauthorizedError라는 전용 에러 클래스로 매핑하고, staleTime·retry 정책에서 401은 재시도 대상에서 제외한다. #42는 아직 머지되지 않아 로컬에서 실제 401을 재현할 수는 없으므로, 이번 이슈는 fetch를 모킹한 단위·통합 테스트로 계약 준수를 검증하고 실제 미들웨어와의 통합 확인은 #42 머지 이후로 남긴다. 이 훅은 후속 P2-5(로그인 게이트)와 P4의 모든 화면이 공통으로 소비할 진입점이라, 재사용 가능한 에러 타입 설계가 이번 이슈의 실질적 무게중심이다.

2현황 분석

  • src/App.tsx:20-23가 지금 유일한 데이터 페치 코드 — /api/health를 순수 fetch + useEffect로 호출하는 P0 스캐폴딩 수준. src/main.tsx:1-11에는 QueryClientProvider는커녕 어떤 프로바이더도 없다. package.json의 dependencies에는 react/react-dom뿐, @tanstack/react-query가 아직 설치돼 있지 않다.
  • shared/domain.ts:139-147의 RecordsResponse가 최종 계약(이슈 #21이 확정, §07)이고 이미 functions/api/records.ts가 그대로 구현·테스트돼 있어 프론트가 참조할 타입은 안정적이다.
  • #42(인증 미들웨어)는 아직 머지 전이다 — gh issue view 42 기준 state: OPEN. 레포 전체(메인 체크아�트 + 이 이슈를 포함한 모든 .worktrees/*)를 뒤져도 functions/_middleware.ts는 존재하지 않는다. 즉 오늘 시점의 /api/records(functions/api/records.ts:1-33)는 인증 게이트가 전혀 없고("인증 게이트는 P2 범위 — 여기서는 적용하지 않는다", functions/api/records.ts:2-3 주석), 성공 시 200, 실패 시 502(sheets_api_error)/500(missing_roster_tab 등)만 반환한다 — 401은 오늘 이 엔드포인트에서 물리적으로 발생하지 않는다. 401 응답 바디 계약({"error":"unauthorized"})은 이슈 #42 본문에만 명시돼 있고, 예외 경로는 /api/login·/api/health다.
  • 테스트 인프라: vitest.config.ts:1-6는 environment: 'node' 전역 설정 하나뿐이고, @testing-library/react·jsdom 둘 다 package.json에 없다 — src/ 쪽 테스트가 이 레포에 하나도 없다는 뜻(find src -name "*.test.*" 결과 없음). 가장 가까운 컨벤션은 functions/lib/sheetsApi.test.ts:1-20·functions/api/records.test.ts:1-15의 vi.stubGlobal(전역 모킹) + afterEach(() => vi.unstubAllGlobals()) 패턴이다.
  • tsconfig.app.json:22의 include: ["src", "shared"]가 새로 만들 src/hooks/·src/lib/ 파일과 그 테스트 파일을 자동으로 npm run typecheck(tsc -b) 범위에 포함시킨다 — 별도 tsconfig 수정 불필요.
  • 다른 워크트리(44-routing-skeleton, 48-card-badge-status-pill, 49~51 등, P3/P4의 나머지 작업들)를 확인한 결과 전부 main 기준 커밋(6e66b2e)에 머물러 있고 src/에 손댄 게 없다 — src/hooks/·src/lib/ 디렉터리 컨벤션은 이 이슈가 처음 확립한다.
  • 깨끗한 워크트리에서 npm install → npm run typecheck 클린, npx vitest run → 13개 파일 · 194개 테스트 전부 그린 — 회귀 기준선.

3설계

  • src/lib/api-error.ts — ApiError extends Error { status: number }와 그 서브클래스 UnauthorizedError(status 고정 401, 생성자 인자 없음)를 export. instanceof UnauthorizedError로 401을 다른 실패(502/500/네트워크 오류)와 타입 수준에서 구분할 수 있게 한다. 이슈 요구사항이 "401을 별도 에러 타입으로 구분"이라 명시했고, 소비 측(P2-5 로그인 게이트)이 error.status === 401 매직넘버 비교 대신 타입 체크로 분기할 수 있어야 하기 때문. records 전용이 아니라 src/lib/에 둬 향후 다른 훅(예: /api/refresh)도 재사용하게 한다.
  • src/lib/query-client.ts — export const queryClient = new QueryClient() 싱글턴만 export. 전역 defaultOptions는 건드리지 않는다 — staleTime/retry는 아래처럼 useRecords 훅 안에 지역화한다.
  • src/hooks/useRecords.ts — 세 개를 export:
    • fetchRecords(): fetch('/api/records') 후 res.status === 401이면 UnauthorizedError를 던지고, !res.ok면 바디를 {error, message} 형태로 파싱 시도해(functions/api/records.ts의 에러 바디 계약과 동일) ApiError(message, res.status)를 던진다. 성공이면 res.json()을 RecordsResponse로 반환.
    • shouldRetryRecordsQuery(failureCount, error): 순수 함수로 분리해 export — UnauthorizedError면 항상 false, 그 외엔 failureCount < 2(TanStack 기본 재시도 횟수 3회와 동일한 상한 유지). 실제 지수 백오프 타이머를 기다리지 않고도 재시도 판단 로직만 단위 테스트할 수 있어 테스트가 빠르고 결정적이다.
    • useRecords(): useQuery<RecordsResponse, ApiError>({ queryKey: ['records'], queryFn: fetchRecords, staleTime: 5 * 60 * 1000, retry: shouldRetryRecordsQuery }). 제네릭에 ApiError를 명시해 소비 측 error가 ApiError | null로 타입되고 instanceof UnauthorizedError 좁히기가 그대로 작동한다.
  • src/main.tsx — <App />를 <QueryClientProvider client={queryClient}>로 감싼다.
  • 테스트 인프라 — @testing-library/react·jsdom을 devDependencies에 추가. 전역 vitest.config.ts의 environment: 'node'는 그대로 두고, 새 테스트 파일(src/hooks/useRecords.test.tsx) 맨 위에 // @vitest-environment jsdom 지시자로 그 파일에만 jsdom을 적용한다.

기각한 대안

  • 전역 QueryClient defaultOptions에 staleTime/retry를 몰아넣기 — 기각: 이슈 범위가 "useRecords 훅"의 정책으로 명시돼 있고, 후속 엔드포인트(예: P5의 /api/refresh, /api/admin/*)가 다른 재시도 정책을 원할 수 있어 훅별로 지역화하는 편이 안전하다.
  • 전역 vitest.config.ts의 environment를 jsdom으로 통째로 바꾸기 — 기각: functions/lib/sheetsApi.test.ts 등 13개 기존 테스트 파일이 Workers 런타임 전역(crypto.subtle, caches 스텁 등)을 node 환경 기준으로 짜여 있어 전역 교체는 회귀 위험이 크다. per-file 지시자로 국소화하면 위험이 새 파일 하나로 좁혀진다.
  • 401 구분을 별도 클래스 대신 { status: number } 필드만 있는 단일 에러 타입으로 처리 — 기각: 이슈가 "별도 에러 타입"을 명시했고, 소비 측이 instanceof로 분기하는 편이 매직넘버 비교보다 오탈자에 안전하다.
  • zod 등으로 /api/records 응답에 런타임 스키마 검증 추가 — 기각: 서버가 shared/domain.ts 계약을 이미 강제(#21)하고 functions/api/records.test.ts가 그 계약을 서버 쪽에서 검증 중이다. 같은 계약을 클라이언트에서 다시 검증하는 건 이 이슈 범위를 넘는 과설계다.

5리스크

리스크영향완화책
#42(인증 미들웨어)가 아직 머지되지 않아 실제 401을 curl/브라우저로 재현 불가401 분기 코드가 실전에서 한 번도 hit되지 않은 채 머지될 수 있음fetch를 모킹해 401 응답을 흉내낸 단위·통합 테스트로 계약 준수를 검증. #42 머지 후 wrangler pages dev로 실제 재현 확인은 별도 후속 확인 사항으로 남김
jsdom을 새로 도입하며 vitest 전역 환경(node)과 충돌 가능기존 13개 functions/* 테스트 파일의 Workers 전역 스텁(caches, crypto.subtle)이 깨질 위험전역 environment는 그대로 node 유지, 새 테스트 파일에만 // @vitest-environment jsdom 지시자로 국소 적용 후 전체 스위트 재실행으로 확인
staleTime(5분)·재시도 상한(2회) 수치가 다소 임의적너무 짧으면 화면 전환마다 리페치 폭주, 너무 길면 관리자가 시트를 갱신해도 팀원 화면엔 오래된 데이터 표시서버 캐시 TTL(RECORDS_CACHE_TTL_SECONDS = 6시간, functions/lib/records-cache.ts)보다 훨씬 짧게 보수적으로 시작 — 실사용 피드백을 보고 조정할 여지를 남김. 재시도 상한은 TanStack 기본값(3회 시도)과 동일하게 맞춤

6테스트 계획

  • fetchRecords: 200 응답 → RecordsResponse 그대로 반환하는지 확인
  • fetchRecords: 401 응답 → UnauthorizedError 인스턴스를 던지는지 확인(instanceof)
  • fetchRecords: 502/500 등 기타 실패 → ApiError(에러 바디의 message·status 반영)를 던지는지 확인
  • shouldRetryRecordsQuery: UnauthorizedError면 failureCount와 무관하게 항상 false, 그 외 에러면 failureCount < 2 구간에서만 true
  • useRecords(renderHook + QueryClientProvider 래퍼): 성공 시 data가 타입대로 채워지고 isPending이 false로 전이되는지 확인
  • useRecords: 401 시 error가 UnauthorizedError 인스턴스이고, fetch가 1회만 호출되는지(재시도 안 함) 확인
  • 회귀: npm run typecheck · npx vitest run 전체 그린(기존 13파일/194테스트 + 신규 테스트 모두 통과)
