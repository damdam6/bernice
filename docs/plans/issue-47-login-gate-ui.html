P2 — 로그인 게이트 UI

1요약

로그인 게이트 UI는 팀 전용 열람 화면(홈·랭킹·개인) 진입 시 인증 여부를 확인해, 미인증 상태면 데이터 화면 대신 팀 패스코드 입력 화면을 전면 노출하고, 인증 상태면 기존 3탭 레이아웃(MainLayout)을 그대로 보여주는 게이트 컴포넌트를 추가하는 작업이다. 인증 판정은 새 엔드포인트 없이 기존 useRecords 훅(#46)이 던지는 UnauthorizedError를 그대로 활용하고, 로그인 성공 시 POST /api/login(#41)이 발급한 세션 쿠키를 반영해 records 쿼리를 무효화·재조회함으로써 게이트가 자동으로 해제되고 원래 화면으로 복귀하도록 한다. 관리자 로그인 화면(#67)은 이 작업 범위가 아니다.

✅ 구현·테스트·검증 완료 (2026-07-19). vitest 전체 325개 통과(신설 LoginGate.test.tsx 6개 포함), tsc -b typecheck 무오류, vite build 정상. vite 개발 서버 + 브라우저로 게이트 화면(아이콘·카피·입력·오답 에러 메시지)을 직접 렌더해 눈으로 확인 — #42 미머지로 실제 401 왕복까지는 못 보고, 그 부분은 fetch 스텁 단위 테스트로 대체 검증(리스크 §5 참고).

2현황 분석

  • 라우팅 구조 — src/App.tsx:19-23에서 /·/rankings·/players 3개 라우트가 <Route element={<MainLayout />}>로 감싸여 있고, 관리자 스택은 src/App.tsx:26-34의 별도 <Route element={<AdminLayout />}>로 완전히 분리돼 있다 — 이 이슈는 전자에만 손을 댄다.
  • 401 감지는 이미 확정 — src/hooks/useRecords.ts:21-25의 fetchRecords가 401을 UnauthorizedError(src/lib/api-error.ts:14-19)로 던지고, shouldRetryRecordsQuery(src/hooks/useRecords.ts:42-45)가 4xx 재시도를 막아 즉시 에러 상태로 정착한다(#46 완료).
  • POST /api/login은 이미 구현돼 있다(#41 완료, functions/api/login.ts:16-40) — {code} JSON을 받아 성공 시 {ok:true, role} + Set-Cookie(HttpOnly·SameSite=Lax), 실패 시 401 {error:'invalid_passcode', message}를 돌려준다.
  • functions/api/records.ts:1-3 주석대로 /api/records는 아직 인증 검사를 하지 않는다 — 인증 미들웨어는 별도 이슈(#42, OPEN)이고 #47의 의존성 목록엔 없다. 즉 이 작업은 "401이 오면"이라는 이미 확정된 계약에만 반응하도록 짜고, 실제로 401이 발생하는 시점(#42 머지 여부)과 독립적으로 완결돼야 한다.
  • src/pages/Home.tsx·Rankings.tsx·Players.tsx는 전부 P0/P3 스텁이라 아직 useRecords를 호출하지 않는다 — 따라서 게이트가 인증 신호로 쓸 수 있는 유일한 실제 쿼리는 useRecords뿐이다.
  • src/index.css:11-49 @theme 토큰엔 카드 보더(#edeef2)는 있지만 입력 필드 전용 배경(#f6f7f9)·보더(#e3e5ec) 토큰(docs/prd-design.html:134)이 없다 — 이 화면이 앱 최초의 텍스트 입력 UI라 토큰 추가가 필요하다.
  • src/components/common/Spinner.tsx·ErrorPanel.tsx·EmptyState.tsx(#49)는 이미 인디고 테마로 확정돼 재사용 가능. Spinner는 현재 어디서도 실사용되지 않고 정의만 존재한다 — 게이트의 초기 로딩 상태가 첫 소비처가 된다.
  • docs/prd-design.html §09(라인 289)에 "팀 패스코드 게이트 화면 — 관리자 로그인 화면과 같은 레이아웃 문법(아이콘 블록+입력+버튼)으로 1화면 추가 설계"가 열린 항목으로 명시돼 있어, 이번 구현이 그 설계를 확정짓는다.

3설계

채택안 — LoginGate라는 새 컴포넌트를 만들어 App.tsx의 3탭 라우트 그룹 element로 MainLayout 대신 배치한다. LoginGate는 내부에서 useRecords()를 호출해:

  • isPending → 전면 로딩(Spinner)
  • error instanceof UnauthorizedError → 전면 패스코드 입력 화면
  • 그 외(성공 또는 401이 아닌 다른 실패) → <MainLayout /> 렌더(그 안의 <Outlet/>이 원래 매치된 라우트를 그대로 보여줌)

라우트 자체는 바뀌지 않으므로(주소창 URL 유지) 로그인 성공 후 "원래 화면 복귀"가 리다이렉트 없이 자연스럽게 이뤄진다 — 제출 성공 시 queryClient.invalidateQueries({'{'}queryKey:['records']{'}'})가 같은 useRecords 관찰자를 재조회시키고, 성공하면 LoginGate가 자동으로 MainLayout 분기로 다시 렌더된다.

거부한 대안

  • 별도 /api/whoami 세션 확인 엔드포인트 신설 — 이슈 의존성(#41·#46)에 없고, #46이 이미 확정한 401 계약을 useRecords가 그대로 노출하므로 새 API 표면을 만들 필요가 없다.
  • <Navigate> 리다이렉트로 /login 전용 라우트 분리 — 인증 상태가 바뀔 때마다 URL을 오가야 해서 "원래 화면 복귀"가 라우터 히스토리에 의존하게 되고, 목업 부재 화면(§09)의 취지("데이터 화면 자리에 게이트만 노출")와도 어긋난다. 레이아웃 스위치가 더 단순하고 URL 불변이라 안전하다.
  • 인증 상태를 별도 Context/전역 상태로 관리 — TanStack Query가 이미 records 쿼리의 캐시·재시도·무효화를 담당하므로, 별도 상태를 두면 두 소스가 어긋날 위험(예: 쿠키는 만료됐는데 로컬 상태는 "인증됨")이 생긴다. 쿼리 상태를 단일 진실 공급원으로 둔다.

패스코드 화면 자체는 확정 목업이 없어(§09) 관리자 로그인과 같은 레이아웃 문법(아이콘 블록 + 안내 카피 + password 입력 + 에러 문구 + 버튼)으로 새로 설계한다: 자물쇠 아이콘(lucide-react Lock) + "팀 패스코드를 입력하면 기록을 볼 수 있어요" 안내 + 자간 넓은 password/numeric 입력 + 빨간 에러 문구(role="alert") + "입장하기" 버튼(빈 값·제출 중 비활성).

4작업 단계

  1. 입력 필드 전용 디자인 토큰 추가 — @theme에 --color-input-bg: #f6f7f9, --color-input-line: #e3e5ec(docs/prd-design.html §03 근거) src/index.css ✔ 완료 — 계획대로 두 토큰 추가.
  2. LoginGate 컴포넌트 신설 — useRecords 상태(pending/401/그외)에 따라 Spinner·패스코드 화면·MainLayout 3분기 렌더 src/components/auth/LoginGate.tsx ✔ 완료 — 계획대로 3분기 구현.
  3. 패스코드 제출 폼 구현 — fetch('/api/login', {'{'}method:'POST', body: JSON.stringify({'{'}code{'}'}){'}'}), 실패(!res.ok) 시 응답 message로 에러 노출·재입력 허용, 성공 시 queryClient.invalidateQueries({'{'}queryKey:['records']{'}'})로 게이트 자동 해제 src/components/auth/LoginGate.tsx ✔ 완료 — 자물쇠 아이콘 블록 + password/numeric 입력(자간 넓게) + role="alert" 에러 문구 + 빈 값/제출 중 비활성 버튼까지 포함.
  4. App.tsx의 3탭 라우트 그룹 element를 <MainLayout />에서 <LoginGate />로 교체(관리자 스택 라우트는 그대로 둠) src/App.tsx ✔ 완료 — 관리자 라우트(AdminLayout)는 변경 없음.
  5. 단위 테스트 작성 — 로딩/401/200 각 상태별 렌더 분기, 정답 제출 후 게이트 해제, 오답 제출 후 에러 메시지 유지·재제출 가능, 네트워크 실패 시 일반 에러 메시지(useRecords.test.tsx의 fetch 스텁 패턴 재사용, URL별 분기 필요: /api/records vs /api/login) src/components/auth/LoginGate.test.tsx ✔ 완료 — 계획한 6개 케이스 모두 작성, 전부 통과.
  6. npm run typecheck · npm test 실행해 회귀 확인 전체 ✔ 완료 — typecheck 무오류, vitest 325개 통과(31개 파일), vite build도 정상 확인. 추가: npm run dev로 실제 브라우저 렌더도 확인 — 잠시 LoginGate가 항상 패스코드 화면을 반환하도록 임시 수정해 아이콘·입력·오답 에러 메시지 스타일을 눈으로 검증한 뒤 원복(커밋에는 포함 안 됨).

5리스크

리스크영향완화책
#42(인증 미들웨어)가 아직 머지되지 않아 로컬 wrangler pages dev·실제 배포 환경에서 /api/records가 아직 401을 내지 않음브라우저로 전체 흐름(미인증→게이트→인증→데이터)을 실사용 환경에서 눈으로 확인하기 어려움#42 이슈 노트가 이미 이 순서 무관성을 인지하고 있음(프로덕션 배포 전이라 영향 없음) — 컴포넌트 테스트로 fetch를 스텁해 401/200 분기를 검증하고, #42 머지 후 별도로 수동 확인
패스코드 화면 카피·문구가 확정 목업 없이 새로 설계됨실제 팀 사용자 경험과 다를 수 있음§09 열린 항목에 이미 "구현 이슈에서 확정"으로 위임돼 있음 — 필요시 후속 PR에서 카피만 조정
이후 Home/Rankings/Players가 각자 useRecords를 호출하기 시작하면 LoginGate와 같은 쿼리를 공유(TanStack Query는 queryKey로 옵저버 병합)낮음 — 정상적인 캐시 공유 동작이라 충돌 없음후속 이슈에서 useRecords 옵션을 바꿀 경우 LoginGate의 분기 조건(isPending/UnauthorizedError)에 영향 없는지 확인

6테스트 계획

  • npm run typecheck — 타입 회귀 없음 확인
  • npm test(vitest) — 신설 LoginGate.test.tsx 전부 통과 + 기존 스위트(useRecords·api-error 등) 회귀 없음 확인
  • 로딩 중엔 Spinner만 노출 — 게이트 화면·MainLayout 콘텐츠 모두 미노출
  • 401 응답이면 게이트 화면만 노출 — MainLayout/원래 라우트 콘텐츠는 미노출
  • 200 응답이면 MainLayout(원래 라우트 콘텐츠) 노출 — 게이트 미노출
  • 정답 코드 제출 → /api/login 200 → invalidateQueries → /api/records 재조회 200 → 게이트 사라지고 원래 라우트 콘텐츠 노출
  • 오답 코드 제출 → /api/login 401 → 에러 메시지 노출, 게이트 유지, 입력값으로 재시도 가능
  • 네트워크 실패(fetch reject) 시 일반 에러 메시지 노출, 게이트 유지
  • (수동, #42 머지 후) wrangler pages dev로 실제 쿠키 발급까지 눈으로 1회 확인 — 이 이슈의 완료 조건은 아니지만 회귀 방지용 메모로 남김

7완료 조건

  • 미인증(401) 상태에서 팀 패스코드 게이트만 노출되고 홈/랭킹/개인 데이터 화면은 노출되지 않는다
  • 올바른 패스코드 입력 → 제출 성공 → 게이트 해제 & (기존 라우트의) 데이터 로드 재개
  • 틀린 패스코드 입력 → 에러 메시지 노출, 게이트 유지
  • 관리자 로그인 화면(#67)·관리자 라우트는 이 변경의 영향을 받지 않는다
