관리자 제출 로직 공통 훅 추출 (useSubmitMutation)

damdam6/bernice 브랜치 feat/95-extract-submit-mutation-hook ← main 라벨 enhancement 2026-07-19

1요약

AddPlayers·CreateSheet·RecordsPlayerInput 세 관리자 화면이 동일한 제출 흐름 — submitting/submitError 상태 → API 호출 → 실패 시 메시지 노출·화면 유지, 성공 시 records 캐시 무효화 → 화면별 성공 처리(navigate·토스트) — 를 그대로 복제하고 있다. 세 API가 공유하는 { ok: true, … } | { ok: false, message } 결과 계약을 활용해, 제출 상태·실패 처리·invalidateQueries(RECORDS_QUERY_KEY)까지를 공통 훅 useSubmitMutation으로 끌어올리고, 화면마다 다른 부분(호출할 API·성공 콜백)만 주입하도록 바꾼다. PR #91에서 "rule of three(3번째 사례 #68) 나오면 추출"로 보류됐고, #68 머지로 세 사례가 모두 충족됐다.

2현황 분석

세 화면의 제출 핸들러는 상태 선언부터 분기까지 사실상 바이트 단위로 같다:

  • AddPlayers — src/pages/admin/AddPlayers.tsx:28-29에 submitError/submitting 상태, :78-94의 handleConfirm에서 addPlayers(session.date, [...selected]) 호출 → 실패 시 setSubmitError+setSubmitting(false)+return, 성공 시 invalidateQueries({ queryKey: RECORDS_QUERY_KEY, exact: true })(:90) 후 navigate('/admin/records/{date}', { state: { toast } }). 추가로 selectSession(:72-76)이 회차 전환 때 setSubmitError(null)로 에러를 지운다.
  • CreateSheet — src/pages/admin/CreateSheet.tsx:22-23 상태, :45-61의 handleConfirm에서 createSheet([...selected]) 호출 → 같은 실패 분기, 성공 시 invalidateQueries(:57) 후 navigate+토스트.
  • RecordsPlayerInput — 내부 PlayerInputContent의 src/pages/admin/RecordsPlayerInput.tsx:92-93 상태, :104-119의 handleSave에서 saveRecord(session.date, entry.playerId, raw) 호출 → 같은 실패 분기, 성공 시 invalidateQueries(:117) 후 onSaved(toast) 콜백 (부모가 navigate로 연결, :70).

세 API 래퍼(src/lib/add-players-api.ts·create-sheet-api.ts·records-write-api.ts)는 모두 Promise<{ ok: true, … } | { ok: false, error, message }>로 항상 resolve하도록 설계돼 있어 (네트워크 오류도 ok:false로 변환), 호출부가 try/catch 없이 결과만 분기한다 — 공통 훅이 기댈 수 있는 안정적 계약이다.

재시도/입력값 보존은 별도 코드가 아니라 "실패 시 화면에 머문다"의 자연스러운 결과다: 실패해도 컴포넌트가 언마운트되지 않아 선택 상태(useMultiSelect)·입력 필드(fields)가 그대로 남고, 버튼을 다시 누르면 재호출된다. 훅은 이 성질을 깨지 않기만 하면 된다.

화면마다 다른 것은 딱 둘: ① 호출할 API 함수(+인자), ② 성공 시 동작(navigate 대상·토스트 문구, 혹은 콜백 위임). 나머지는 전부 동일하다.

3설계

채택: src/hooks/useSubmitMutation.ts에 상태·흐름을 소유하고, 화면은 "무엇을 호출할지"와 "성공하면 무엇을 할지"만 주입한다. 제출 시점의 최신 상태(selected·fields)를 읽어야 하므로 클릭 시점에 thunk를 넘기는 형태(설정형이 아닌 호출형)로 설계해 stale-closure를 원천 차단한다. 

// src/hooks/useSubmitMutation.ts
export interface SubmitMutationSuccess { ok: true }
export interface SubmitMutationFailure { ok: false; message: string }

export function useSubmitMutation() {
  const queryClient = useQueryClient()
  const [submitting, setSubmitting] = useState(false)
  const [submitError, setSubmitError] = useState<string | null>(null)

  const clearError = useCallback(() => setSubmitError(null), [])

  const submit = useCallback(
    async <R extends SubmitMutationSuccess | SubmitMutationFailure>(
      mutationFn: () => Promise<R>,
      onSuccess: (result: Extract<R, { ok: true }>) => void | Promise<void>,
    ) => {
      setSubmitting(true)
      setSubmitError(null)
      const result = await mutationFn()
      if (!result.ok) {
        setSubmitError(result.message)
        setSubmitting(false)      // 실패 → 화면 유지, 입력·선택 보존, 재시도 가능
        return
      }
      await queryClient.invalidateQueries({ queryKey: RECORDS_QUERY_KEY, exact: true })
      await onSuccess(result as Extract<R, { ok: true }>)   // 성공 → 언마운트 전이라 submitting 유지
    },
    [queryClient],
  )

  return { submitting, submitError, submit, clearError }
}

설계 근거

  • invalidate를 훅에 내장 — 세 화면 모두 RECORDS_QUERY_KEY를 exact:true로, 성공 콜백 직전에 무효화한다(동일 바이트). 이 한 줄이 중복의 핵심이라 훅이 소유한다. 다른 캐시가 필요한 미래 화면이 나오면 그때 매개변수화한다(현재 YAGNI).
  • 제네릭 R + Extract<R,{ok:true}> — () => addPlayers(...)의 반환 타입(AddPlayersResult)에서 R이 깔끔히 추론되고, 성공 콜백은 AddPlayersSuccess를 정확히 받는다(result.sessionDate·result.added 접근 가능). 내로잉 보강을 위해 콜백 호출부에서만 캐스팅.
  • 실패 시 setSubmitting(false), 성공 시 미변경 — 원본과 동일. 성공은 navigate로 언마운트되므로 submitting을 되돌릴 필요가 없다.
  • clearError 노출 — AddPlayers의 selectSession이 회차 전환 시 에러를 지우는 동작을 보존하기 위함.

범위 경계(제외): AdminLogin.tsx·LoginGate.tsx(PasscodeGate)도 같은 invalidateQueries(RECORDS_QUERY_KEY)를 쓰지만, 제출 형태가 다르다 — try/catch로 감싸고 (loginWithPasscode는 결과 유니온이 아니라 예외를 던지는 계약), role !== 'admin' 같은 도메인 검증과 토스트 없는 navigate/무이동을 포함한다. 이 훅에 끼워 맞추면 훅이 try/catch·옵션 분기로 비대해지므로 이번 rule-of-three 추출에서 의도적으로 제외한다(이슈 완료 조건도 세 화면으로 한정).

기각한 대안: ① TanStack useMutation 전면 도입 — 이득 대비 세 화면 리라이트 폭이 크고 기존 결과-유니온 계약과 겹쳐 중복 추상화가 된다. ② 설정형(useSubmitMutation({ mutationFn, onSuccess })) — selected/fields가 렌더마다 바뀌어 stale-closure·불필요한 의존성 배열 관리가 생긴다. 호출형이 더 단순.

5리스크

리스크영향완화책
제네릭 추론이 R = 전체 유니온이 아닌 성공 타입으로 좁혀지지 않아 onSuccess 인자 타입이 어긋남 타입 에러 또는 콜백에서 result.added 등 접근 불가 R extends 성공|실패 제약 + Extract<R,{ok:true}>로 성공만 추출, 호출부에서 캐스팅. tsc -b로 즉시 확인
invalidate를 훅에 내장하면서 성공 콜백과 순서가 바뀜 navigate가 캐시 무효화보다 먼저 실행돼 목록이 옛 데이터로 잠깐 보임 훅이 await invalidateQueries → await onSuccess 순서를 원본과 동일하게 고정
AddPlayers의 selectSession 에러 클리어 누락 회차 전환 후에도 이전 실패 메시지가 남음 clearError()로 명시 교체, 기존 동작 유지
실패 후 재시도/입력 보존 회귀 실패 시 선택·입력이 초기화되면 UX 손상 훅은 상태를 리셋하지 않음(컴포넌트 언마운트 없음). 세 페이지 테스트의 "실패 → 화면 유지" 케이스로 가드
