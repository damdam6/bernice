P2 — POST /api/login (패스코드 검증 → HttpOnly 쿠키)

damdam6/bernice 브랜치 feat/41-api-login-passcode-cookie ← main 라벨 enhancement 2026-07-18

1요약

팀 공용 패스코드와 관리자 코드를 검증해 HttpOnly 세션 쿠키를 발급하는 POST /api/login 엔드포인트를 신규 작성한다. 입력 코드를 TEAM_PASSCODE·ADMIN_CODE와 고정시간(constant-time) 비교해 일치한 쪽에 따라 세션 role(team/admin)을 정하고, 이미 완료된 #40의 세션 쿠키 유틸(functions/lib/session-cookie.ts)을 확장해 토큰 페이로드를 {iat, exp} → {iat, exp, role}로 넓힌다. 성공 시 Set-Cookie(HttpOnly·Secure·SameSite=Lax·Max-Age), 실패 시 401 JSON을 돌려주며, 검증 유틸은 role을 노출하도록 확장해 후속 #42 미들웨어가 그대로 소비하게 한다. vitest 테스트 포함.

✅ 구현·테스트·검증 완료 (2026-07-18). vitest 전체 272개 통과, tsc -b typecheck 무오류. wrangler pages dev로 실제 HTTP 스모크 확인 — 팀/관리자 코드가 각각 role=team/role=admin 쿠키(HttpOnly·SameSite=Lax·Max-Age=2592000, http에선 Secure 생략) 발급, 오답·누락 401(쿠키 없음), 비-JSON 400. 발급 쿠키 payload가 {iat, exp, role}로 디코드됨.

2현황 분석

  • 세션 쿠키 유틸은 이미 있으나 role 개념이 없다. functions/lib/session-cookie.ts는 #40에서 완료됐고, issueSessionToken(secret, now)(session-cookie.ts:50)는 페이로드를 { iat, exp }만으로 발급한다(session-cookie.ts:54). verifySessionToken은 boolean만 반환하고 exp만 검사한다(session-cookie.ts:97-100) — 팀/관리자를 구분할 클레임이 없다.
  • 쿠키 조립·파싱은 이미 #41 계약대로 준비돼 있다. buildSessionSetCookie(session-cookie.ts:135)가 HttpOnly·Secure·SameSite=Lax·Max-Age를 이미 붙이고, secure 옵션으로 로컬 http를 위한 Secure 끄기까지 지원한다(session-cookie.ts:129, 136-138). SESSION_COOKIE_NAME='bernice_session'·SESSION_TTL_SECONDS(30일)도 상수로 있다 (session-cookie.ts:16, 19). 즉 토큰 발급/검증의 role 확장 + 새 핸들러가 이 이슈의 핵심이다.
  • Pages Function 핸들러 패턴이 확립돼 있다. functions/api/records.ts:14는 onRequestGet: PagesFunction<Env>로 선언하고 interface Env로 필요한 바인딩만 타입한다(records.ts:10-12). 응답은 Response.json(body, { status })로 만든다(records.ts:37-48). login.ts는 이 패턴을 onRequestPost로 따른다.
  • 필요한 시크릿은 이미 문서·설정에 자리가 있다. .dev.vars.example에 TEAM_PASSCODE·SESSION_SECRET·ADMIN_CODE가 모두 정의돼 있고(값은 change-me 플레이스홀더), wrangler.jsonc 주석은 이 셋을 Secret으로 관리하라고 명시한다(로컬 .dev.vars / 프로덕션 Pages Secret).
  • Workers 런타임엔 node:crypto가 없다. 기존 코드는 전부 crypto.subtle(Web Crypto)만 쓴다 (session-cookie.ts:3, 37, 57, googleAuth.ts:2, 118). 따라서 고정시간 비교도 node:crypto.timingSafeEqual이 아니라 Web Crypto로 구현해야 한다.
  • 테스트 환경. vitest는 node 환경(vitest.config.ts)이며, 기존 records.test.ts는 Response.json·new Request·crypto.subtle를 전역으로 그대로 쓴다. lib/*.ts마다 짝이 되는 *.test.ts가 있는 컨벤션(순수 유틸 = 전용 단위테스트).
  • 현재 /api/login·role 소비자는 없다. 프론트/미들웨어에 role·verifySessionToken 소비처가 아직 없어(grep 결과 유틸 자기 자신·주석뿐) 검증 유틸의 반환 타입을 지금 바꿔도 깨질 외부 호출부가 없다.

3설계

PRD 확정안(§03②, docs/prd-record-input.html:171)을 그대로 채택한다: 쿠키는 1개, 토큰 페이로드에 role: 'team' | 'admin' 클레임을 넣어 미들웨어(#42)에서 role을 검사한다. (대안인 "관리자용 별도 쿠키"는 쿠키 2개 관리 부담으로 PRD가 이미 차선으로 정리.)

  • 토큰 발급 시그니처: issueSessionToken(secret, role, now?). role을 필수 위치 인자로 추가하고 페이로드에 role을 써넣는다. TOKEN_VERSION은 v1 유지 — role이 없던 구 v1 토큰은 검증 단계에서 role 누락으로 거부되므로(아래) 사실상 무효화된다(운영에 발급된 토큰 없음).
  • 검증 유틸은 role을 노출하도록 확장: verifySessionToken → SessionClaims | null. 기존 boolean 대신 성공 시 { role, iat, exp }, 실패 시 null을 반환한다. 이유: (1) 이슈 범위가 "발급·검증 유틸 확장 포함"을 명시, (2) 후속 #42 미들웨어가 role을 필요로 함, (3) 현재 외부 소비자가 없어 지금이 반환 타입을 바꿀 유일하게 깨끗한 시점. if (await verify…)식 불리언 사용도 truthy/null로 그대로 성립한다. role이 'team'|'admin'이 아니면 null(엄격 검증). 대안 "boolean 유지 + role 검증만" → #42에서 반환 타입을 또 깨야 해 기각.
  • 고정시간 비교는 double-HMAC 방식으로 신규 유틸 분리: functions/lib/constant-time-equal.ts. 요청마다 임의 키를 만들어 두 문자열의 HMAC-SHA256을 각각 계산하고 crypto.subtle.verify(타이밍 세이프 내장)로 비교한다 — 항상 32바이트 다이제스트를 비교하므로 길이 노출이 없고, 임의 키라 사전계산 공격면도 없다. JS 수동 XOR 루프(JIT로 상수시간 보장 불가·길이 노출)보다 Workers에서 견고. lib/* = 전용 테스트 컨벤션에 맞춰 단위테스트를 붙인다.
  • role 결정 순서와 빈 코드 가드(보안 핵심). 제출 코드를 ADMIN_CODE·TEAM_PASSCODE 양쪽 모두 비교(단락 없이)한 뒤 role을 정한다. 관리자 우선(둘이 같게 설정된 오설정 시 admin). 단, 설정값이 빈 문자열이면 그 경로는 절대 매치로 치지 않는다 — 그렇지 않으면 change-me 미설정 상태에서 빈 입력이 통과하는 구멍이 생긴다. 제출 코드가 없거나 빈 문자열이어도 401.
  • 핸들러 응답 계약. 성공 → 200 { ok: true, role } + Set-Cookie (buildSessionSetCookie(token, { secure })). 실패(코드 불일치·누락·빈값) → 401 { error: 'invalid_passcode' }, Set-Cookie 없음. 바디 파싱 실패(비-JSON) → 400 { error: 'bad_request' }. 열거 오라클을 피하려 team/admin 성공은 role만 다르고 실패는 단일 메시지로 통일.
  • 로컬 http 개발 대응. Set-Cookie의 Secure는 요청 스킴으로 결정한다 — secure: new URL(request.url).protocol === 'https:'. 프로덕션(https)은 Secure 포함(계약 충족), wrangler pages dev의 http://localhost에선 Secure 생략(브라우저가 저장 가능). 유틸의 secure 옵션(session-cookie.ts:129)이 정확히 이 용도로 이미 존재.
  • 메서드 제한. onRequestPost만 export하면 다른 메서드는 Pages가 405로 처리한다(추가 코드 불필요).

4작업 단계

  1. 고정시간 비교 유틸 신규 작성 — 요청별 임의 키 double-HMAC(crypto.subtle.sign/verify)으로 상수시간 문자열 동등 비교. 빈 입력·다른 길이 처리 포함. functions/lib/constant-time-equal.ts ✔ 완료 — 계획대로 double-HMAC로 구현.
  2. 고정시간 비교 유틸 단위테스트 — 같은 문자열 true / 다른 문자열 false / 다른 길이 false / 빈 문자열 경계. functions/lib/constant-time-equal.test.ts ✔ 완료 — 4개 케이스 통과.
  3. 세션 유틸 role 확장 — SessionRole·SessionClaims 타입 export, issueSessionToken(secret, role, now?)가 페이로드에 role 기록, verifySessionToken이 SessionClaims | null 반환 + role('team'|'admin') 엄격 검증. 파일 상단 계약 주석도 갱신. functions/lib/session-cookie.ts ✔ 완료 — iat/exp/role 모두 검증 후 claims 반환. 상단 계약 주석 갱신.
  4. 세션 유틸 테스트 갱신 — 기존 issueSessionToken 호출에 role 인자 추가, .toBe(true/false) 어서션을 toBeNull/toMatchObject({ role })로 조정. 신규: 페이로드에 role 포함 / role 누락·이상값 토큰은 null(signPayload 헬퍼로 위조). functions/lib/session-cookie.test.ts ✔ 완료 — 파일 재작성, 'role 클레임' describe 추가.
  5. POST /api/login 핸들러 신규 — interface Env { TEAM_PASSCODE; ADMIN_CODE; SESSION_SECRET }, JSON 바디 { code } 파싱, 빈 설정값 가드 + 양쪽 고정시간 비교로 role 결정(admin 우선), 성공 시 토큰 발급·Set-Cookie(스킴 기반 secure)·200 { ok, role }, 실패 401 / 비-JSON 400. functions/api/login.ts ✔ 완료 — onRequestPost만 export. 빈 code는 검증 전 차단 + matches 이중 방어.
  6. /api/login 핸들러 테스트 — 팀→role=team·관리자→role=admin(쿠키 파싱해 verifySessionToken 왕복 확인), 오답·빈코드→401·Set-Cookie 없음, 빈 설정값→401, 비-JSON→400, 쿠키 속성(HttpOnly·SameSite=Lax·https일 때 Secure) 검증. records.test.ts의 context 조립 패턴 참고. functions/api/login.test.ts ✔ 완료 — admin 우선 케이스·http Secure 생략 케이스 추가.

5리스크

리스크영향완화책
빈/미설정 코드(change-me 방치, secret 누락)가 빈 입력과 매치 인증 우회 — 누구나 로그인 설정값이 빈 문자열이면 그 비교는 매치 불가로 강제. 제출 코드 빈값도 401. 전용 테스트로 회귀 방지.
고정시간 비교가 실제로 상수시간이 아님(타이밍 사이드채널) 패스코드 브루트포스 가속 수동 JS 루프 대신 임의키 double-HMAC + crypto.subtle.verify(타이밍 세이프). 동일 길이 다이제스트 비교로 길이 노출 제거.
verifySessionToken 반환 타입 변경(boolean → claims|null) 기존 호출부·테스트 파손 현재 외부 소비자 0(유틸/주석뿐). 세션 테스트를 같은 PR에서 갱신. 불리언 문맥(if)은 truthy/null로 그대로 동작.
로컬 http 개발에서 Secure 쿠키 미저장 로컬 로그인 검증 불가 요청 스킴으로 Secure 결정(https만 Secure). 유틸의 secure 옵션 활용 — 프로덕션 계약은 유지.
role 없던 구 v1 토큰이 있었다면 확장 후 거부됨 기존 세션 강제 로그아웃 운영 발급 이력 없음(#41이 최초 발급 경로). 버전 v1 유지가 안전 — 재로그인만 필요.

6테스트 계획

  • 고정시간 유틸(constant-time-equal.test.ts): 같은 값 → true, 다른 값 → false, 길이가 다른 값 → false, 빈 문자열 경계 처리.

  • 빈 설정값 회귀: TEAM_PASSCODE=''·ADMIN_CODE=''에 빈 입력 → 401(우회 없음).
