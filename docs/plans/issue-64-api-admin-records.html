기록 입력 W3 — POST /api/admin/records (선수 단위 점수 저장)

damdam6/bernice 브랜치 feat/64-api-admin-records ← main 라벨 enhancement 2026-07-19

1요약

관리자가 입력 화면에서 저장한 한 선수의 회차 점수 전체를 회차 탭의 해당 행에 기록하는 신규 엔드포인트 POST /api/admin/records를 만든다. 요청은 { sessionDate, playerId, scores }로 받고, 서버는 id → 명단 이름 → 회차 탭 이름 매칭 순으로 대상 행을 찾아 점수 셀 범위만 values.update(RAW)로 덮어쓴다(행 단위 upsert, 마지막 저장 승리).

검증은 새 규칙을 만들지 않고 기존 normalize-score + valueKind 교차검증을 그대로 재사용한다 — invalid·형식 불일치는 400으로 거부하고 시트에 쓰지 않는다. 저장 성공 시에만 RECORDS_CACHE_KEY를 응답 전 await로 무효화한 뒤 정규화 결과(EventScore)를 돌려준다. 인증(401/403)은 이미 배포된 _middleware.ts가 담당하므로 이 이슈는 200/400/404×2/502 계약만 구현한다.

2현황 분석

  • 플랫폼 — Cloudflare Pages Functions. 파일 경로가 곧 라우트라 functions/api/admin/records.ts의 onRequestPost가 POST /api/admin/records로 매핑된다. 기존 functions/api/records.ts는 GET /api/records(다른 경로)라 충돌 없음.
  • 인증은 이미 완료 — functions/api/_middleware.ts:37이 /api/admin/*에 admin 세션이 아니면 403, 세션 없으면 401을 fail-closed로 반환한다. 아직 함수 파일이 없는 admin 경로에도 걸리게 설계돼 있어(주석 8–10행), 이 핸들러는 인증을 다시 구현하지 않는다.
  • 쓰기 래퍼(#63)는 준비됨 — functions/lib/sheetsWriteApi.ts:28 updateValues(env, sheetId, range, values)가 쓰기 scope 토큰 발급 + values.update(RAW) + 429·5xx 재시도(멱등)를 담당하고, 4xx는 즉시 SheetsApiError. 범위 문자열 조립·행 찾기·검증은 "이 모듈 범위 밖"으로 남겨져(주석 4행) 이번 이슈 몫이다.
  • 파서·검증 로직도 준비됨 — functions/lib/parse-session.ts의 mapHeaderToEvents(헤더→종목 열 매핑, 열 순서 가정 없음)와 buildEventScore(normalize-score + valueKind 교차검증 → EventScore)가 이미 존재하나 모듈 밖으로 export되지 않은 내부 함수다. PRD §07이 "parse-session의 mapHeaderToEvents와 같은 규칙"을 명시하므로 재구현 대신 export해 재사용한다.
  • 읽기 경로 — functions/lib/sheetsApi.ts의 fetchSheetBundle가 메타(탭 목록) + batchGet으로 명단·목표·전 회차 원시값을 한 번에 가져오고, quoteSheetName으로 A1 탭 이름을 인용한다. classifySheetTabs(sheetTabs.ts)가 YYYY-MM-DD 회차 탭을 캘린더 유효성까지 왕복 검증한다.
  • 도메인 계약 — shared/domain.ts:48 EventScore(recorded/exempt/unmeasured/invalid), EventDefinition(key·valueKind·direction 등). 200 응답의 scores는 정확히 buildEventScore가 만드는 Record<string, EventScore>다.
  • 캐시 — functions/lib/records-cache.ts:7 RECORDS_CACHE_KEY는 고정 합성 URL. GET 경로가 caches.default에 이 키로 저장하므로(records.ts:29), 무효화는 같은 키로 cache.delete.
  • 테스트 관행 — 핸들러 단위 테스트는 records.test.ts처럼 vi.mock으로 fetchSheetBundle을 스텁하고 caches를 stubGlobal로 대체해 onRequestGet(context)를 직접 호출한다. 쓰기 래퍼 테스트(sheetsWriteApi.test.ts)는 fetch를 큐로 스텁한다.

3설계

핵심 원칙: 새 코드는 오케스트레이션(요청 파싱 → 읽기 → 대상 행 찾기 → 검증 → 쓰기 → 캐시 → 응답)과 A1 범위 조립뿐. 검증·헤더 매핑·이름 매칭 규칙은 기존 순수 함수를 재사용한다(PRD §08 "규칙 원천은 하나").

읽기 전략 — fetchSheetBundle 재사용

  • 대상 행을 찾으려면 명단(playerId→이름)·목표(events·valueKind)·해당 회차 탭 3종이 필요하다. 전용 메타+batchGet을 새로 짜는 대신 GET 경로와 동일한 fetchSheetBundle을 재사용해 번들에서 셋을 골라 쓴다 — 회차 탭 전부를 읽는 약간의 초과 비용은 팀 규모(약 21명 × 소수 회차)에서 무시 가능하고, 검증된 단일 읽기 경로를 유지하는 이점이 더 크다. (회차 탭 존재 판정 = bundle.rounds에서 날짜 매칭 → 404 자연 도출.)

쓰기 범위 조립

  • mapHeaderToEvents는 이름 열(A) 뒤 모든 열이 종목이고 전 종목이 빠짐없이 존재함을 강제한다. 따라서 점수 열은 B열부터 연속이며 개수 = events.length. 쓰기 범위는 '{탭}'!B{행}:{마지막열}{행} 한 건 — 이름 열·헤더에는 절대 쓰지 않는다(PRD §07).
  • 쓰는 값은 요청 원본 문자열을 헤더 열 순서대로 재배열한 배열(요청 key 순서가 아님). RAW로 저장돼 읽기 쪽 FORMATTED_VALUE와 왕복 일관. 빈 문자열 ""은 셀 비움 = 미측정.

에러 계약 & 판정 순서

순서: 요청 형식(400) → 자원 존재(404) → 값 유효성(400) → 쓰기(502). "잘 만들어진 요청인가 → 대상이 있는가 → 값이 유효한가" 흐름. 401/403은 미들웨어가 이 핸들러 도달 전에 처리.

코드상황 / 판정
400 validation_failed 비-JSON 바디 · 타입 오류(sessionDate/playerId/scores) · sessionDate 날짜 형식 오류 · 존재하지 않는 playerId · scores key 집합이 events와 불일치(누락·미지) · normalize-score invalid · valueKind 불일치. invalid·불일치는 사유(reason) 포함해 반환, 시트 미기록.
404 session_not_found 형식은 맞으나 그 날짜의 회차 탭이 번들에 없음 → 프론트 [기록지 만들기] 안내.
404 not_participant id→이름은 풀렸으나 회차 탭 이름 열에 그 이름 행이 없음 → 프론트 [참가자 추가] 안내.
502 sheets_api_error 읽기/쓰기 Sheets API 실패(재시도 소진 포함). GET과 동일한 {error, message, upstreamStatus}.
500 sheet_data_invalid 시트 무결성 문제(헤더 깨짐 · 목표/명단 탭 누락 · 회차 탭 이름 열 동명 중복으로 대상 행 특정 불가). GET 경로의 파서 Error → 500 매핑과 일관. (계약표 밖 방어적 폴백.)
200 { sessionDate, playerId, name, scores: Record<key, EventScore> } — 저장값의 정규화 결과. name은 명단 기준 이름.

검토했으나 택하지 않은 것

  • 전용 3-range batchGet(회차 탭 1개만) — 데이터는 줄지만 메타+인용+오케스트레이션을 재구현해야 함. 절감 효과가 팀 규모에서 미미해 fetchSheetBundle 재사용을 택함.
  • normalize-score를 shared/로 승격 — PRD §08이 "구현 이슈에서 검토"로만 남긴 항목. 프론트 재사용(후속 입력 화면 이슈) 필요 시점에 옮기는 게 맞고, 이 백엔드 이슈에서는 functions/ 내부에서 직접 import하면 충분하므로 이번 범위 밖.
  • parse-session 전체를 회차 탭에 실행 — Session 전체를 파싱하면 대상 시트 행 번호를 얻을 수 없어 쓰기 범위를 못 만든다. 헤더 매핑·행 찾기만 뽑아 쓰는 소형 헬퍼로 분리.

5리스크

리스크영향완화책
이름 열·헤더에 실수로 쓰기참조 수식 파괴 → 명단/랭킹 붕괴 쓰기 범위를 B{행}부터로 고정, 값 배열 길이 = events.length. 테스트로 범위 시작열이 B임을 단언.
요청 key 순서 ≠ 시트 열 순서다른 종목 칸에 값 저장 쓰기 배열을 헤더 열 순서(eventColumns)로 재배열. key는 값 조회에만 사용.
회차 탭 이름 열 동명이인엉뚱한 행 덮어쓰기 2건 이상 매칭이면 쓰지 않고 500 sheet_data_invalid(parse-session과 동일한 fail-loud).
invalid가 시트에 유입사고 유형 재발 검증을 쓰기 이전에 완료, 하나라도 invalid면 400 후 즉시 반환(updateValues 미호출).
캐시 무효화가 응답보다 늦음프론트가 오래된 데이터 refetch cache.delete를 waitUntil 아닌 await로 응답 전에 수행(PRD §09).
행 찾기↔쓰기 사이 행 이동레이스로 다른 행 기록 회차 탭은 append-only(행 삭제·재배치 금지). 벌어지면 파서가 fail-loud로 드러냄(PRD §07 근거).

  • 순수 헬퍼(record-write.test.ts): columnLetter 경계(1→A,2→B,26→Z,27→AA), locateParticipantRow 정상 행번호·미참가·동명중복·NFC 정규화 매칭, 범위 조립이 B.. 연속인지, scores 평가가 recorded/exempt/unmeasured/invalid/valueKind불일치를 정확히 분류하고 빈칸을 미측정으로 두는지.

  • invalid 점수·valueKind 불일치 → 400 validation_failed(reason 포함) + updateValues 미호출.

  • 쓰기 SheetsApiError → 502 sheets_api_error(+upstreamStatus), 동명중복 → 500.
