#26기능 개선 (파서 구현)

P1 — 목표 파서 (종목·목표치·만점·방향)

damdam6/bernice 브랜치 feat/26-goals-parser ← main 라벨 없음 2026-07-13

1요약

목표 탭 원시 2D 배열(string[][])을 받아 shared/domain.ts의 EventDefinition[]으로 변환하는 순수 함수 파서를 새로 만든다. 열 구성은 종목 | 목표 | 만점 | 방향(헤더 1행) — 목표치·만점의 시간형 값은 이미 병합된 #20의 normalizeScore를 그대로 재사용해 초 단위로 변환하고, 종목명은 NFC로 정규화해 이후 회차 탭 종목 헤더와 매칭할 키(EventDefinition.key)로 삼는다. 이슈 본문의 #3·#6 참조는 각각 #21(도메인 타입, 이미 shared/domain.ts로 머지)과 #20(값 정규화, 이미 functions/lib/normalize-score.ts로 머지)을 가리키므로 재구현하지 않고 그대로 import해 쓴다.

2현황 분석

functions/lib/에는 아직 "원시 2D 배열 → 도메인 타입" 파서가 하나도 없다 — 형제 이슈인 #25(명단 파서)·#27(회차 파서)·#24(Sheets API 래퍼)도 전부 미착수(OPEN) 상태라, 이 이슈가 사실상 이 저장소의 첫 파서이자 이후 두 파서가 참고할 패턴이 된다.

  • 계약 대상 타입은 shared/domain.ts:34-42의 EventDefinition: { key, valueKind, target, targetValue, maxScore, direction }. RANK_DIRECTIONS = ['낮을수록', '높을수록'](domain.ts:9-10), EVENT_VALUE_KINDS = ['count', 'time'](domain.ts:13-14)도 같은 파일에 이미 정의돼 있다.
  • 시트 실제 레이아웃은 docs/sheet-integration.html:128-134 — 헤더 종목 | 목표 | 만점 | 방향, 예시 4행(드리블셔틀런 1:17/-/낮을수록, 골밑슛 5/10/높을수록, 자유투 2/5/높을수록, 45도패스캐치 5/7/높을수록). docs/sheet-rules.html:76-79가 이 탭의 행 순서를 고정 규칙으로 못박고 있고(정렬·재배치 금지), docs/sheet-rules.html:112-115는 표 중간 빈 행 삽입도 금지한다.
  • docs/records-schema.html:324-327의 응답 예시가 이 파서의 정답 형태를 그대로 보여준다: { key: "드리블셔틀런", valueKind: "time", target: "1:17", targetValue: 77, maxScore: null, direction: "낮을수록" }.
  • 재사용 대상 normalizeScore(functions/lib/normalize-score.ts:12-22)는 콜론 포함 여부로 count/seconds를 판별하고, NFKC 정규화로 전각 콜론·전각 숫자까지 처리하며, 이상값은 { kind: 'invalid', reason }으로 표현한다(throw 없음) — 셀 표기 형식이 목표 탭의 목표 열과 동일해 그대로 재사용 가능.
  • NFC 정규화 선례는 functions/lib/sheetTabs.ts:14-15,60(탭 이름 매칭)에 이미 있음 — 비교/키 생성 전 항상 .normalize('NFC') 하는 패턴을 그대로 따른다.
  • 참고 메모: 과거 PR #16과 PR #14(이슈 #6)가 동시 진행되며 normalize-status.ts가 shared/domain.ts의 PlayerStatus를 로컬 재정의하는 드리프트가 있었다. 이번 파서는 shared/domain.ts를 import만 하고 타입·상수를 재정의하지 않아 같은 문제를 만들지 않는다.

3설계

입력 형태 — parseGoals(rows: string[][]): EventDefinition[]. rows[0]은 헤더(스킵), 이후 행이 데이터. 열 인덱스는 docs/sheet-integration.html 순서 그대로 0=종목, 1=목표, 2=만점, 3=방향으로 고정 매핑한다(열 순서 자체가 시트 편집 규칙으로 고정돼 있음).

  • 종목(key) — 셀 값을 trim() 후 .normalize('NFC'). 회차 탭 종목 헤더 매칭·랭킹/추이 계산에서 조인 키로 쓰이는 값이라, sheetTabs.ts처럼 원본을 보존하지 않고 정규화된 문자열 자체를 key로 채택한다(Range 참조용 원본 보존이 필요한 탭 이름과는 성격이 다름).
  • 목표(target/targetValue/valueKind) — normalizeScore(raw)를 그대로 호출. kind: 'seconds' → valueKind: 'time', kind: 'count' → valueKind: 'count', value를 targetValue로. target은 셀 원본 문자열 그대로(정규화 없음 — domain.ts 주석의 "원본 표시값" 그대로 보존). blank/exempt/invalid가 나오면 설정 오류로 보고 즉시 에러.
  • 만점(maxScore) — normalizeScore는 재사용하지 않는다(면제 리터럴 등 목표 셀과 무관한 개념이 섞여 있어 오히려 헷갈림). 로컬 로직: trim + NFKC 후 빈 문자열 또는 -면 null(셔틀런처럼 만점 없는 종목), 그 외에는 /^\d+$/만 허용해 Number(). 시트가 실제로 빈칸을 쓸지 -를 쓸지 불확실하므로 방어적으로 둘 다 허용.
  • 방향(direction) — trim + NFKC 후 RANK_DIRECTIONS(domain.ts에서 import) 화이트리스트와 정확히 일치해야 통과. normalizeStatus와 동일한 접근(화이트리스트 매칭)이지만, 상태값처럼 "unknown"을 값으로 흡수할 도메인 슬롯이 EventDefinition에는 없어 불일치 시 에러로 처리한다.
  • 에러 정책 — 목표 탭은 회차마다 바뀌지 않는 설정 시트라(팀 운영자가 드물게 편집), 회차 점수 셀과 달리 EventDefinition에는 "이상값" 상태를 담을 필드가 없다(EventScore의 invalid와 대비). 행 파싱이 실패하면 시트 행 번호(헤더 다음이 2행)와 종목명을 담아 즉시 throw한다 — 운영자가 바로 시트에서 원인 셀을 찾을 수 있게.
  • 완전 공백 행 — 모든 셀이 빈 문자열인 행은 에러 없이 건너뛴다. docs/sheet-rules.html은 표 중간 빈 행 삽입을 금지하지만, #24(Sheets API 래퍼)가 아직 없어 실제 batchGet range가 데이터 끝보다 넓게 잡혀 trailing 빈 행이 섞여 올 가능성을 배제할 수 없다 — 방어적으로 스킵.

검토한 대안: 만점/방향도 normalizeScore/normalizeStatus처럼 "invalid를 값으로 반환"하는 방식도 고려했으나, EventDefinition에 그 상태를 실을 필드가 없어 결국 조립 단계에서 에러로 바꿔야 한다 — 그럴 바엔 파싱 시점에 바로 에러를 내는 편이 더 단순하고 원인 추적이 쉽다고 판단.

  2. functions/lib/parse-goals.test.ts 작성 — docs/sheet-integration.html 예시를 그대로 픽스처로 사용(4종목, 양방향, 시간형+개수형, 만점 null/숫자 혼합), NFC 정규화 케이스(NFD 종목명 입력), 만점 빈칸/- 동치 케이스, 공백 행 스킵 케이스, 에러 케이스(빈 종목명·잘못된 목표치·잘못된 만점·잘못된 방향)를 it.each 스타일로 커버. functions/lib/parse-goals.test.ts
  3. npm test·npm run typecheck 실행해 회귀 확인(기존 normalize-score·normalize-status·sheetTabs 테스트도 함께 통과하는지). package.json

6테스트 계획

  • npm test -- parse-goals — 신규 테스트 전부 통과 확인.
  • 양방향 케이스: 낮을수록(드리블셔틀런)·높을수록(골밑슛 등)가 각각 올바른 direction으로 매핑되는지.
  • 시간형 목표치: 1:17 → valueKind: 'time', targetValue: 77 — docs/records-schema.html 예시 JSON과 정확히 일치하는지 값 단위로 비교.
  • 개수형 목표치 + 만점 숫자/null 혼합(골밑슛 5/10 vs 드리블셔틀런 1:17/-)이 한 배열 안에서 모두 올바르게 나오는지.
  • NFD로 분해된 종목명을 넣어도 key가 NFC로 정규화되는지(sheetTabs.test.ts와 동일한 검증 패턴).
