선행 — 회차 파서 이름-매칭 정합화 (참가자만 포맷)

damdam6/bernice 브랜치 feat/60-session-parser-name-matching ← main 라벨 enhancement 2026-07-17

1요약

functions/lib/parse-session.ts는 아직도 "회차 탭 데이터 행 N = 버니스명단 행 위치 N+1"이라는 전원-미러링 가정으로 playerId를 도출한다. 그런데 실제 2025-05-16 회차 탭은 참가자 6명만 가나다순으로 손으로 입력돼 있고, PRD(docs/prd-record-input.html §03①)도 이 파서 정합화를 기록 입력 기능 전체의 필수 선행 작업으로 못박아 뒀다. 이 작업은 파서를 "이름 셀 텍스트 → 버니스명단 이름 매칭 → playerId" 방식으로 다시 짜고, 미등록 이름·동명이인을 fail-loud로 막고, 관련 문서의 "전원 미러링" 서술을 걷어낸다.

2현황 분석

현재 파서의 위치 기반 도출 — parse-session.ts:63 const playerId = index + 1이 회차 탭의 데이터 행 순서를 그대로 버니스명단 행 위치로 취급한다. 이름 셀은 parse-session.ts:79-85에서 이 playerId로 조회한 player.name과 일치하는지 확인하는 2차 검증으로만 쓰이고, 불일치하면 "명단 행 순서가 어긋났을 수 있습니다"라는 메시지로 Error를 던진다 — 참가자만·순서 무관 탭에서는 애초에 이 가정 자체가 성립하지 않는다.

명단 쪽 id 체계는 그대로 둔다 — functions/lib/roster.ts:67에서 Player.id는 버니스명단의 원본 행 위치(빈 행·이상값 행을 스킵해도 뒤 id를 당기지 않는 sparse 배열)로 이미 확정돼 있다. 이 이슈는 회차 탭 쪽의 매칭 방식만 바꾼다 — 명단 파서(#25)·Player.id 정의는 손대지 않는다.

호출 경로 — functions/lib/build-records-response.ts:47-53에서 parseRoster가 만든 players[](활동·탈퇴·비대상·휴식 전부 포함, 탈퇴 필터링 이전)가 그대로 parseSession에 넘어간다. 탈퇴 필터링은 excludeWithdrawn으로 파서 이후에 한 번만 하므로, 파서 자체는 여전히 모든 상태의 선수 이름을 매칭 대상으로 봐야 한다(과거 회차에 탈퇴자 기록이 남아 있을 수 있음) — 이 부분은 바꿀 필요 없다.

하류 소비자의 암묵적 전제 — functions/lib/player-summary.ts:37 session.entries.find(candidate => candidate.playerId === player.id)는 한 세션 안에 같은 playerId가 두 번 나타나지 않는다고 전제하고 첫 매치만 쓴다. 지금까지는 playerId = index + 1이라 세션 내 유일성이 자동으로 보장됐지만, 이름 매칭으로 바꾸면 같은 이름이 실수로 두 행에 입력되는 경우 이 유일성이 더 이상 자동으로 보장되지 않는다.

문서 불일치 — docs/sheet-integration.html:136-142는 2025-05-16 탭 예시를 "명단 전원 미러링, 참여자만 점수 입력"이라고 서술하고, 미참여자(선수7)도 빈 점수로 행이 존재하는 것으로 그린다. 같은 파일 :107 결정 표 8번 항목도 "회차는 이름 참조 수식"이라고 못박는다. 둘 다 이슈가 지적하는 실제 운영(참가자만, 가나다순, 수기 입력)과 모순된다. docs/prd-record-input.html:146-155가 이 정합화를 P5 이전 필수 선행 작업으로 명시한다.

3설계

채택안 — 이름 텍스트 매칭. players[]로부터 Map<정규화된 이름, Player[]>를 만들고, 각 데이터 행의 이름 셀(trim + NFC 정규화)로 이 맵을 조회해 playerId를 도출한다.

  • 매칭 0건 → "회차 탭 N행 이름(...)이 명단에 없습니다" Error (오타·미등록).
  • 매칭 2건 이상(명단 내 동명이인, sheet-rules.html §03이 이름 구분을 권장하지만 강제하진 않음) → "...동명이인으로 N명 있어 특정할 수 없습니다" Error.
  • 매칭 1건 → 그 Player.id를 playerId로 사용.

추가 안전장치 — 탭 내 중복 검사. 위치 기반 도출 시절엔 playerId가 세션 내에서 자동으로 유일했지만, 이름 매칭으로는 같은 이름이 실수로 두 행에 입력되는 경우가 새로 가능해진다. 이미 골라잡은 player.id를 Set으로 추적해, 같은 회차 탭 안에서 두 번째로 같은 사람이 나오면 즉시 Error를 던진다. 이슈의 완료 조건에 문구로 명시돼 있진 않지만, 현황 분석의 player-summary.ts:37 .find()가 조용히 첫 매치만 쓰는 실제 위험을 막기 위한, 기존 fail-loud 원칙의 자연스러운 연장이다.

기각한 대안 1 — 회차 탭에 별도 id 열 추가. 명시적이지만 시트 스키마 변경이 필요하고(sheet-rules.html §01 "헤더·컬럼 순서 고정" 규칙과 충돌), 사람이 관리해야 할 값이 하나 늘어난다 — 애초에 "선수 식별은 참조 구조, id 없음"이 확정 결정(sheet-integration.html §01)이라 그 결정 자체를 뒤집는 셈이라 기각.

기각한 대안 2 — 위치 기반 유지 + "앞에서부터 채움" 가정 추가. 참가자만 앞에서부터 순서대로 채운다면 여전히 위치로 풀 수 있다는 발상이지만, 이슈가 지적하는 실제 데이터는 가나다순이라 명단 순서(가입순)와 애초에 무관하다 — 위치 기반으로는 원천적으로 풀리지 않아 기각.

4작업 단계

  1. 파서 재작성: players[]로 Map<NFC 정규화 이름, Player[]>를 만들고, 데이터 행마다 이 맵으로 playerId를 조회하도록 parseSession 로직을 바꾼다. 매칭 0건/2건 이상/탭 내 중복 각각 전용 메시지로 Error. 기존 "명단에 없는 위치를 참조합니다(playerId=N)" 에러와 위치 기반 무결성 검증 분기는 제거한다. 파일 상단 주석(1~27행, "행 위치가 1차 키" 설명)도 이름 매칭 기준으로 다시 쓴다. functions/lib/parse-session.ts — 완료
  2. 기존 테스트 정리: "이름 셀이 명단과 일치하지 않으면"과 "명단에 없는 위치를 참조하면"을 "미등록 이름 → Error" 한 케이스로 통합하고, "결번 위치를 참조하는 행"(위치 기반 전용 시나리오라 이름 매칭 하에서는 성립 자체가 안 됨)은 제거한다. "결번 위치의 빈 행 스킵"은 이름 매칭으로도 여전히 유효한 회귀 케이스라 유지하되 설명 문구만 다듬는다. functions/lib/parse-session.test.ts — 완료
  3. 신규 테스트 추가: (a) 참가자만·명단 순서와 무관한 나열(명단은 가입순 4명, 회차는 그중 3명만 가나다순)이 정상 파싱되는 케이스, (b) 명단에 없는 이름 → Error, (c) 명단 내 동명이인 → Error, (d) 같은 회차 탭에 같은 사람이 두 행으로 나타남 → Error. functions/lib/parse-session.test.ts — 완료 (27개 테스트 그린)
  4. 통합 테스트 추가: 실제 2025-05-16 형태(명단 5명 가입순, 회차엔 4명만 가나다순·명단 순서와 다르게 등장)를 흉내 낸 픽스처로 buildRecordsResponse가 에러 없이 정상 조립되고 sessions[0].entries가 올바른 playerId로 매핑됨을 검증한다 — 완료 조건의 "실제 탭 형태로 /api/records 정상 응답"에 대응. functions/lib/build-records-response.test.ts — 완료
  5. 문서 갱신 — sheet-integration.html: §01 결정 표 8번 행의 "회차는 이름 참조 수식" 문구를 "회차는 이름 텍스트 매칭"으로 완화하고, §02 회차 탭 절의 부제("명단 전원 미러링, 참여자만 점수 입력")와 예시를 "참가자만 입력, 이름 매칭" + 명단 순서와 다른 나열로 교체한다. "이름 칸은 참조 수식, 직접 타이핑 X" 콜아웃도 실제 운영(수기 입력 확인됨)에 맞게 완화한다. docs/sheet-integration.html — 완료 (§01 표 8번 행, §02 회차 탭 절, §03 "선수 식별 & 명단 변화"의 개명 전파 서술도 함께 정합화 — 계획에 없던 추가 수정)

5리스크

리스크영향완화책
실제 명단에 동명이인이 등록돼 있는 상태에서 여러 회차가 이미 쌓여있는 경우배포 즉시 그 이름이 나오는 모든 회차 파싱이 Error로 막힘(fail-loud라 의도된 동작이지만 즉시 영향)배포 전 버니스명단(현재 21명 규모)에 동명이인이 있는지 한 번 수동 확인, 있다면 sheet-rules.html §03 안내대로 이름 구분(예: 선수5, 선수5B)
회차 탭 이름 셀에 미세한 오타(트레일링 공백 외의 실제 철자 차이)그 참가자가 "명단에 없음" Error로 회차 전체 파싱이 막힘에러 메시지에 원본 셀 값을 그대로 포함해 관리자가 시트에서 바로 찾아 고칠 수 있게 함(기존 관례 유지)
scripts/seed-sheet.mjs는 여전히 "명단 전원 미러링" 포맷으로 새 회차를 생성함(코드 상 scripts/seed-sheet.mjs:142)낮음 — 새 파서는 이름만 일치하면 전원 미러링 포맷도 참가자만 포맷도 동일하게 파싱하는 상위 호환이번 이슈 범위 밖으로 남겨둠. P5 create-sheet가 참가자만 생성 방식으로 자연히 대체할 예정
탭 내 중복 이름 검사는 이슈 완료 조건 문구에 명시되지 않은 추가 안전장치(설계 §3 참고)없음 — 오히려 player-summary.ts의 잠재적 조용한 오류를 막음계획 단계에서 근거를 명시했으니 리뷰 시 범위 동의 여부만 확인
