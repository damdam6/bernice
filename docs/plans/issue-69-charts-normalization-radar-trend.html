1요약

3설계

① 정규화 함수 확정 (§07 권장안 채택 + 방어 보강) — src/lib/performance-scale.ts 신설, 반환값은 항상 0~1 클램프:

  • 개수 종목(valueKind=count, maxScore≠null, direction=높을수록): value / maxScore.
  • 시간 종목 및 만점 없는 종목(valueKind=time 또는 maxScore=null): 전 회차·전 선수 recorded 값의 min~max 범위 스케일. direction 반영 — 낮을수록: (max−v)/(max−min), 높을수록: (v−min)/(max−min). 만점이 있어도 direction이 낮을수록면 value/maxScore가 의미 역전이라 범위 스케일로 폴백.
  • 엣지: min==max(관측값 전원 동률·단일 기록) → 0.5 중립. 관측값 0건 종목 → 0.5. 목표선 y = normalize(event, targetValue) — 목표가 관측 범위 밖이면 0/1로 클램프되어 차트 가장자리에 붙음(의도된 동작, 주석 문서화).
  • API: buildPerformanceScale(events, sessions) → { normalize(eventKey, value): number }. 레이더·추이·미니바가 이 인스턴스 하나를 공유 → 세 차트의 y값이 정의상 항상 일치.

② Recharts 제거, 경량 SVG 채택 — 목업이 전부 경량 SVG이고 §07 스펙이 픽셀 단위(1.6px/3.2px 라인, 점선 4 4, 폴리곤 14% 투명)라 라이브러리 기본값과 싸우는 것보다 직접 그리는 편이 정확하고 짧다. 4개 차트 모두 툴팁·줌 등 인터랙션 요구가 없고(모바일 우선), recharts는 번들에서 통째로 빠진다. 기각안: Recharts 유지 — 배경 라인(선수별 희소 시리즈) 피벗이 번거롭고, 미니바·게이지 바에는 애초에 불필요.

5리스크

리스크영향완화책
시간 종목 범위 스케일은 새 기록이 min/max를 갱신하면 과거 점의 y값도 함께 움직임회차가 쌓이면 같은 기록의 차트 위치가 달라져 보일 수 있음§07이 선택한 상대 성능 표현의 본질 — 모듈 주석에 명시. 세 차트가 단일 scale 공유라 화면 간 불일치는 없음
목표선이 관측 범위 밖(전원 미달/전원 초과)이면 0/1 클램프로 차트 가장자리에 붙음목표선이 데이터와 겹쳐 보일 수 있음의미상 자연스러움("목표가 전원 밖") — 주석 문서화, 테스트로 고정
min==max(초기 데이터 1건·전원 동률) → 0.5 중립값레이더가 절반 크기로 그려짐데이터 축적 시 자연 해소 — 테스트로 규칙 고정
구 원형 Gauge 삭제가 열린 이슈 #49와 겹침#49 완료 조건 일부가 무의미해짐사용처 전무 + 디자인 개편으로 형태 자체가 대체됨 — PR 본문에 #49 게이지 파트 대체를 명시
recharts 제거후속 이슈에서 복잡한 차트가 필요해지면 재도입 비용사용처 전무 grep 확인 완료, 삭제 확정된 전체 추이 외 수요 없음 — 재도입은 npm install 한 줄
