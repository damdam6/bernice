P2 — 인증 미들웨어 (/api/* 보호, 401 응답 계약 확정)

1요약

모든 /api/* 요청 앞에 세션 쿠키 검증 미들웨어를 세운다. 쿠키 없음·무효·만료면 401 {"error":"unauthorized"}, /api/admin/*에 team 세션이 오면 403 {"error":"forbidden"}으로 응답 계약을 여기서 확정한다(프론트 P3-3 useRecords·관리자 UI가 참조). 예외 경로는 /api/login·/api/health 두 개. 검증 로직은 #40이 만든 functions/lib/session-cookie.ts의 parseCookies+verifySessionToken을 그대로 소비만 한다 — 이 유틸은 처음부터 #42 미들웨어를 소비자로 설계돼 있다.

2현황 분석

  • 토큰 발급은 완료 — functions/api/login.ts:16이 패스코드 검증 후 issueSessionToken+buildSessionSetCookie로 role(team|admin) 세션 쿠키를 발급한다(#41, 머지됨). functions/api/login.ts:5 주석이 "role 검사는 후속 #42 미들웨어 몫"이라고 분업을 명시.
  • 검증 유틸도 완료 — functions/lib/session-cookie.ts:81 verifySessionToken은 서명→만료→role 순 검사, 모든 비정상 입력에 throw 없이 null 반환 (session-cookie.ts:79: "미들웨어(#42)가 try/catch 없이 쓰고 role로 분기하는 것이 계약"). parseCookies(session-cookie.ts:126)와 쿠키명 상수 SESSION_COOKIE_NAME(session-cookie.ts:17)도 준비돼 있다. HMAC 키 캐시(session-cookie.ts:38)가 요청마다 도는 미들웨어 비용을 이미 흡수한다.
  • 보호는 아직 없음 — functions/api/records.ts:2 주석 그대로 "인증 게이트는 P2 범위 — 여기서는 적용하지 않는다". 현재 /api/records는 누구나 호출 가능. functions/_middleware.ts·functions/api/_middleware.ts 모두 부재.
  • 에러 계약 근거 — docs/prd-record-input.html:305-306: 401 unauthorized(세션 없음·만료) · 403 forbidden(팀 세션이지만 관리자 아님). 기존 에러 바디는 전부 {error, message} 형태(login.ts:60, records.ts:40-47).
  • /api/admin/* 엔드포인트는 아직 없다 — 계약(403)만 미리 확정하는 단계. 미들웨어가 /api/* 전체에 걸리므로 admin 라우트가 생기기 전에도 게이트는 즉시 유효하다.

3설계

  • 파일 위치: functions/api/_middleware.ts (이슈 본문의 functions/_middleware.ts에서 의도적 변경). 루트 미들웨어는 Pages 자동 라우팅에 /*를 추가해 정적 자산 요청까지 전부 Function 호출로 끌어들인다. functions/api/ 하위 배치는 라우트가 정확히 /api/*로 잡혀 보호 대상과 실행 범위가 일치하고, 정적 서빙은 기존 무료 경로를 유지한다. 이슈의 기능 요구(/api/* 보호)는 동일하게 충족.
  • 흐름: ① pathname이 예외 경로(/api/login·/api/health 정확 일치)면 next() → ② parseCookies로 bernice_session 추출, verifySessionToken 호출, null이면 401 {error:'unauthorized', message} → ③ /api/admin 또는 /api/admin/*인데 claims.role !== 'admin'이면 403 {error:'forbidden', message} → ④ next().
  • admin 판정: pathname === '/api/admin' || pathname.startsWith('/api/admin/') — /api/adminx 같은 접두 오탐 방지. 판정 전에 pathname을 decodeURIComponent로 정규화 (실패 시 원문 사용) — 라우터가 %2F를 디코드해 매칭하는 경우에도 미들웨어 판정이 더 좁아지는(우회) 일이 없게 한다.
  • 에러 바디: 계약 필드는 error(unauthorized/forbidden), 기존 컨벤션대로 한국어 message 동봉. 401에 Set-Cookie 삭제는 하지 않는다(만료 쿠키는 무해, 로그아웃은 별도 이슈).
  • 기각한 대안: 각 엔드포인트에 검증 헬퍼 호출(신규 라우트마다 누락 위험 — fail-open) · 루트 functions/_middleware.ts에서 pathname 필터링(위의 정적 자산 비용) · 예외 경로 prefix 매칭(/api/login-x 같은 의도치 않은 통과 — 정확 일치가 fail-closed).
  • SESSION_SECRET 미설정: verifySessionToken이 명시적 throw(session-cookie.ts:42) → 500. 설정 누락을 조용한 401로 위장하지 않는 fail-loud가 유틸의 설계 의도이므로 그대로 둔다.
