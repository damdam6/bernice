P3 — 공통 컴포넌트 ③: 랭킹 테이블

damdam6/bernice 브랜치 feat/50-ranking-table ← main 라벨 enhancement 2026-07-17

1요약

shared/domain.ts의 RankingEntry를 표시 기반으로 하는 순수 프레젠테이션 테이블 컴포넌트를 새로 만든다. 종목 1개의 랭킹을 받아 순위·이름·기록·달성 여부를 표로 그리고, 정렬 방향(낮을수록/높을수록)을 화살표로 표시하며, 동순위(1,1,3)와 면제·미측정 선수도 함께 렌더한다. 데이터 fetch나 라우팅은 포함하지 않고 목데이터로만 검증한다(다른 P3 컴포넌트 이슈 #48·#49와 동일한 패턴, 의존성 없음).

2현황 분석

도메인 계약 — shared/domain.ts:70-78의 RankingEntry는 { playerId, name, value, display, rank, achieved }이고, 공동순위는 이미 1,1,3 규칙으로 계산되어 들어온다(functions/lib/compute-rankings.ts:46-62). 단, 이 배열 자체에는 면제·미측정·이상값·비활동 상태 선수가 아예 없다 — computeEventRanking이 활동 상태 + recorded 점수만 후보로 걸러낸 뒤 (functions/lib/compute-rankings.ts:32-42) 랭킹을 매기기 때문이다. 이는 docs/records-schema.html §02·§04(면제·미측정·이상값·활동 외 상태는 랭킹에서 제외, PR #16에서 확정)에 문서화된 백엔드 계약이다.

즉 완료 조건이 요구하는 "면제·미측정이 포함된 목데이터로 렌더"는 백엔드가 주는 EventRanking.entries 배열 그대로는 만들어질 수 없는 상황이다 — 실데이터 조립(어떤 선수가 왜 랭킹에서 빠졌는지)은 이 이슈의 범위 밖(P4 랭킹 페이지에서 players/ sessions와 대조해 처리할 문제)이다. 이 컴포넌트는 표시 전용이므로, RankingEntry를 확장한 로컬 prop 타입을 만들어 recorded·exempt·unmeasured 세 종류의 행을 모두 받아 그리는 것으로 범위를 잡는다(아래 3절).

프론트 현황 — src/에는 아직 App.tsx·main.tsx· index.css뿐이고 src/components/는 없다(그린필드). 디자인 토큰은 src/index.css:5-32에 정의된 brand-*(오렌지)·accent-*(블루)· good/warn/bad 시맨틱 색상이 전부이며 App.tsx:26-36이 이를 쓰는 유일한 참고 예시다. 형제 이슈 #48(카드·뱃지·상태 pill)·#49(게이지·로딩/에러/빈 상태)도 아직 미구현 상태라 이 컴포넌트가 의존할 공용 Badge/Pill이 없다 — "의존성 없음"이므로 필요한 만큼만 컴포넌트 내부에서 인라인으로 처리한다.

테스트 인프라 공백 — vitest.config.ts:1-6은 environment: 'node'뿐이고 React 컴포넌트를 렌더할 jsdom·@testing-library/react가 package.json에 없다(devDependencies에 React 관련 테스트 도구 전무, 순수 함수 테스트인 functions/lib/*.test.ts만 존재). 이 worktree에는 node_modules도 아직 설치돼 있지 않다. 완료 조건의 "목데이터로 올바르게 렌더"를 자동 테스트로 검증하려면 이 devDependencies 추가가 선행돼야 한다.

3설계

선택한 설계 — src/components/RankingTable.tsx에 다음 판별 유니온을 로컬 export 타입으로 정의한다:

  • RankingTableRow = ({ status: 'recorded' } & RankingEntry) | { status: 'exempt' | 'unmeasured'; playerId: number; name: string }
  • Props: { entries: RankingTableRow[]; direction: RankDirection } — 정렬·필터는 컴포넌트 책임이 아니고 호출부가 이미 원하는 순서로(recorded는 rank 오름차순, exempt/unmeasured는 뒤에) 넘긴다고 가정한다(Card/Badge와 동일하게 "순수 표시" 원칙, #48 범위 설명과 일치).

순위 열 — 동순위 표시: 단순히 rank 숫자를 그대로 찍으면 1,1,3 규칙 자체는 맞지만 "동순위 처리"를 시각적으로 증명하지 못한다. 인접 행과 rank가 같은 recorded 행에는 공동 N위, 유일한 순위에는 N위를 렌더한다(계산은 이미 끝난 rank 값을 보고 컴포넌트가 라벨만 결정 — 재계산 없음).

정렬 방향 표시(↑↓): "기록" 열 헤더 옆에 방향 배지를 둔다 — 낮을수록 → ↓ 낮을수록 좋음, 높을수록 → ↑ 높을수록 좋음. 화살표는 "값이 이 방향일 때 좋다"는 의미로 통일(값 자체의 정렬 방향이 아니라 direction 문자열과 1:1 대응) — 별도 유틸 없이 컴포넌트 내부 상수 맵으로 처리.

exempt/unmeasured 행: 순위 열에 면제/미측정 텍스트(회색, 숫자 없음), 기록 열은 –, 달성 열은 빈 칸. shared/domain.ts의 ScoreStatus 라벨과 동일한 한글 문자열을 그대로 재사용해 문서·시트·화면 표기가 어긋나지 않게 한다.

기각한 대안: (a) RankingEntry[]만 받고 exempt/unmeasured는 별도 prop으로 분리 — 두 개의 배열을 순서 맞춰 합치는 책임이 호출부로 넘어가 인터페이스가 더 복잡해져 기각. (b) 백엔드 EventRanking 타입을 그대로 쓰고 exempt/unmeasured를 아예 지원하지 않음 — 완료 조건이 명시적으로 요구하므로 기각. (c) #48의 PlayerStatus pill을 이 컴포넌트가 가져다 쓰기 — #48이 아직 없고 "의존성 없음" 조건과 어긋나 기각(추후 페이지 조립 단계에서 합성 가능).

4작업 단계

  1. 테스트 인프라 추가 — jsdom, @testing-library/react, @testing-library/dom, @testing-library/jest-dom을 devDependencies에 추가하고 npm install 실행. 전역 vitest.config.ts는 그대로 두고 (백엔드 functions/lib/*.test.ts가 node 환경에 의존) 컴포넌트 테스트 파일 상단에 // @vitest-environment jsdom 도크블록으로 해당 파일만 jsdom 전환. package.json
  2. RankingTableRow 판별 유니온과 방향 라벨 상수를 정의 — RankingEntry, RankDirection은 shared/domain.ts에서 import type (verbatimModuleSyntax 준수, tsconfig.app.json:11). src/components/RankingTable.tsx, shared/domain.ts
  3. RankingTable 컴포넌트 본체 구현 — <table> 마크업, 헤더에 방향 배지, 행 매핑에서 recorded/exempt/unmeasured 세 분기 렌더, 동순위(공동 N위) 라벨 로직, entries가 빈 배열이면 "표시할 기록이 없습니다" 인라인 문구(별도 EmptyState 컴포넌트 없이 최소 처리 — #49 범위와 중복 방지). Tailwind는 src/index.css의 brand-*/good 토큰만 사용. src/components/RankingTable.tsx
  4. 목데이터 팩토리 + 렌더 테스트 작성 — functions/lib/compute-rankings.test.ts:5-27의 헬퍼 팩토리 패턴(작은 빌더 함수)을 프론트 버전으로 재현. 케이스: (1) 동순위 2명(rank 1,1) + 단독 3위, (2) 면제 1명, (3) 미측정 1명, (4) 낮을수록/높을수록 방향별 화살표·문구, (5) 빈 배열일 때 안내 문구. @testing-library/react의 render+ screen으로 검증. src/components/RankingTable.test.tsx
  5. npm run typecheck·npm run test 로컬 실행해 기존 functions/lib 스위트(23개 테스트 파일)가 jsdom 전환 영향 없이 그대로 통과하는지 함께 확인. vitest.config.ts

5리스크

리스크영향완화책
이 컴포넌트의 RankingTableRow가 실제 백엔드 RankingEntry[]와 형태가 다름(exempt/unmeasured 포함) P4에서 실데이터를 붙일 때 변환 로직이 추가로 필요, 계약 오해 가능성 타입에 "표시 전용 확장" 주석을 남기고, 이 플랜의 2절 분석을 P4 이슈에서 재확인하도록 완료 보고에 명시
테스트 인프라(jsdom·testing-library)를 이 이슈에서 처음 도입 #48·#49가 같은 걸 각자 다시 추가하면 devDependencies 중복/버전 불일치 가능 표준 패키지(jsdom, @testing-library/react)만 사용해 다른 브랜치와 머지 시 충돌 나더라도 동일 버전으로 쉽게 합쳐지도록 함
동순위 라벨(공동 N위)이 향후 실제 화면 톤과 다를 수 있음 디자인 재조정 필요 가능성 — 다만 로직(어떤 행이 동순위인지)은 재사용 가능 순수 표시 로직과 문자열을 분리해 라벨 문구만 바꾸기 쉽게 구현

6테스트 계획

  • npm install 후 npm run test — 신규 RankingTable.test.tsx가 동순위 2건(공동 1위 라벨 2회 노출)·단독 3위·면제 1건·미측정 1건이 각각 올바른 텍스트로 렌더되는지 확인.
  • 같은 테스트 파일에서 direction: '낮을수록'과 '높을수록' 각각에 대해 헤더의 화살표/문구가 바뀌는지 확인.
  • entries: []일 때 안내 문구가 렌더되고 에러 없이 마운트되는지 확인.
  • 기존 functions/lib/*.test.ts 전체가 jsdom 도입 이후에도 그대로 통과(node 환경 유지 확인 — 회귀 없음).
  • npm run typecheck — RankingTableRow 판별 유니온과 shared/domain.ts 타입이 어긋나지 않는지, verbatimModuleSyntax 하에서 타입 임포트가 올바른지 확인.

7완료 조건

  • 동순위(1,1,3)·면제·미측정이 섞인 목데이터가 각각 올바른 라벨(순위 숫자/공동 표시, "면제", "미측정")로 렌더된다.
  • 종목 방향(낮을수록/높을수록)에 따라 헤더의 화살표·문구가 올바르게 바뀐다.
  • RankingTableRow props 타입이 shared/domain.ts의 RankingEntry/RankDirection과 어긋나지 않는다(확장만 하고 필드 재정의 없음).
  • npm run typecheck·npm run test가 모두 통과한다(기존 백엔드 테스트 회귀 없음 포함).
