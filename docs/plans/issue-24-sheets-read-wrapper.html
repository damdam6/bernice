P1 — Sheets API 읽기 래퍼 (batchGet · 토큰 주입)

1요약

서비스계정 액세스 토큰(functions/lib/googleAuth.ts의 getAccessToken, 이슈 #19)과 탭 분류 유틸(functions/lib/sheetTabs.ts의 classifySheetTabs, 이슈 #18)을 그대로 재사용해, 스프레드시트 메타(탭 목록)를 조회하고 values.batchGet으로 명단·목표·회차 탭의 원시 값을 한 번의 API 호출로 가져오는 순수 IO 레이어를 functions/lib/sheetsApi.ts에 신규 추가한다.

이 레이어는 파싱을 하지 않고 string[][] 원시 2D 배열만 반환하며, 이후 파서 이슈(#25 명단·#26 목표·#27 회차)가 이 결과를 입력으로 받는다. Sheets API의 4xx/5xx 오류는 상태코드를 담은 전용 에러 타입으로 구분해 그대로 전파한다.

2현황 분석

  • functions/lib/googleAuth.ts — 서비스계정 JWT 서명 + scope별 액세스 토큰 캐시 완성 (getAccessToken(env, scope), Env = { GOOGLE_SERVICE_ACCOUNT_KEY }). Workers 런타임 제약(node:crypto 없음)을 이미 Web Crypto로 해결해둔 상태 — 그대로 재사용 대상.
  • functions/lib/sheetTabs.ts:53 — classifySheetTabs(tabNames: string[])가 탭 이름 배열을 받아 { roster, goals, rounds, unclassified }로 분류(NFC 비교·원본 문자열 보존·날짜 오름차순 정렬)까지 완성. 이 이슈는 이 함수의 입력(탭 이름 배열)을 실제 API에서 만들어주는 부분만 담당하면 된다.
  • functions/api/health.ts — 유일한 기존 Functions 엔드포인트, 참고용 최소 예시. /api/records(이슈 #30)는 아직 없음 — 이번 이슈는 그 엔드포인트가 쓸 IO 레이어만 준비.
  • wrangler.jsonc — vars.SHEET_ID가 이미 설정되어 있고(비밀 아님), GOOGLE_SERVICE_ACCOUNT_KEY는 로컬 .dev.vars/프로덕션 Secret으로 분리되어 있음.
  • scripts/seed-sheet.mjs:87-93, scripts/protect-sheet.mjs — 과거 1회성 Node 스크립트에서 sheets.googleapis.com/v4/spreadsheets/{id} 호출 패턴(메타 조회 ?fields=..., :batchUpdate)을 이미 써본 선례가 있음. 다만 node:crypto 기반이라 Workers에 그대로 못 씀 — 엔드포인트 URL 관례만 참고하고 토큰 발급은 googleAuth.ts를 쓴다.
  • functions/lib/normalize-score.ts:12, functions/lib/normalize-status.ts:9 — 둘 다 셀 값을 string | null | undefined로 받는 계약. 이 이슈가 돌려주는 원시 값의 타입이 이 계약과 맞아야 한다(아래 §3-5 참고).
  • 현재 functions/lib/에 Sheets API를 직접 호출하는 코드가 전혀 없음 — 이 이슈가 그 첫 진입점.

3설계

  1. 파일 위치: functions/lib/sheetsApi.ts (+ sheetsApi.test.ts) — googleAuth.ts/sheetTabs.ts와 동일 계층·네이밍 스타일.
  2. 토큰 재사용: import { getAccessToken, type Env } from './googleAuth' — JWT 서명·토큰 캐시 로직은 재구현하지 않고 그대로 호출. scope는 https://www.googleapis.com/auth/spreadsheets.readonly(읽기 전용, 최소 권한) 상수로 고정. 쓰기가 필요한 관리자 기능(P5)은 별도 이슈에서 다른 scope로 자체 처리.
  3. 공통 호출 헬퍼 callSheetsApi(env, path): fetch(`https://sheets.googleapis.com/v4/spreadsheets/${path}`, { headers: { Authorization: `Bearer ${accessToken}` } }) → response.ok가 아니면 SheetsApiError(message, status)를 던짐. 모든 하위 함수가 이 헬퍼를 거치므로 에러 처리가 한 곳에 모임.
  4. 에러 타입: export class SheetsApiError extends Error { constructor(message: string, readonly status: number) }. 4xx(권한 없음·존재하지 않는 범위)와 5xx(구글 쪽 일시 장애)를 status 필드로 호출자가 구분할 수 있게 함 — 이후 /api/records(#30)가 이 값을 보고 응답 코드를 매핑할 수 있음.
  5. 메타 조회 getSpreadsheetTabTitles(env, sheetId): GET {sheetId}?fields=sheets.properties.title → 응답의 sheets[].properties.title을 시트 내 실제 탭 순서 그대로 배열로 반환. 이 배열이 바로 classifySheetTabs의 입력.
  6. 값 일괄 조회 batchGetValues(env, sheetId, ranges): GET {sheetId}/values:batchGet?ranges=A&ranges=B&... (요청 range 개수만큼 ranges 쿼리 파라미터 반복 — 이슈가 요구하는 "일괄 조회"). 구글 API는 응답 valueRanges 순서가 요청 순서와 동일함을 보장하므로 인덱스로 매핑. values가 없는 빈 탭은 []로 정규화.
  7. range 문자열: 탭 이름은 항상 '{name}'처럼 작은따옴표로 감싸고(A1 표기에서 공백·하이픈·숫자 시작 탭 이름은 quoting 필수 — 회차 탭 2025-05-16이 정확히 이 케이스), 이름 내 작은따옴표는 ''로 이스케이프. 셀 범위를 지정하지 않고 탭 이름만 넘기면 Sheets API가 "그 탭 전체"를 반환하는 동작을 활용 — 회차가 늘어도 헤더/컬럼 추가에 코드 변경이 필요 없음.
  8. valueRenderOption 명시 안 함: 기본값(FORMATTED_VALUE)에 의도적으로 의존 — 이 경우 셀 값이 항상 문자열로 내려와 normalizeScore/normalizeStatus(#20)의 string | null | undefined 입력 계약과 그대로 맞음. UNFORMATTED_VALUE로 바꾸면 숫자 셀이 JS number로 내려와 그 계약이 깨지므로, 코드 주석 + 테스트로 이 의존을 명시.
  9. 조합 진입점 fetchSheetBundle(env, sheetId): getSpreadsheetTabTitles → classifySheetTabs(재사용, 무수정) → roster/goals(존재할 때만)와 rounds 전체를 대상으로 range 목록 구성 → batchGetValues 1회 호출 → 아래 SheetRawBundle로 재조립해 반환.
    interface SheetRawTable { name: string; values: string[][] } interface SheetRawBundle { roster: SheetRawTable | null goals: SheetRawTable | null rounds: (SheetRawTable & { date: Date })[] // classifySheetTabs 순서(날짜 오름차순) 유지 unclassified: string[] }
  10. 기각한 대안 1: scripts/seed-sheet.mjs의 api() 헬퍼를 그대로 옮겨오기 — node:crypto 기반이라 Workers에서 동작 안 함. googleAuth.ts가 이미 Web Crypto로 이 문제를 풀어뒀으므로 그 결과물(getAccessToken)만 재사용.
  11. 기각한 대안 2: 탭별로 개별 values.get 호출 — 이슈가 명시적으로 batchGet을 요구하고, 회차가 3개월마다 늘어나는 구조상 탭 수만큼 API 호출이 늘어나는 건 레이턴시·쿼터 낭비.

4작업 단계

  1. functions/lib/sheetsApi.ts 신규 생성 — SheetsApiError 클래스, scope 상수, 내부 callSheetsApi(env, path) 헬퍼(getAccessToken 재사용 + response.ok 체크 + 에러 래핑) 작성. functions/lib/sheetsApi.ts
  2. getSpreadsheetTabTitles(env, sheetId) 구현 — 메타 조회(fields=sheets.properties.title) → 탭 이름 배열(원본 순서) 반환. 응답 형태가 예상과 다르면(예: title 누락) 명확한 에러. functions/lib/sheetsApi.ts
  3. batchGetValues(env, sheetId, ranges) 구현 — values:batchGet 쿼리 조립(quoteSheetName 헬퍼로 각 range escape), 응답 valueRanges를 요청 순서 그대로 매핑, ranges가 빈 배열이면 API 호출 없이 즉시 [] 반환. functions/lib/sheetsApi.ts
  4. fetchSheetBundle(env, sheetId) 구현 — getSpreadsheetTabTitles → classifySheetTabs(sheetTabs.ts, 재사용·무수정) → roster/goals(null이 아닐 때만)+rounds 전체로 range 목록 구성 → batchGetValues 1회 호출 → SheetRawBundle로 재조립. functions/lib/sheetsApi.ts
  5. 단위 테스트 작성 — googleAuth.test.ts의 스텁 패턴(테스트용 RSA 키페어 생성 + vi.stubGlobal('fetch', ...))을 확장해 oauth2 토큰 엔드포인트와 sheets.googleapis.com 엔드포인트를 함께 스텁. 커버 케이스: 정상 조회(roster/goals/rounds 모두 채워짐), 빈 탭(values 없음 → []), 탭 이름에 특수문자/하이픈 있을 때 quoting, roster/goals 미분류(null) 시 해당 range를 요청하지 않음, 4xx(권한 없음)·5xx(서버 오류) 응답이 SheetsApiError.status로 구분되어 전파됨. functions/lib/sheetsApi.test.ts
  6. 로컬 실데이터 검증 — .dev.vars에 실제 GOOGLE_SERVICE_ACCOUNT_KEY를 채운 상태로 임시 디버그 라우트(예: functions/api/_debug-sheets.ts, 커밋하지 않음)에서 fetchSheetBundle(env, env.SHEET_ID)를 호출해 npm run dev:all로 실제 시딩 데이터(선수1~21, 종목 4개, 2025-05-16 회차)가 채워지는지 확인 후 라우트 제거. (임시 파일, 커밋 제외)
  7. npm run typecheck · npm test 전체 통과 확인. (검증만)

5리스크

리스크영향완화책
탭 이름에 하이픈(회차 YYYY-MM-DD)·공백·작은따옴표가 섞여 A1 range 문법이 깨질 수 있음values:batchGet 요청이 400으로 실패해 전체 조회가 막힘quoteSheetName에서 항상 '{name}'로 감싸고 내부 '는 ''로 이스케이프 — 예외 없이 모든 range에 적용
valueRenderOption 기본값(FORMATTED_VALUE)에 대한 암묵적 의존추후 누군가 UNFORMATTED_VALUE로 바꾸면 숫자 셀이 문자열이 아닌 값으로 내려와 #20 정규화 유틸 입력 계약이 깨짐코드 주석으로 근거 명시 + 테스트에서 반환 타입이 문자열임을 명시적으로 검증
회차 탭이 3개월마다 계속 늘어나 ranges 쿼리 파라미터가 많아짐이론상 URL 길이 제한 — 현재 규모(연 4개 내외)에서는 문제 없음지금은 무시, 수십 개 이상으로 늘어나면 청크 분할을 별도 이슈로 분리
roster/goals 탭이 없는(분류 실패) 스프레드시트 상태fetchSheetBundle이 해당 필드에 null을 반환 — 호출자가 체크 없이 쓰면 다운스트림에서 예외타입에 | null 명시해 컴파일 타임에 처리 강제. 이 이슈 범위에서는 에러로 취급하지 않고 그대로 반환(파서 이슈에서 처리할 문제)
