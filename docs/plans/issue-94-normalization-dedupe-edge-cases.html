1요약

두 이슈 모두 normalize-status·normalize-score를 만지므로 이 워크트리에서 순차 처리해 한 PR로 낸다(본문에 Closes #94 · Closes #96). #94는 functions/lib/normalize-status.ts에 중복 정의된 PLAYER_STATUSES/PlayerStatus를 제거하고 shared/domain.ts를 단일 출처로 통일 — seed 스크립트의 하드코딩 드롭다운 값도 domain import로 바꾸고, seconds↔time 용어 대응은 매핑 지점 주석으로 명시한다. #96은 조사 결과 5개 항목 중 3개(NFKC · 분 상한 999 · blank raw 보존)가 이미 PR #22(07442d6)에서 반영·테스트 고정되어 있어, 남은 2개 — 상태 빈칸(blank) vs 이상값(unknown) 구분, 시각 표기(오전/오후·AM/PM) reason 구분 — 만 구현한다.

2현황 분석

#94 — 중복 정의 현황

  • shared/domain.ts:5-6 — PLAYER_STATUSES·PlayerStatus의 정본. 주석(:4)이 "scripts/seed-sheet.mjs와 동일해야 함"이라고 수동 동기화를 요구 중.
  • functions/lib/normalize-status.ts:1-3 — 동일 상수·타입을 그대로 재정의(중복). 이 모듈에서 상수·타입을 import하는 외부 소비자는 없음(roster.ts:2가 normalizeStatus 함수만, 테스트가 StatusValue만 사용) → 재정의 제거 시 파급 없음.
  • scripts/seed-sheet.mjs:176(상태 4종) · :181(방향 2종) — 드롭다운 규칙에 값 하드코딩. Node 24(로컬 v24.11.0)의 TS type-stripping으로 import('./shared/domain.ts')가 동작함을 실증 완료 → 실제 import로 통일 가능. 단 이 스크립트는 재실행 금지(전 탭 리셋, 헤더 경고)라 실행 검증은 하지 않는다.
  • 런타임 시트 생성 경로(functions/lib/create-sheet.ts)와 scripts/protect-sheet.mjs에는 상태·방향 값 정의가 없음(확인) — 정의처는 위 3곳이 전부.
  • 용어 불일치: shared/normalize-score.ts:7의 kind:'seconds'(셀 정규화 결과, 초 단위) vs shared/domain.ts:13-14의 EventValueKind 'time'(종목 기준). 매핑 지점은 이미 존재 — shared/build-event-score.ts:19-21이 seconds↔time·count↔count 교차검증을 수행하고, domain.ts:17-20 주석이 kind→status 대응을 문서화. normalize-score 쪽에는 이 대응을 알려주는 주석이 없다(이슈가 지적한 "매핑 지점 부재"의 잔여분).

#96 — 항목별 현재 상태 (PR #22 07442d6에서 상당 부분 기반영)

  • 기반영 ✓ NFKC — shared/normalize-score.ts:17, functions/lib/normalize-status.ts:10. 테스트 고정: 전각 콜론·전각 숫자(normalize-score.test.ts:23-32), NFD 자모(:18-22, normalize-status.test.ts:12-16).
  • 기반영 ✓ 분 상한 — MAX_MINUTES=999(normalize-score.ts:3,39-41). 경계 테스트: 999:00 유효(:11), 1000:00 invalid(:55).
  • 기반영 ✓ blank raw 보존 — normalize-score.ts:18이 원본 raw 보존(' ' 테스트 :34). null/undefined는 raw:''(:13).
  • 부분반영 시간 자동변환 — 3파트 h:mm:ss는 전용 reason("시트가 시각으로 자동 변환했을 가능성", normalize-score.ts:27-33) + 테스트(:71-77). 잔여: 오전 1:15·1:15 AM 같은 시각 표기는 파트가 정수가 아니어서 일반 reason("mm:ss 형식이어야 함")으로 빠짐 — 진단 구분 없음.
  • 미반영 — 상태 빈칸 vs 이상값: normalizeStatus는 빈칸(미선택)과 이상값(정지)을 모두 unknown으로 접음(normalize-status.ts:15). 소비자 roster.ts:56-64는 rawStatus === ''를 다시 검사해 '상태 없음'/'알 수 없는 상태값'을 구분 — 구분 로직이 정규화기 밖에서 재유도되고 있어 타입이 운영 구분을 표현하지 못함.

3설계

  • [#94] 상수 단일화 방향 — normalize-status.ts가 shared/domain을 import(값 PLAYER_STATUSES + 타입 PlayerStatus). 재export는 하지 않음(외부 소비자 없음 확인, 새 소비자는 domain에서 직접 가져가는 게 규칙). functions/tsconfig.json이 ../shared를 include하므로 typecheck·번들 모두 문제 없음(이미 roster.ts 등이 같은 경로로 import 중).
  • [#94] seed 스크립트 — 하드코딩 값 2곳을 PLAYER_STATUSES.map(v => ({ userEnteredValue: v }))·RANK_DIRECTIONS.map(…)으로 교체하고 ../shared/domain.ts를 import. 헤더의 "의존성 없음 — Node 18+" 주석을 "Node 23.6+(TS type-stripping)"로 갱신. domain.ts:4 주석은 "동일해야 함"(수동 동기화)에서 "seed 스크립트도 이걸 import"로 수정. 기각한 대안: 주석만 상호 참조(드리프트 위험 존치), JSON 별도 파일(출처가 하나 늘어남).
  • [#94] seconds↔time — 런타임 매핑 함수 추가는 불필요: build-event-score.ts가 이미 그 매핑 함수다(#92에서 도입). 남은 것은 표기 명시 — ScoreValue의 'seconds'에 "종목 기준 EventValueKind 'time'과 대응, 교차검증은 build-event-score" 주석, domain.ts EVENT_VALUE_KINDS 주석에 역방향 포인터 추가. 용어 자체는 의도적 유지(셀 정규화 결과의 단위(초) vs 종목 표기 형식으로 의미가 다름).
  • [#96] 상태 blank 구분 — StatusValue에 { kind: 'blank'; raw: string } 추가(트림·NFKC 후 빈 문자열; null/undefined 포함, raw는 원본 보존 — unknown·normalize-score의 blank와 대칭). roster.ts는 rawStatus === '' 재유도 대신 kind === 'blank'로 분기하되 reason 문자열('상태 없음' 등)은 그대로 유지 → RosterRowIssue 계약·기존 roster 테스트 무변경. 기각한 대안: 현행 유지+테스트 고정(소비자가 정규화기 판단을 재유도하는 비대칭이 남음).
  • [#96] 시각 표기 구분 — parseTime 진입 직후 오전/오후/AM/PM 토큰 감지(NFKC 후라 전각 ＡＭ도 접힘) → invalid + "시트가 시각으로 자동 변환했을 가능성" 계열 reason. 유효한 mm:ss는 숫자만이라 오탐 불가. 기각한 대안: h:mm:ss·시각 표기를 시간으로 허용 — 입력 의도(mm:ss)와 시트 자동 변환 결과(시각)를 구별할 수 없어 조용히 틀린 값을 만들 수 있음. 진단(reason) 구분이 안전.
