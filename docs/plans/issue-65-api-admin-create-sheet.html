P5 — POST /api/admin/create-sheet (회차 탭 생성)

damdam6/bernice 브랜치 feat/65-api-admin-create-sheet ← main 라벨 enhancement 2026-07-19

1요약

관리자가 오늘(Asia/Seoul) 회차 탭을 "참가자만 · 가나다 · 빈 점수" 포맷으로 만드는 POST /api/admin/create-sheet 엔드포인트를 신설한다. 탭 생성과 헤더(종목 참조 수식)·이름 참조 수식 기록을 단일 spreadsheets.batchUpdate로 원자적으로 처리해 실패 시 반쪽 탭이 남지 않게 한다. 오늘 날짜 탭이 이미 있으면 409로 거부하고, 성공 시 /api/records 캐시를 무효화한다. 로직 대부분은 순수 함수(create-sheet.ts)로 분리해 fetch 없이 단위 테스트한다. 인증(admin 권한)은 이미 있는 _middleware.ts가 담당한다.

2현황 분석

  • 인증 게이트는 이미 완비. functions/api/_middleware.ts:37이 /api/admin/*를 admin role로 강제하고, 함수 파일이 없어도 403 계약이 유효하다 — 이 엔드포인트는 인증을 다시 구현할 필요가 없다.
  • 쓰기 래퍼는 단일 범위 values.update만 지원. functions/lib/sheetsWriteApi.ts:28의 updateValues는 429·5xx 재시도가 붙은 PUT values 하나뿐 — 탭을 만드는 spreadsheets.batchUpdate 경로가 없다.
  • 읽기 래퍼의 메타 조회는 title만 가져온다. functions/lib/sheetsApi.ts:45의 getSpreadsheetTabTitles는 fields=sheets.properties.title만 요청해 sheetId를 모른다 — 새 탭에 부여할 빈 sheetId를 고르려면 기존 sheetId 목록이 필요하다.
  • 회차 탭 정답 포맷은 "참가자만, 이름 매칭". functions/lib/parse-session.ts:5 주석과 docs/prd-record-input.html §03①이 확정했다 — 회차 탭엔 참가자 행만 존재(생성 시 가나다 정렬), 명단 연결은 행 위치가 아니라 이름 셀(NFC) 매칭이다.
  • 정확한 수식 포맷은 시딩 스크립트가 이미 보여준다. scripts/seed-sheet.mjs:138: 헤더 종목은 ='목표'!A{i+2}, :143: 이름 열은 ='버니스명단'!A{row}. 파서는 FORMATTED_VALUE로 읽어 수식이 해석된 텍스트(종목명·선수명)를 본다.
  • 명단 id ↔ 시트 행 관계가 이름 수식의 열쇠. functions/lib/roster.ts:67에서 player.id = 데이터 행 위치 = 시트 행 − 1이므로, id인 선수의 명단 행은 A{id+1}이다(docs/prd-record-input.html의 playerId: 7 예시와 일치).
  • 캐시 무효화 훅은 상수 하나로 열려 있다. functions/lib/records-cache.ts:7의 RECORDS_CACHE_KEY를 caches.default.delete에 넘기면 /api/records 캐시가 비워진다.

3설계

핵심 결정 — 단일 원자적 batchUpdate. 시딩 스크립트처럼 addSheet 후 별도 values.update를 여러 번 부르면 중간 실패 시 빈 탭이 남는다. 이를 피하려 한 번의 spreadsheets.batchUpdate에 두 요청을 담는다: ① addSheet(내가 고른 명시적 sheetId 부여) → ② 같은 배치의 updateCells가 그 sheetId를 forward-reference해 헤더·이름 수식을 기록. Google이 배치를 원자적으로 처리하므로 실패 시 탭 자체가 만들어지지 않는다.

  • 명시적 sheetId = max(기존 sheetId들) + 1. 같은 배치에서 새 탭을 참조하려면 sheetId를 미리 알아야 한다. 기존 탭 메타를 이미 읽으므로(중복 가드용) 거기서 sheetId도 받아 충돌 없는 값을 결정론적으로 고른다 — 랜덤 sheetId의 희박한 충돌 실패를 원천 배제.
  • 참가자 = 요청 바디의 participantIds(활동 선수 id). 프론트 모달(#67)이 "활동만·가나다·기본 해제"로 고른 선수 id를 보낸다. 이름이 아니라 id로 받는 이유는 /api/records가 이미 id를 쥐고 있고 이름은 개명될 수 있어서다(§06과 동일 원칙). 서버가 명단에서 활동 여부·이름을 다시 확인하는 신뢰 원천이다.
  • 가나다 정렬은 Intl 없이 NFC 코드포인트 비교. 한글 완성형 음절(U+AC00–U+D7A3)은 유니코드 블록 자체가 초성→중성→종성 순, 즉 가나다 순으로 배열돼 있어 NFC 문자열의 코드유닛 비교가 곧 가나다 순이다. Workers 런타임의 불확실한 ICU 로케일에 의존하지 않는다. (동명이인은 id로 안정 정렬)
  • 헤더·이름 모두 참조 수식(literal 아님). 시딩·기존 회차 탭과 동일하게 종목/이름을 참조 수식으로 써서 목표·명단에서 종목명·선수명을 고치면 회차 탭에 자동 반영된다. updateCells의 userEnteredValue.formulaValue로 기록한다.
  • 점수 셀은 안 쓴다(=빈칸). 참가자 행엔 A열(이름 수식) 한 칸만 채우고 B열~ 점수 칸은 생략 → 빈 셀로 남아 "빈 점수" 조건 충족.
  • 순수 함수 분리. 검증·정렬·배치 요청 조립은 create-sheet.ts(순수)로 몰아 fetch 스텁 없이 테스트하고, 엔드포인트는 읽기→빌더→쓰기→무효화 오케스트레이션과 에러 매핑만 담당한다.

배제한 대안: (a) 시딩식 다중 호출 — 원자성 없어 반쪽 탭 위험, 이슈가 명시적으로 금지. (b) 헤더/이름을 literal 문자열로 기록 — 종목·개명 시 desync, 기존 참조 수식 규약과 불일치. (c) 랜덤 sheetId — 희박하나 충돌 시 spurious 실패, 메타를 어차피 읽으므로 이득 없음.

응답 계약(안): 생성 성공 201 {sessionDate, participantCount, participants[]} · 오늘 탭 존재 409 {error:"sheet_already_exists", sessionDate} · 참가자 0명 400 {error:"no_participants"} · 명단에 없거나 비활동 id 400 {error:"invalid_participants", invalidIds[]} · 잘못된 바디 400 {error:"bad_request"} · 명단/목표 탭 없음 500 {error:"missing_roster_tab"|"missing_goals_tab"} · Sheets API 실패 502 {error:"sheets_api_error"}.

5리스크

리스크영향완화책
중복 가드 조회와 쓰기 사이 레이스(다른 관리자가 동시 생성) 드물게 두 번째 요청의 addSheet가 "이미 존재" 4xx 원자적 배치라 반쪽 탭은 절대 안 생김. 사전 가드가 정상 경로의 409를 담당하고, 레이스는 안전하게 실패(4xx→502) 후 재시도. 주석으로 명시.
목표 탭 중간에 빈 행이 있으면 헤더 수식 A{i+2}가 어긋남 회차 탭 읽을 때 파서가 fail-loud 목표는 연속 편집되는 설정 탭(빈 행 없음)이라는 시딩·파서의 기존 전제를 그대로 승계. 시딩 스크립트와 동일한 참조 방식이라 회귀 위험 동일. 코드 주석에 전제 기록.
동명이인이 둘 다 참가자로 선택됨 생성은 되지만 이후 회차 탭 읽기에서 파서가 동명이인 fail-loud 이는 명단 자체 문제(선수5/선수5B로 구분)로 기존 원칙과 동일. 생성 단계에서 막지 않고 id별 이름 수식은 정확한 행을 참조하므로 시트는 올바름.
Pages 중첩 라우팅(functions/api/admin/create-sheet.ts) 라우트 미매칭 시 404 파일 기반 라우팅 규약대로 경로가 /api/admin/create-sheet로 잡힘. 미들웨어의 /api/admin/* 가드 테스트가 이미 이 경로 계약을 커버.

6테스트 계획

  • seoul-date: UTC 15:00(=KST 익일 00:00) 경계에서 날짜가 하루 넘어가는지, 월말/연말 넘김, 한 자리 월/일 zero-pad.
  • getSpreadsheetSheets: sheetId+title 매핑, fields 쿼리에 sheets.properties(sheetId,title) 포함, 특수문자 sheetId 인코딩, 4xx/5xx → SheetsApiError.
  • batchUpdate: 정상 1회 성공(POST :batchUpdate·write scope·requests 바디), 429 1회 후 복구, 4xx 즉시 실패, 재시도 소진 실패.
  • buildCreateSheetPlan(순수, 핵심): 참가자만·가나다 정렬·이름 수식 ='버니스명단'!A{id+1}·헤더 ='목표'!A{i+2}·점수 칸 없음·sheetId=max+1· frozen 1/1 / 0명 → no_participants / 미등록·비활동 id → invalid_participants / 중복 id dedupe / 동명이인 안정 정렬.
  • 엔드포인트: 201 성공(batchUpdate 호출·응답 바디·캐시 delete 호출) / 오늘 탭 존재 → 409(쓰기 미호출) / 비활동 id → 400 / 명단·목표 없음 → 500 / 명단 헤더 깨짐 → 500 sheet_data_invalid / SheetsApiError → 502 / 잘못된 바디 → 400. (fetch·caches 스텁은 records.test.ts/sheetsWriteApi.test.ts 패턴 재사용)
  • 회귀: npm test 전체 green + npm run typecheck — 기존 읽기/파서 테스트가 건드린 sheetsApi.ts 수정으로 깨지지 않는지 확인.

7완료 조건

  • 참가자만 · 가나다 정렬 · 빈 점수 · 이름/종목 참조 수식 회차 탭이 단일 원자적 batchUpdate로 생성된다.
  • 오늘(KST) 탭이 이미 있으면 생성하지 않고 409 sheet_already_exists로 거부하는 응답 계약이 성립한다.
  • 생성 성공 시 /api/records 캐시(RECORDS_CACHE_KEY)가 무효화된다.
  • 참가자 0명·미등록·비활동 id·잘못된 바디에 대한 400 계약이 명확하다.
  • vitest 단위·엔드포인트 테스트가 모두 통과하고 typecheck가 green이다.
