📲 PRD — 사이트에서 기록 입력

측정일에 관리자가 폰으로 사이트에서 직접 점수를 입력·수정하는 기능. 시트는 계속 데이터 원천(SoT)이고, 사이트는 시트에 쓰는 클라이언트다.

01배경 & 문제 정의

현재 사이트는 읽기 전용이다 — /api/records가 시트를 파싱해 랭킹·추이·PB를 계산해 보여줄 뿐, 기록을 넣는 방법은 구글 시트 직접 편집뿐이다. 측정일(3개월마다)의 실제 장면은 "체육관에서 폰으로 구글 시트 앱을 열어 좁은 셀에 점수를 타이핑"이고, 여기서 반복되는 문제가 있다:

#문제결과
1폰에서 시트 편집이 불편 — 셀이 작고, 확대/축소·행 찾기에 시간이 든다측정 진행이 끊기고 입력 누락이 생김
2이름·종목 헤더가 참조 수식인데 실수로 덮어쓸 수 있다 (편집 규칙 §01 절대 규칙)명단↔회차 연결이 끊어져 복구 작업 필요
3형식 오류가 조용히 들어간다 — 1:75, 시각 자동 변환(오전 1:15), 소수 입력 등파서가 invalid로 분류해 랭킹에서 빠지는데, 그 사실을 나중에야 발견
4입력해도 사이트 반영은 캐시 TTL(6시간)까지 지연 (수동 새로고침 /api/refresh는 P2 예정)측정 직후 팀원들이 결과를 바로 못 봄

목표: 측정일에 시트 앱을 한 번도 열지 않고, 관리자가 폰으로 회차 생성 → 참가자 선택 → 선수별 점수 입력 → 팀원 즉시 열람까지 끝낼 수 있게 한다. 입력 경로가 사이트를 거치면 검증된 값만 들어가고(형식 오류 원천 차단), 참조 수식을 건드릴 일이 없으며, 저장 즉시 캐시가 무효화돼 바로 반영된다.

포지셔닝: 이 기능은 작업 계획 P5 관리자 페이지(쓰기)의 확장이다. P5가 계획한 회차 탭 생성(/api/admin/create-sheet)·참가자 추가(/api/admin/add-players)에 점수 입력·수정을 더해, 관리자 페이지에서 측정일 운영이 완결되게 한다.

02확정 결정 요약

기존 확정 (뒤집지 않음)

#항목결정
A데이터 원천시트 = SoT 유지. 사이트는 시트에 쓰는 클라이언트. 별도 DB 도입 없음. 시트 직접 편집(수동 정정)은 계속 허용되는 escape hatch
B회차 탭 포맷"참가자만, 이름 매칭" — 회차 탭에는 그 회차 참가자 행만 존재(생성 시 가나다 정렬), 명단과의 연결은 위치가 아니라 이름 매칭. §03 선행 의존성 ① 참고
C점수 검증기존 normalize-score 규칙 재사용 — 새 검증 규칙 발명 금지 (§08)
D캐시쓰기 성공 시 records 캐시 무효화 — P5에 계획된 패턴 재사용 (§09)

이번 인터뷰로 확정 (2026-07-17)

#항목결정
1입력 권한관리자만 입력. ADMIN_CODE 세션만 쓰기 API 허용. 팀 패스코드 세션은 열람 전용 유지. 측정 진행자(운영진)가 기록하는 현행 운영 방식과 일치
2입력 동선선수 선택 → 종목 타입별 기입. 참가자 목록에서 선수를 고르면 그 선수의 전 종목을 타입별 입력기로 채우는 선수 중심 화면
3저장 단위선수 단위. 저장 1회 = 회차 탭에서 그 선수 행의 점수 셀 전체를 요청 값으로 교체(행 1개 업데이트). 화면 동선과 저장 단위가 1:1
4수정·삭제자유 수정. 관리자는 언제든(과거 회차 포함) 같은 입력 화면에서 덮어쓰기·빈칸(미측정) 되돌리기 가능. 시트 직접 편집과 동등한 능력이라 별도 잠금·이력 없음
5동시 입력 충돌마지막 저장 승리. 저장 단위가 행이라 서로 다른 선수 입력은 충돌 자체가 없고, 같은 선수 행을 동시에 고칠 때만 나중 저장이 이김 — 시트 수동 편집과 같은 의미론
6쓰기 실패자동 재시도 후 에러. 429·5xx 계열만 짧은 백오프로 1~2회 자동 재시도. 그래도 실패하면 입력값을 보존한 채 에러 표시 + 수동 재시도 버튼. 캐시 무효화는 쓰기 성공 후에만
7모바일 입력기종목 타입별 맞춤 입력기. 시간 종목(셔틀런)은 분:초 2필드 숫자 키패드, 개수 종목은 숫자 키패드, 면제는 토글, 빈칸 = 미측정

※ 인터뷰 과정 기록: 저장 단위는 처음 "종목별 저장"이 선택됐으나, 입력 동선을 "선수 선택 → 타입별 기입"으로 확정하면서 선수 단위 저장으로 대체했다(동선과 저장 단위 1:1). 종목 단위 일괄 기입 화면은 비스코프(§10) — 필요해지면 후속 이슈.

03전제 & 선행 의존성

① 회차 파서 이름-매칭 정합화 (선행 확정 — 이 PRD 범위 밖, 별도 이슈)

현재 functions/lib/parse-session.ts는 "회차 탭 행 위치 = 명단 순서(전원 미러링)"를 가정한다(데이터 행 N → playerId = N). 그러나 실제 회차 탭(2025-05-16)은 참가자 6명만 가나다순으로 수기 입력돼 있고, P5 create-sheet도 같은 "참가자만" 포맷을 생성하도록 계획돼 있다 — 파서 가정과 실데이터·계획이 모순이다.

확정된 해소 방향: 회차 탭 포맷은 "참가자만, 이름 매칭"이 정답이고, 파서를 이름 매칭으로 바꾸는 정합화가 이 기능의 선행 작업이다. 이 PRD는 그 포맷을 전제로 쓴다. 파서 정합화 이슈에는 다음이 포함돼야 한다:
  • 회차 탭 각 행의 이름 셀(NFC 정규화)을 버니스명단 이름과 매칭해 playerId 도출 — 행 위치 가정 제거
  • 명단에 없는 이름·명단 내 중복 이름(동명이인 미구분)은 기존 fail-loud 원칙대로 에러
  • Session.entries는 "그 회차 탭에 실제로 존재하는 행만" — 참가자만 포맷과 자연스럽게 일치 (shared/domain.ts의 주석이 이미 이 의미)
  • 시트 연동 확정안 §02의 "명단 전원 미러링" 서술 등 관련 문서 갱신

② P2 로그인 + 관리자 세션 구분 (선행 — 이슈 #41·#42 + 관리자 role)

functions/lib/session-cookie.ts(HMAC 서명 토큰 유틸)는 이미 있으나 POST /api/login(#41)·인증 미들웨어(#42)는 미구현이고, 현행 v1 토큰 페이로드는 {iat, exp}뿐이라 팀/관리자를 구분할 클레임이 없다. 이 PRD의 쓰기 API는 "관리자 세션"을 요구하므로 다음 중 하나가 선행돼야 한다 (P5 1번 항목 "관리자 코드 검증 유틸"과 같은 자리에서 결정):

  • 권장: /api/login이 입력 코드가 TEAM_PASSCODE인지 ADMIN_CODE인지에 따라 토큰 페이로드에 role: 'team' | 'admin' 클레임을 넣어 발급 — 쿠키 1개, 미들웨어에서 role 검사
  • 대안: 관리자용 별도 쿠키/토큰 — 쿠키 2개 관리 부담이 있어 차선

③ P5 create-sheet · add-players (같은 마일스톤에서 병행)

점수 입력 흐름은 "입력할 회차 탭이 존재하고, 선수가 그 회차 참가자로 등록돼 있다"를 전제한다. 회차 탭 생성(참가자 선택 모달 → 가나다 정렬 → 점수 비움)과 늦은 참가자 추가(맨 아래 추가)는 P5에 이미 계획된 기능을 그대로 쓴다 — 이 PRD가 재설계하지 않으며, 함께 구현돼야 입력 E2E가 완성된다.

04유저 플로우

측정일 시나리오 (주 시나리오 — 체육관, 폰)

관리자 로그인 (관리자 코드 입력 → role: admin 세션) │ ▼ 관리자 페이지 — 오늘 회차 탭 없으면: [기록지 만들기] → 참가자 선택 모달(활동만·가나다·기본 해제) → 회차 탭 생성 ← P5 create-sheet │ ▼ 참가자 목록 — 가나다 정렬 · 선수마다 입력 상태 표시 (미입력 / 일부 / 완료) │ 선수 탭 ▼ 선수별 입력 화면 — 전 종목을 타입별 입력기로 기입 시간(셔틀런): [분]:[초] · 개수: 숫자 키패드 · 면제: 토글 · 빈칸 = 미측정 │ [저장] ▼ 저장 = 회차 탭 그 선수 행 1개 업데이트 → 성공 시 records 캐시 무효화 │ 목록 복귀 (다음 미입력 선수로 이동 유도) ▼ (늦은 합류자) [참가자 추가] → 회차 탭 맨 아래 추가 → 목록에 등장 → 점수 입력 ← P5 add-players │ ▼ 전원 완료 — 팀원은 열람 패스코드로 랭킹·추이 즉시 확인

정정 시나리오 (자유 수정)

관리자 페이지 → 회차 선택 (최신·과거 모두) → 참가자 목록 → 선수 선택 → 기존 값이 프리필된 입력 화면 → 값 덮어쓰기 / 비우기(미측정 되돌리기) / 면제 전환 → [저장] — 행 전체가 요청 값으로 교체 (마지막 저장 승리)

"삭제"는 별도 동작이 아니라 값을 비우고 저장(= 미측정)이다. 회차 탭 자체의 삭제·날짜 변경, 참가자 행 제거는 비스코프(§10) — 시트에서 직접 한다.

실패 시나리오

[저장] → Sheets API 429·5xx → 자동 재시도 1~2회 (짧은 백오프) ├─ 성공 → 정상 흐름 (사용자는 지연만 체감) └─ 소진 → 에러 표시 + 입력값 보존 + [다시 저장] 버튼 · 캐시는 건드리지 않음 4xx(검증 실패·권한·회차 없음)는 재시도 없이 즉시 사유 표시

05모바일 입력 UX

주 사용 환경은 체육관에서 폰이다. 화면은 두 개뿐이다: 참가자 목록 → 선수별 입력.

참가자 목록 화면

  • 그 회차 참가자를 가나다 정렬로 표시 (add-players로 뒤에 추가된 선수도 사이트에선 정렬해 표시 — 탭 내 물리 순서와 무관)
  • 선수마다 입력 상태 뱃지: 미입력(전 종목 미측정) / 일부 / 완료(전 종목이 기록·면제 중 하나)
  • 저장 후 목록 복귀 시 다음 미입력 선수로 빠른 이동 동선 제공 — 20명 연속 입력을 최소 탭으로
  • [참가자 추가] 버튼 (P5 add-players 재사용)

선수별 입력 화면 — 종목 타입별 입력기

종목 타입입력기비고
시간 (valueKind: time — 드리블셔틀런)분·초 2필드, inputmode="numeric" 숫자 키패드, 분 입력 후 초로 자동 포커스초 필드는 0–59만 — 1:75 같은 값이 UI에서 성립 불가
개수 (valueKind: count — 골밑슛·자유투·45도패스캐치)숫자 키패드 단일 필드 (0 이상 정수)만점(maxScore)은 힌트로만 표시(예: "/ 10") — 초과 입력을 막지 않음 (§08)
면제토글 — 켜면 값 필드 비활성화시트에는 면제 리터럴로 저장
미측정빈칸으로 두면 됨 (별도 조작 없음)시트에는 빈 셀 — 기존 값 지우기 = 필드 비우고 저장
  • 기존 값이 있으면 프리필 (자유 수정과 같은 화면)
  • 저장 버튼은 하단 고정, 터치 타깃 크게 — 장갑·땀 손 환경 고려
  • 형식 오류(있을 수 없는 값)는 필드 인라인으로 즉시 표시 — 제출 후에야 아는 일이 없게
  • 저장 성공 시 짧은 확인 표시 후 목록 복귀. 실패 시 입력값 보존 + 재시도(§04 실패 시나리오)
타입별 입력기의 목적은 normalize-score를 통과할 값만 UI에서 만들어지게 하는 것이다 — 시각 자동 변환·1:75·소수 같은 기존 시트 직접 입력의 사고 유형이 입력기 수준에서 사라진다. 그래도 서버 검증(§08)은 별도로 최종 게이트로 존재한다.

06API 엔드포인트 설계 초안

경로·메서드·요청/응답 개요 수준 — 상세 스키마·에러 바디 확정은 구현 이슈 몫. 모든 /api/admin/*은 관리자 세션(§03 ②) 필수.

엔드포인트역할출처
POST /api/admin/records선수 단위 점수 저장 (이번 PRD 신규)신규
POST /api/admin/create-sheet회차 탭 생성 (참가자 선택·가나다·점수 비움, 오늘 KST·중복 가드)P5 기존 계획
POST /api/admin/add-players대상 회차에 활동 선수 맨 아래 추가P5 기존 계획

POST /api/admin/records — 선수 단위 저장

의미론: 그 회차 탭에서 그 선수 행의 점수 셀 전체를 요청 값으로 교체(행 단위 upsert, 부분 업데이트 아님). 화면이 전 종목을 한 번에 보여주므로 요청도 전 종목을 담는다 — 마지막 저장 승리(§02 결정 5)와 의미가 정확히 일치한다.

07시트 쓰기 방식

쓰기 파이프라인 (저장 1회)

1. 액세스 토큰 — 기존 googleAuth.getAccessToken(env, scope) 재사용, scope만 읽기·쓰기용 https://www.googleapis.com/auth/spreadsheets로 (sheetsApi.ts는 readonly scope 고정 — "쓰기는 P5가 별도 scope로 자체 처리" 주석 그대로) 2. 회차 탭 읽기 — 헤더 행 + 이름 열(A열) 1회 조회 ├─ 헤더 → 종목 → 열 매핑 (parse-session의 mapHeaderToEvents와 같은 규칙 — 열 순서 가정 없음) └─ 이름 열 → playerId의 명단 이름과 NFC 매칭 → 대상 행 번호 결정 (없으면 404 not_participant) 3. 값 검증 — normalize-score + valueKind 교차검증 (§08) · invalid면 여기서 400, 시트에 안 씀 4. values.update 1건 — 그 행의 점수 셀 범위(예: '2025-08-16'!B7:E7), valueInputOption=RAW 5. 429·5xx면 짧은 백오프로 1~2회 재시도 (update는 멱등 — 같은 값 재쓰기 무해) 6. 성공 → records 캐시 무효화 (§09) → 200 응답

설계 근거

  • 쓰기 단위 = values.update 1건. 선수 단위 저장은 항상 단일 행의 연속 범위라 values.batchUpdate(다중 범위)가 필요 없다. 원자적 spreadsheets.batchUpdate(탭 복제·구조 변경)는 P5 create-sheet만의 몫 — 점수 저장은 구조를 건드리지 않는다.
  • valueInputOption=RAW. USER_ENTERED는 구글이 1:15를 시각으로 자동 해석할 수 있다 — 시트 직접 입력에서 이미 알려진 사고 유형(편집 규칙 §04). RAW는 문자열 그대로 저장돼, 읽기 쪽 FORMATTED_VALUE(문자열 계약)와 왕복이 일관된다.
  • 이름 열(A열)·헤더에는 절대 쓰지 않는다. 참조 수식 보호 — 쓰는 범위는 점수 셀뿐. 편집 규칙 §01("사람이 채우는 건 점수 숫자뿐")을 API도 동일하게 지킨다.
  • 행 찾기 ↔ 쓰기 사이 레이스: 회차 탭은 append-only(add-players는 맨 아래 추가만)라 기존 행 번호가 밀리지 않는다 — 조회한 행 번호로 그대로 써도 안전. 회차 탭 내 행 삭제·재배치는 시트 편집 규칙으로 금지돼 있고, 벌어졌다면 파서가 fail-loud로 드러낸다(기존 원칙).
  • 동시 저장은 마지막 승리(§02 결정 5). 서로 다른 선수 = 서로 다른 행이라 충돌 없음. 같은 선수 행 동시 저장만 나중 요청이 덮어씀 — 잠금·버전 비교 없음.
  • 재시도는 서버(Pages Function) 안에서. 클라이언트 재전송보다 중복·순서 문제가 없고, 429·5xx에만 적용(4xx는 즉시 실패). 대상이 단일 행 전체 교체 update라 재시도가 항상 멱등.

08검증 규칙 — normalize-score 재사용

검증 규칙의 원천은 functions/lib/normalize-score.ts 하나다. 새 규칙을 만들지 않는다. 서버가 최종 게이트, 클라이언트는 같은 규칙의 선반영일 뿐이다.

입력normalize-score 결과서버 처리
1:12 (시간 종목)seconds저장 (recorded)
6 (개수 종목)count저장 (recorded)
면제exempt저장 (면제 리터럴)
"" (빈 문자열)blank저장 (빈 셀 = 미측정)
1:75 · 소수 · 음수 · h:mm:ss …invalid400 거부 — reason 그대로 반환, 시트에 쓰지 않음
1:15 (개수 종목에)seconds ≠ valueKind400 거부 — valueKind 교차검증 (records-schema §03 규칙 그대로)
  • 사이트 경로로는 invalid가 시트에 들어갈 수 없다. invalid는 앞으로도 시트 직접 편집(escape hatch)에서만 생길 수 있고, 파서·화면의 기존 invalid 처리(원본 보존 + reason)는 그대로 유지된다.
  • 만점(maxScore) 초과는 막지 않는다. normalize-score에 없는 규칙이라 발명하지 않음 — 시트 직접 입력과 동일 의미론. UI는 만점을 힌트로만 표시.
  • 클라이언트 검증: 타입별 입력기(§05)가 형식 오류를 원천 차단하고, 제출 전 같은 규칙으로 인라인 피드백. 프론트가 규칙을 재사용할 수 있도록 normalize-score를 shared/로 승격하는 것을 구현 이슈에서 검토한다(순수 함수라 이동 비용 없음).

09캐시 무효화

  • 쓰기 성공 시에만 caches.default.delete(RECORDS_CACHE_KEY) — functions/lib/records-cache.ts의 키를 그대로 import (P2 /api/refresh·P5와 동일 패턴).
  • 무효화는 응답을 돌려주기 전에 await — 프론트가 저장 직후 refetch하므로, 삭제가 응답보다 늦으면(waitUntil) 오래된 캐시를 다시 받을 수 있다.
  • 프론트는 저장 성공 콜백에서 TanStack Query의 records 쿼리를 invalidate → 다음 조회가 시트를 재조립. 저장마다 전체 재조립이 일어나지만 팀 규모(약 21명 × 회차 수)에서 비용은 무시 가능.
  • 쓰기 실패(재시도 소진 포함) 시 캐시는 건드리지 않는다 — 실패했는데 무효화만 하면 불필요한 재조립만 생긴다.
알려진 한계 (기존 패턴 그대로 상속): Cloudflare Cache API는 데이터센터(콜로) 로컬이라 cache.delete는 요청이 닿은 콜로에서만 지워진다. 팀 전원이 같은 지역(한국)에서 접속하는 실사용에선 문제가 안 되지만, 다른 콜로에 남은 캐시는 TTL(6시간)까지 살 수 있다. P2 /api/refresh와 공유하는 한계이며 이 PRD에서 새로 해결하지 않는다.
