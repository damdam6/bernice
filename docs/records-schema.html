01공용 타입 위치

src/(프론트)도 functions/(Pages Functions)도 소유하지 않는 최상위 shared/ 폴더에 domain.ts 하나를 둔다. tsconfig.app.json의 include에 "shared"를, functions/tsconfig.json의 include에 "../shared"를 추가해 두 TS 프로그램이 대칭적으로 같은 파일을 본다. 값 export는 작은 상수 배열 몇 개뿐이고 대부분 타입/인터페이스라 번들 크기에 영향이 없다.

파서 쪽에서 src/를 참조하거나 프론트가 functions/를 참조하는 방식은 레이어 역전을 만들어 기각했다. shared/는 어느 쪽에도 속하지 않는 중립 위치.

02상태 4종 정책의 응답 반영

PlayerStatus는 버니스명단 드롭다운 문자열(활동/탈퇴/비대상/휴식)을 번역 없이 그대로 리터럴 유니온으로 쓴다. 정책은 타입이 아니라 응답을 구성하는 규칙으로 반영된다:

상태players[]과거 sessions[].entriesrankings / home.achievementRates
탈퇴응답 어디에도 등장하지 않음 — 파서 단계에서 완전히 제외 (사이트 전체 숨김). PlayerSummary.status는 Exclude<PlayerStatus, '탈퇴'>로 타입에서도 강제
활동포함포함집계 대상
비대상 · 휴식포함(상태 뱃지용 status)포함(프로필에서 과거 기록 열람 — trends/personalBests도 정상 채움)집계 제외
"탈퇴 = 사이트 전체 숨김"은 과거 회차 기록까지 포함해 완전히 제외하는 것으로 해석했다(이슈 #21 리뷰에서 확정). docs/sheet-rules.html의 "과거 기록은 보존되고 현재 랭킹에서만 빠진다"는 문구는 이 결정에 맞춰 갱신했다 — 시트 자체의 데이터는 보존되지만, 사이트 응답(/api/records)에서는 완전히 걸러진다.
⚠️ 소급 효과(리뷰에서 지적) — 반드시 인지하고 수용할 것. 상태는 "현재값 하나"만 저장되고 회차별 이력이 없다. 그래서:
  • 누군가 탈퇴하면 과거 회차의 랭킹도 그 사람을 뺀 채로 재계산된다 — 남아있는 선수들의 그 당시 등수가 지금 시점 조회 시 올라갈 수 있다(예: 2등이었는데 1등이 탈퇴하면 지금 보면 1등으로 보임).
  • 같은 이유로 지금 휴식·비대상인 선수는 그 사람이 실제로 활동 중이었던 과거 회차 랭킹에서도 빠진다 — "그 당시엔 활동이었다"는 이력이 없어 불가피함.
즉 rankings·home.achievementRates는 "그 회차 당시의 스냅샷"이 아니라 "응답 생성 시점 상태 기준으로 매번 재계산"이다. 과거 스냅샷 고정이 필요해지면 별도 이슈로 상태 이력을 추가해야 한다.

모든 점수 칸은 판별 유니온 EventScore로 표현한다 — status로 분기하면 value/display를 null 체크 없이 쓸 수 있다.

회차 전체 미참여는 별도 상태값이 아니라 SessionEntry.participated(모든 종목이 unmeasured면 false)로 파생 표현한다. exempt·invalid는 "무언가 입력됨"으로 보아 참여로 친다. EventDefinition.valueKind('time' | 'count')는 목표 탭에 별도 열이 없어 파서가 셀의 콜론 포함 여부로 판별해 명시적으로 싣는다.

정규화 유틸(이슈 #6 / functions/lib/normalize-score.ts)과의 매핑

시트 원값을 이 계약의 EventScore로 옮기는 실제 정규화 로직은 이슈 #6에서 별도로 구현된다. 그 함수의 ScoreValue.kind(셀 표기만 보고 판별 — 콜론 있으면 시간)와 이 문서의 ScoreStatus(종목 정의까지 고려)는 레이어가 달라 이름도 다르다. 매핑:

ScoreValue.kind(셀 기준, 이슈 #6)→ScoreStatus(이 계약)
count→recorded
seconds→recorded
exempt→exempt
blank→unmeasured
invalid→invalid
⚠️ valueKind와 셀 판별의 불일치는 여기서 invalid로 승격한다. normalizeScore는 셀 표기만 보고 독립적으로 판별하므로(콜론 있으면 무조건 seconds), 개수 종목(valueKind: 'count') 셀에 실수로 1:15를 입력하면 { kind: 'seconds', value: 75 }가 반환된다. 이걸 그대로 recorded로 실으면 개수 종목 랭킹에 초 단위 값이 섞이는 오류가 생긴다. 규칙: EventScore를 조립하는 단계에서 EventDefinition.valueKind와 정규화된 kind(count↔seconds)가 일치하는지 확인하고, 불일치하면 invalid로 처리(reason 예: "종목 형식과 값 형식이 다름"). 이 교차 검증은 조립 단계(파서/엔드포인트 이슈)의 책임 — normalizeScore 자신은 종목 정보를 모르므로 여기서 할 수 없다.

04랭킹·달성·추이 규칙

동점자 처리

표준 공동순위(1, 1, 3 — 동점자는 같은 등수, 다음 등수는 동점자 수만큼 건너뜀)로 확정(PR #16 리뷰 승인). 두 가지 세부 규칙:

  • 동점 판정은 정규화된 value 기준 — display 문자열을 비교하지 않는다.
  • 건너뛰기 계산은 EventRanking.entries에 실제로 포함된 엔트리(활동 상태 + recorded)만 대상으로 한다 — 제외된 선수는 순번에 영향을 주지 않는다.

달성(achieved) 판정 — 경계값 포함

목표치와 정확히 같아도 달성으로 처리한다: 낮을수록 종목은 value <= targetValue, 높을수록 종목은 value >= targetValue. RankingEntry.achieved·TrendPoint.achieved·PlayerPersonalBest.achieved 전부 동일 규칙.

달성률(rate) — 0/0 처리

EventAchievementRate.eligibleCount가 0이면(예: 그 회차에 그 종목이 전원 면제) rate는 0. achievedCount·eligibleCount 둘 다 활동 상태 + 최신 회차(home.latestSession) 기준.

추이 변화(deltaFromPrevious / improved)

deltaFromPrevious는 직전 유효 기록 대비 원값 그대로의 부호 있는 차이(시간 종목은 초, 개수 종목은 개수)다 — 이것만으로는 "좋아졌는지"를 알 수 없다(낮을수록 종목은 감소가 개선, 높을수록 종목은 증가가 개선). 그래서 direction까지 반영해 미리 계산한 improved: boolean | null을 함께 내려준다 — 프론트는 direction과 조합할 필요 없이 improved만 보고 ▲▼를 그리면 된다. 둘 다 첫 기록이면 null.

참여 인원(home.latestSession.participantCount)

활동 상태 선수 중 최신 회차에서 participated: true인 인원만 센다 — 비대상·휴식은 그 회차에 기록이 남아 있어도 제외(rankings·achievementRates와 동일 범위로 통일).

05타입 정의 전문

실제 소스: shared/domain.ts. 아래는 그 내용 그대로.

// shared/domain.ts
// 버니스 실력 기록 — 공용 도메인 타입. 파서·계산(functions)과 프론트(src)가 함께 import.

// ── 선수 상태 — 버니스명단 상태 드롭다운 4종 ──
export const PLAYER_STATUSES = ['활동', '탈퇴', '비대상', '휴식'] as const
export type PlayerStatus = (typeof PLAYER_STATUSES)[number]

// ── 종목 정렬 방향 — 목표 탭 방향 드롭다운 ──
export const RANK_DIRECTIONS = ['낮을수록', '높을수록'] as const
export type RankDirection = (typeof RANK_DIRECTIONS)[number]

// ── 점수 값 형식 — 셀 표기로 파서가 판별(콜론 포함 = 시간) ──
export const EVENT_VALUE_KINDS = ['count', 'time'] as const
export type EventValueKind = (typeof EVENT_VALUE_KINDS)[number]

// ── 개별 점수 상태 — 정상 기록 · 면제 · 미측정 · 이상값 ──
// functions/lib/normalize-score.ts(이슈 #6)의 ScoreValue.kind → 이 상태로 매핑:
//   count/seconds → recorded · exempt → exempt · blank → unmeasured · invalid → invalid
// EventDefinition.valueKind(종목 기준)와 정규화된 kind(셀 기준)가 어긋나면(예: 개수 종목 셀에
// "1:15" 입력) invalid로 처리 — 상세 규칙은 §03 참고.
export const SCORE_STATUSES = ['recorded', 'exempt', 'unmeasured', 'invalid'] as const
export type ScoreStatus = (typeof SCORE_STATUSES)[number]

/** 선수 — 버니스명단 1행 = 선수 1명.
 *  id는 시트 열이 아니라 행 위치 기반 파생값(1부터, 헤더 제외).
 *  행 순서 = 신원(정렬·삭제 금지)이라 재계산해도 항상 안정적. */
export interface Player {
  id: number
  name: string
  status: PlayerStatus
}

/** 종목 정의 — 목표 탭 1행 = 종목 1개. 달성 판정은 direction 기준 경계값 포함(같아도 달성). */
export interface EventDefinition {
  key: string
  valueKind: EventValueKind
  target: string            // 목표치 원본 표시값 ("1:17" | "5")
  targetValue: number        // 정규화 목표치(시간=초, 개수=그대로)
  maxScore: number | null    // 만점(표시용 분모). 없으면 null
  direction: RankDirection
}

/** 회차 1건, 종목 1개의 점수 한 칸 — status로 분기하면 null 체크 없이 쓸 수 있다.
 *  invalid는 이상값이 실제로 입력된 경우로 display에 원본을 보존, reason에 사유. */
export type EventScore =
  | { status: 'recorded'; value: number; display: string }
  | { status: 'exempt' | 'unmeasured'; value: null; display: null }
  | { status: 'invalid'; value: null; display: string; reason: string }

/** 회차 1건에서 선수 1명의 기록 행. */
export interface SessionEntry {
  playerId: number
  name: string
  scores: Record<string, EventScore>   // events[] 전체 key가 항상 존재(누락 없음)
  participated: boolean                   // 전 종목 unmeasured면 false. exempt·invalid는 참여로 침
}

/** 회차 1개 = 날짜 탭 1개. */
export interface Session {
  date: string
  entries: SessionEntry[]
}

/** 종목별 랭킹 1행. */
export interface RankingEntry {
  playerId: number
  name: string
  value: number
  display: string
  rank: number        // 공동순위(정규화 value 기준, 1,1,3) — 활동+recorded 엔트리만 대상으로 건너뜀
  achieved: boolean
}

export interface EventRanking {
  event: string
  entries: RankingEntry[]   // 면제·미측정·이상값·활동 외 상태 제외
}

/** 회차 1개에 대한 종목별 랭킹 묶음. */
export interface SessionRankings {
  sessionDate: string
  events: EventRanking[]
}

/** 개인 추이 — 종목 1개의 회차별 기록 흐름. */
export interface TrendPoint {
  sessionDate: string
  value: number
  display: string
  achieved: boolean
  deltaFromPrevious: number | null   // 직전 유효 기록 대비 원값 변화. 첫 기록이면 null
  improved: boolean | null            // direction 반영한 개선 여부. 첫 기록이면 null
}

export interface PlayerEventTrend {
  event: string
  points: TrendPoint[]   // 유효 기록을 남긴 회차만(면제·미측정·이상값 제외), 날짜 오름차순
}

export interface PlayerPersonalBest {
  event: string
  value: number
  display: string
  sessionDate: string
  achieved: boolean
}

/** 선수 1명의 프로필 요약. status가 '탈퇴' 제외 타입이라 탈퇴 선수는 타입 수준에서 등장 불가. */
export interface PlayerSummary extends Player {
  status: Exclude<PlayerStatus, '탈퇴'>
  trends: PlayerEventTrend[]         // events[] 전체에 대해 항상 하나씩(기록 없으면 points: [])
  personalBests: PlayerPersonalBest[] // 유효 기록이 있는 종목만(스파스)
}

export interface EventAchievementRate {
  event: string
  achievedCount: number
  eligibleCount: number   // 활동 상태 + recorded만. 0이면(전원 면제 등) rate도 0
  rate: number
}

export interface HomeSummary {
  latestSession: {
    date: string
    participantCount: number   // 활동 상태 + participated:true만 (비대상·휴식 제외)
  } | null
  achievementRates: EventAchievementRate[]
}

/** GET /api/records 최종 응답. */
export interface RecordsResponse {
  generatedAt: string
  events: EventDefinition[]
  players: PlayerSummary[]
  sessions: Session[]
  rankings: SessionRankings[]
  home: HomeSummary
}

07이 계약을 따라야 하는 이슈

shared/domain.ts와 이 문서는 강제 계약이다 — 아래 후속 이슈들은 이 타입을 그대로 사용해야 하고, 필드를 바꾸려면 이 문서와 프론트 소비처를 함께 갱신해야 한다.

  • 이슈 #6(functions/lib/normalize-score.ts/normalize-status.ts) — 셀 원값 → ScoreValue/StatusValue 정규화. §03 매핑표대로 이 계약의 ScoreStatus로 변환되고, valueKind 교차검증(§03)은 이 정규화 다음 단계(조립 단계)의 책임
  • 시트 파서(버니스명단/목표/회차 탭 읽기) — Player · EventDefinition · Session/SessionEntry/EventScore 조립
  • 랭킹·추이·PB 계산 — SessionRankings · PlayerEventTrend · PlayerPersonalBest · HomeSummary 생성, §04 규칙(동점자·경계값·0/0·participantCount) 적용
  • /api/records 엔드포인트 — 위 결과를 RecordsResponse 형태로 조립해 반환 + 캐싱
  • P3~P4 프론트 전체 — 이 응답을 추가 가공 없이 그대로 렌더링
